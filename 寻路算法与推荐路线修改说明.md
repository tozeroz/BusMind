# 寻路算法与推荐路线修改说明

## 1. 修改背景

首页选择起点 `Aft Braddell Rd`、终点 `New Tech Pk` 后，推荐接口曾返回：

```text
车辆不存在：vehicle_id=0
```

排查发现，原来的公交寻路图基于全部静态线路和站点构建。寻路算法可以找到线路，但部分静态线路在 `bus_vehicle` 表中没有对应的非离线车辆。此类候选路线生成时使用占位值 `vehicle_id=0`，随后进入 ETA、客流和拥堵计算，最终在查询车辆信息时失败。

以本次测试场景为例，静态寻路曾找到 Bus 105、Bus 159 和 Bus 73，但只有 Bus 105 能绑定真实运行车辆。因此，问题并非寻路算法找不到线路，而是“静态可通行线路”和“当前有车辆运行的线路”没有在推荐阶段正确区分。

## 2. 修改目标

本次修改遵循以下原则：

- 数据库继续保存完整的静态线路、站点和站序，不删除数据。
- 实时推荐只在存在非离线车辆的线路上寻路。
- 当前“有车辆运行”的判断规则为 `bus_vehicle.operation_status != 'offline'`。
- 暂不使用 `last_reported_at` 判断车辆数据是否过期。
- 无法绑定真实车辆的候选路线不进入 ETA、客流和拥堵计算。
- 首页起终点选择只展示能够参与实时推荐的站点。
- 完整地图、线路管理和管理端仍可使用全部静态数据。

## 3. 推荐链路调整

修改后的推荐流程如下：

```text
选择起点和终点
    ↓
查询存在非离线车辆的线路 ID
    ↓
用这些线路及其关联站点构建实时公交子图
    ↓
寻路算法生成直达或换乘候选路线
    ↓
根据首段线路绑定最近上报的非离线车辆
    ↓
过滤 vehicle_id <= 0 的候选
    ↓
计算 ETA、客流、拥堵和推荐评分
    ↓
返回可实际执行的推荐路线
```

### 3.1 构建实时推荐子图

公交图构建时，先从车辆表查询非离线车辆对应的 `line_id` 集合。只有同时满足以下条件的线路才会进入推荐图：

1. `bus_line.status` 为 `running` 或 `active`；
2. 至少存在一辆 `operation_status != 'offline'` 的车辆。

图中的 `line_station` 站序和站点也只从这些线路加载。因此：

- 无车辆线路不会生成直达候选；
- 换乘路线的每一段都必须属于有车辆运行的线路；
- 静态数据库内容保持完整，但不会影响实时推荐结果。

对应文件：

- `backend/app/services/recommend_service/transit_graph.py`

### 3.2 绑定真实车辆并过滤无效候选

寻路算法的核心职责仍是确定线路、站点顺序和换乘关系。路线生成后，系统根据候选路线首段的 `line_id`，绑定该线路最近上报的非离线车辆。

绑定完成后新增最终保护：

- 只保留 `vehicle_id > 0` 的候选；
- 无法绑定车辆的候选直接丢弃；
- 无效候选不再进入 ETA、客流和拥堵计算；
- 如果所有候选都被过滤，推荐接口返回现有的 `40400 no route recommendation found`，而不是车辆 ID 错误。

对应文件：

- `backend/app/services/intelligence_gateway_mysql.py`

## 4. 站点查询接口调整

为以下接口增加可选参数：

```text
active_only: bool = false
```

涉及接口：

- `/locations/search`
- `/locations/nearby`

参数行为：

| 参数 | 行为 |
| --- | --- |
| `active_only=false` | 返回完整静态站点，保持原有接口兼容性 |
| `active_only=true` | 只返回至少关联一条有非离线车辆线路的站点 |

首页文本搜索和附近定位统一传入 `active_only=true`，避免用户选择一个无法参与实时推荐的站点。其他页面未强制启用该参数，因此仍然可以浏览完整静态数据。

对应文件：

- `backend/app/api/v1/locations/router.py`
- `backend/app/services/bus_service.py`
- `frontend/src/modules/home/components/HomePage.vue`

## 5. 推荐性能优化

功能修复后，推荐能够返回 Bus 105，但接口仍需要约 14～24 秒。性能分析发现，主要耗时不在寻路算法，而在路线拥堵计算。

### 5.1 原有问题

原来的交通速度查询会：

1. 查询最新交通快照时间；
2. 读取该时间点的全部交通速度记录；
3. 将约 15 万条记录加载到 Python；
4. 在 Python 中逐条计算距离，寻找离目标路段最近的记录；
5. 对路线的多个路段重复执行类似操作。

一次性能分析中，共解析了约 153,478 条 `TrafficSpeedBand` 数据，拥堵计算占用了推荐链路的大部分时间。

### 5.2 优化方式

优化后将最近交通记录的筛选交给数据库完成：

- 使用标量子查询确定最新交通快照；
- 先限定目标坐标附近约 ±0.03° 的范围；
- 在 SQL 中计算坐标距离并排序；
- 只读取距离最近的一条记录；
- 附近没有记录时，再降级为从最新快照中全局选择最近的一条；
- 拥堵计算直接复用已经加载的道路坐标，不再按路段 ID 重复查询道路信息。

对应文件：

- `backend/app/repositories/transit_repository.py`

### 5.3 性能结果

本地通过前端 Vite 代理实测：

| 场景 | 耗时 |
| --- | ---: |
| 优化前 | 约 14.6 秒 |
| 服务重载后的首次请求 | 约 8.07 秒 |
| 服务预热后的后续请求 | 约 2.06 秒 |

首次请求仍可能受到服务初始化、数据库连接和模型预热影响，但后续推荐请求已经从十几秒降低到约两秒。

## 7. 修改后的业务含义

需要明确区分三个概念：

- `line_id`：数据库内部的线路主键，用来关联线路、站序和车辆。
- `line_name`：面向用户展示的线路名称，例如 Bus 105。
- `vehicle_id`：某一辆实际或模拟运行车辆的唯一标识，用于查询 ETA、位置、客流等车辆级数据。

寻路算法首先确定可以连接起终点的 `line_id` 和线路组合；推荐层再把首段线路绑定到具体的 `vehicle_id`。本次修改通过提前构建有车线路子图，并在计算前再次校验车辆 ID，使“找到静态线路”和“存在可推荐车辆”保持一致。

## 8. 未修改的范围与后续建议

本次没有执行以下操作：

- 没有删除静态线路或站点；
- 没有新增数据库表；
- 没有修改现有推荐响应结构；
- 没有补造缺失车辆数据；
- 没有修改 ETA 和客流预测模型；
- 没有按车辆上报时间判断实时数据新鲜度。

如果后续接入真实车辆数据，建议增加 `last_reported_at` 新鲜度阈值，例如只将最近若干分钟内上报过位置的车辆视为在线。这样可以避免数据库中状态为非离线、但实际上已经停止上报的车辆继续参与推荐。
