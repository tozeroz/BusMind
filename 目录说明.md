# 智行公交项目目录说明

## 一、项目目录总览

本项目采用前后端分离结构，前端使用 **Vue3 + Element Plus + ECharts + 地图 API**，后端使用 **FastAPI**，数据库使用 **MySQL**，算法部分使用 **Python + Pandas + Scikit-learn / XGBoost**，并预留大模型助手接口。完整目录结构如下：

```text
BusMind/
├── frontend/                         # 前端项目目录，存放 Vue3 页面、组件、路由、接口请求等
│   ├── public/                       # 前端静态资源目录
│   ├── src/                          # 前端核心源码目录
│   │   ├── api/                      # Axios 接口请求封装
│   │   ├── assets/                   # 图片、图标、字体等资源
│   │   ├── components/               # 公共组件目录
│   │   │   ├── common/               # 通用组件，如按钮、卡片、弹窗等
```

## 二、前端目录说明

前端所有代码放在：

```text
frontend/
```

前端主要负责用户看到的页面，包括公交线路查询、站点查询、车辆位置展示、拥挤程度展示、舒适推荐页面和大模型助手页面。

### 2.1 前端核心目录

| 目录 | 说明 |
|---|---|
| `frontend/public/` | 存放不会被打包处理的静态资源 |
| `frontend/src/` | 前端核心源码目录 |
| `frontend/src/api/` | 封装所有 Axios 请求 |
| `frontend/src/assets/` | 存放图片、图标、字体等资源 |
| `frontend/src/components/` | 存放可复用组件 |
| `frontend/src/views/` | 存放页面级组件 |
| `frontend/src/router/` | 配置前端页面路由 |
| `frontend/src/stores/` | Pinia 状态管理 |
| `frontend/src/utils/` | 工具函数 |
| `frontend/src/styles/` | 全局样式 |
| `frontend/src/layout/` | 页面布局组件 |
| `frontend/env/` | 前端环境变量 |
| `frontend/tests/` | 前端测试 |

### 2.2 前端页面对应目录

| 页面 | 建议目录 |
|---|---|
| 登录注册页 | `frontend/src/views/login/` |

### 2.3 前端接口请求放置位置

前端请求后端接口的文件统一放在：

```text
frontend/src/api/
```

建议后续可以按业务拆分为：

```text
frontend/src/api/user.js
frontend/src/api/line.js
frontend/src/api/station.js
frontend/src/api/vehicle.js
frontend/src/api/eta.js
frontend/src/api/crowd.js
```

## 三、后端目录说明

后端所有代码放在：

```text
backend/
```

FastAPI 核心应用代码放在：

```text
backend/app/
```

后端主要负责处理前端请求、访问数据库、调用算法模块、调用大模型接口，并把结果返回给前端。

### 3.1 后端核心目录

| 目录 | 说明 |
|---|---|
| `backend/app/api/` | 接口路由目录 |
| `backend/app/core/` | 项目配置目录 |
| `backend/app/db/` | 数据库连接和 Session 管理 |
| `backend/app/models/` | SQLAlchemy 数据库模型 |
| `backend/app/schemas/` | Pydantic 数据校验模型 |
| `backend/app/services/` | 业务逻辑层 |
| `backend/app/repositories/` | 数据库访问层 |
| `backend/app/dependencies/` | FastAPI 依赖注入 |
| `backend/app/middleware/` | 中间件 |
| `backend/app/utils/` | 后端工具函数 |
| `backend/migrations/` | 数据库迁移 |
| `backend/tests/` | 后端测试 |
| `backend/logs/` | 后端日志 |

### 3.2 后端接口目录

接口路由建议统一放在：

```text
backend/app/api/v1/
```

按业务拆分如下：

| 接口模块 | 建议目录 | 作用 |
|---|---|---|
| 用户接口 | `backend/app/api/v1/user/` | 登录、注册、用户信息 |
| 线路接口 | `backend/app/api/v1/line/` | 公交线路查询 |
| 站点接口 | `backend/app/api/v1/station/` | 站点查询 |
| 车辆接口 | `backend/app/api/v1/vehicle/` | 车辆实时状态查询 |
| ETA 接口 | `backend/app/api/v1/eta/` | 预计到站时间查询 |
| 拥挤预测接口 | `backend/app/api/v1/crowd/` | 查询拥挤程度预测结果 |
| 舒适推荐接口 | `backend/app/api/v1/recommend/` | 返回舒适出行方案 |
| AI 助手接口 | `backend/app/api/v1/ai/` | 大模型公交助手问答 |

### 3.3 后端业务逻辑目录

业务逻辑建议放在：

```text
backend/app/services/
```

具体拆分如下：

| 业务模块 | 建议目录 | 作用 |
|---|---|---|
| 用户业务 | `backend/app/services/user_service/` | 用户登录、注册、鉴权 |
| 公交业务 | `backend/app/services/bus_service/` | 线路、站点、车辆相关业务 |
| ETA 业务 | `backend/app/services/eta_service/` | 到站时间计算 |
| 拥挤预测业务 | `backend/app/services/crowd_service/` | 拥挤等级预测 |
| 推荐业务 | `backend/app/services/recommend_service/` | 综合推荐方案计算 |
| AI 业务 | `backend/app/services/ai_service/` | 调用大模型并生成自然语言回答 |

## 五、算法目录说明

```text
algorithm/
```

### 5.1 拥挤预测目录

拥挤预测是项目重点，建议拆得更清楚：

```text
algorithm/crowd_prediction/
├── rule_model/        # 规则模型，适合先做基础版本
├── ml_model/          # 机器学习模型，如 XGBoost、随机森林
├── training/          # 训练模型
└── evaluation/        # 评估模型效果
```

开发顺序建议：

```text
先做规则模型 → 系统先跑通 → 再接 XGBoost 或随机森林
```

## 七、仿真模拟目录说明

模拟程序放在：

```text
simulator/
```

该目录用于在没有真实公交接口时，模拟公交车辆位置、客流变化和拥挤状态。

| 目录 | 说明 |
|---|---|
| `simulator/vehicle_position/` | 模拟车辆在线路上移动 |
| `simulator/passenger_flow/` | 模拟早高峰、晚高峰、雨天等客流变化 |
| `simulator/scheduler/` | 定时任务，比如每 5 秒更新车辆位置 |

该目录的作用是让项目在没有真实数据接口的情况下也能完整演示。

## 八、大模型助手目录说明

大模型相关代码放在：

```text
llm/
```

该目录主要用于管理 Prompt、调用 DeepSeek / 通义千问 / OpenAI API，并生成自然语言出行建议。

| 目录 | 说明 |
|---|---|
| `llm/prompts/` | 大模型 Prompt 模板 |
| `llm/providers/` | 不同大模型服务商接口封装 |
| `llm/providers/deepseek/` | DeepSeek API 封装 |
| `llm/providers/tongyi/` | 通义千问 API 封装 |
| `llm/providers/openai/` | OpenAI API 封装 |
| `llm/context_builders/` | 把线路、ETA、拥挤程度、推荐结果整理成上下文 |

注意：  
大模型不应该直接负责计算 ETA 或拥挤程度。正确流程应该是：

```text
后端 / 算法模块先计算结果
        ↓
整理成结构化上下文
        ↓
交给大模型生成解释文本
```

## 十六、核心业务数据流

系统主要业务流程如下：

```text
用户输入起点和终点
        ↓
前端通过 Axios 请求后端推荐接口
        ↓
后端查询线路、站点、车辆、历史客流数据
        ↓
调用 ETA 模块计算预计到站时间
        ↓
调用拥挤预测模块计算拥挤等级
        ↓
调用推荐模块计算综合推荐分数
        ↓
可选：调用大模型生成推荐解释
        ↓
后端返回推荐方案
        ↓
前端用地图、卡片、图表展示结果
```

## 十七、各模块之间的关系

| 模块 | 依赖模块 | 输出结果 |
|---|---|---|
| 前端模块 | 后端接口 | 页面展示 |
| 后端模块 | 数据库、算法、大模型 | RESTful API 数据 |
| 数据库模块 | 无 | 线路、站点、车辆、客流数据 |
| ETA 模块 | 线路、站点、车辆状态 | 预计到站时间 |
| 拥挤预测模块 | 历史客流、车辆人数、时间、天气 | 拥挤等级 |
| 推荐模块 | ETA、拥挤等级、换乘次数、步行距离 | 舒适出行方案 |
| 大模型模块 | 推荐结果、路线数据、拥挤数据 | 自然语言解释 |

## 十九、总结

本目录结构按照“前端展示、后端接口、数据库存储、算法计算、大模型解释、文档与测试分离”的原则设计，适合本项目的课程开发和小组协作。

整体结构的核心思想是：

```text
frontend 负责页面展示
backend 负责接口和业务调度
database 负责数据存储
algorithm 负责 ETA、拥挤预测和推荐计算
simulator 负责生成模拟车辆和客流数据
llm 负责大模型解释和自然语言建议
docs 负责项目文档和答辩材料
tests 负责测试验证
deployment 负责后期部署
```

这样拆分后，每个成员可以在自己的目录内独立开发，最后通过 RESTful API 和 Axios 完成前后端联调。
