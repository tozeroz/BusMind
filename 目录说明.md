# 智行公交项目代码目录说明

本文档用于说明 `BusMind`目录下各模块的用途，以及 5 位成员日常开发应提交到哪些目录。

## 一、总体目录结构

```text
BusMind/
├── frontend/              # 客户端工程师：前端项目
├── backend/               # 服务端工程师 A/B：后端业务服务
├── algorithm-service/     # 数据处理工程师 + 服务端 B：算法服务
├── data/                  # 数据处理工程师：数据集、建表 SQL、初始化数据
├── ai-service/            # 项目经理 + 服务端 B：大模型提示词与 AI 调用封装
├── api-docs/              # 服务端工程师 A/B：接口文档
├── docs/                  # 项目经理：需求、设计、汇报、答辩文档
├── tests/                 # 服务端 B 牵头：接口测试、联调测试、测试报告
├── scripts/               # 全体可用：启动、导入数据、部署辅助脚本
└── deploy/                # 后期部署配置，如 Nginx、Docker、本地部署说明
```

## 二、成员开发目录分配

| 成员角色 | 主要开发目录 | 说明 |
| --- | --- | --- |
| 项目经理 | `docs/`、`ai-service/prompts/`、`api-docs/` | 负责需求文档、系统设计、PPT 素材、演示脚本、AI 助手提示词设计，并协助维护接口说明 |
| 服务端工程师 A | `backend/`、`api-docs/` | 负责用户、线路、站点、车辆、管理端基础接口，以及数据库访问代码 |
| 服务端工程师 B | `backend/`、`algorithm-service/`、`ai-service/`、`tests/` | 负责 ETA、拥挤预测接口、推荐接口、预警接口、AI 出行助手、AI 调度建议和接口测试 |
| 客户端工程师 | `frontend/` | 负责乘客端、管理端页面、路由、组件、接口联调、ECharts 和地图展示 |
| 数据处理工程师 | `data/`、`algorithm-service/` | 负责模拟数据、CSV、初始化 SQL、数据预处理、拥挤预测模型、图表数据 |

## 三、各目录具体说明

### `frontend/`

前端项目目录，建议使用：

```text
Vue3 + Element Plus + ECharts + 高德地图 API + Axios
```

建议后续结构：

```text
frontend/
├── src/
│   ├── api/          # Axios 接口封装
│   ├── assets/       # 图片、图标、静态资源
│   ├── components/   # 通用组件
│   ├── router/       # 路由配置
│   ├── stores/       # 状态管理
│   ├── views/        # 页面
│   │   ├── passenger/ # 乘客端页面
│   │   └── admin/     # 管理端页面
│   └── utils/        # 前端工具函数
└── package.json
```

### `backend/`

后端主服务目录，建议使用：

```text
SpringBoot + MyBatis/MyBatis Plus + MySQL
```

如果后端最终选择 Flask，也放在该目录，不再另建平行目录。

建议后续模块：

```text
backend/
├── user/        # 登录、注册、角色权限
├── line/        # 公交线路
├── station/     # 站点
├── vehicle/     # 车辆状态
├── eta/         # 到站时间
├── recommend/   # 舒适车次推荐
├── warning/     # 拥挤预警
└── admin/       # 管理端统计与管理接口
```

### `algorithm-service/`

算法服务目录，主要存放 Python 代码。

建议职责：

```text
拥挤程度预测
ETA 参数计算
历史客流数据预处理
模型训练与评估
算法接口 Demo
```

建议后续结构：

```text
algorithm-service/
├── notebooks/   # 试验过程，可选
├── models/      # 训练后的模型文件
├── src/         # 算法代码
├── outputs/     # 预测结果、评估图表
└── README.md
```

### `data/`

数据目录，由数据处理工程师主维护。

建议后续结构：

```text
data/
├── raw/          # 原始或手工整理数据
├── processed/    # 清洗后的数据
├── mock/         # 模拟生成的数据
├── sql/          # 建表 SQL、初始化 SQL
└── dictionary/   # 数据字典、字段说明
```

### `ai-service/`

AI 功能目录，存放大模型调用封装、提示词和样例输入输出。

建议后续结构：

```text
ai-service/
├── prompts/      # AI 出行助手、AI 调度助手提示词
├── examples/     # 示例请求与回复
└── src/          # 大模型 API 调用封装
```

注意：API Key 不要提交到 Git 仓库，应放在本地 `.env` 或系统环境变量。

### `api-docs/`

接口文档目录，服务端工程师 A/B 共同维护。

建议文件：

```text
接口总览.md
用户与登录接口.md
线路站点车辆接口.md
预测推荐接口.md
管理端接口.md
AI接口.md
```

### `docs/`

项目文档目录，由项目经理主维护，全体补充。

建议文件：

```text
需求分析.md
系统设计.md
数据库设计.md
项目进度.md
每日问题清单.md
中期汇报提纲.md
答辩演示脚本.md
```

### `tests/`

测试目录，主要用于接口测试、联调记录和测试报告。

建议后续结构：

```text
tests/
├── api/          # Postman/Apifox 导出文件或接口测试脚本
├── integration/  # 前后端联调记录
└── reports/      # 测试报告
```

### `scripts/`

脚本目录，放辅助命令，不放核心业务代码。

常见脚本：

```text
导入初始化数据
生成模拟数据
启动前端
启动后端
启动算法服务
```

### `deploy/`

部署配置目录。实训阶段本地部署即可，后期如需演示部署，可放：

```text
Nginx 配置
Docker Compose
部署说明
环境变量模板
```

## 四、分支与提交协作规则

### 4.1 基本提交规则

1. 不要把临时文件、IDE 缓存、日志文件、数据库密码、API Key 提交到仓库。
2. 每个成员优先在自己负责的目录下开发，跨目录改动需要提前说明。
3. 接口字段一旦确定，应同步更新 `api-docs/`。
4. 数据字段一旦调整，应同步更新 `data/dictionary/` 和数据库 SQL。
5. 前后端联调问题统一记录在 `docs/每日问题清单.md` 或 `tests/integration/`。
6. 演示前一天冻结核心功能，后续只修复 bug，不再临时大改架构。

### 4.2 分支命名规则

每位成员创建自己的长期开发分支，日常开发都在个人分支完成，不直接在 `main` 分支写代码。

建议分支命名：

```text
main                    # 主分支，保持稳定可运行
pm-docs                 # 项目经理：文档、AI 提示词、答辩材料
backend-a               # 服务端工程师 A：业务后端
backend-b               # 服务端工程师 B：算法接口、AI 接口、测试
frontend                # 客户端工程师：前端页面
data-algorithm          # 数据处理工程师：数据集、算法模型
```

如果需要开发某个较大的功能，也可以在个人分支基础上再开功能分支，例如：

```text
frontend-admin-dashboard
backend-a-user-line-api
data-algorithm-crowd-model
```

### 4.3 每天开始开发前

每天写代码前，先从远程主分支同步最新代码，保证自己的开发基于最新版本。

推荐流程：

```bash
git checkout main
git pull origin main
git checkout 自己的分支
git merge main
```

如果合并时出现冲突，先解决冲突，再继续开发。冲突文件涉及其他成员负责模块时，需要先和对应成员确认。

### 4.4 日常开发流程

每个人只在自己的分支上提交代码。

推荐流程：

```bash
git checkout 自己的分支

# 修改代码后
git status
git add 修改过的文件
git commit -m "说明本次完成的功能"
```

提交信息要写清楚本次做了什么，例如：

```text
完成线路查询接口
新增公交详情页
生成历史客流模拟数据
补充 AI 出行助手提示词
修复登录接口参数校验
```

### 4.5 开发完成后的本地测试流程

个人分支开发完成后，不要直接推到 `main`。应先在本地把个人分支合并到本地主分支测试，确认主分支环境可以正常运行。

推荐流程：

```bash
git checkout main
git pull origin main
git merge 自己的分支

# 在本地主分支运行项目并测试
# 例如：启动后端、启动前端、测试接口、检查页面
```

测试通过后，再把个人分支推送到远程仓库：

```bash
git checkout 自己的分支
git push origin 自己的分支
```

然后在 Gitee/GitHub 上发起合并请求，由项目经理或对应模块负责人确认后再合并到远程 `main`。

### 4.6 合并到主分支的要求

合并到 `main` 前至少满足：

1. 自己负责的功能能正常运行。
2. 不影响其他成员已有功能。
3. 涉及接口变化时，已经更新 `api-docs/`。
4. 涉及数据字段变化时，已经更新 `data/dictionary/` 和 `data/sql/`。
5. 前端页面可以正常访问，后端接口可以正常返回，算法或 AI 接口有基本测试结果。
6. 项目经理或对应模块负责人已经确认可以合并。

### 4.7 简化版每日协作流程

```text
每天开发前：
切到 main → 拉取远程 main → 切回个人分支 → 合并 main

日常开发中：
只在个人分支写代码 → 小步提交 → 写清楚提交信息

开发完成后：
本地 main 合并个人分支 → 本地运行测试 → 推送个人远程分支 → 发起合并请求

合并主分支：
负责人检查 → 通过后合并到远程 main
```
