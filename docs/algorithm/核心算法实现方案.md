# 核心算法实现方案

> 本文档说明 BusMind 当前核心算法实现：将数据集划分为训练集和测试集，分别训练 ETA 预测模型和客载状态预测模型，再将两个预测结果输入出行体验评价与推荐模块

参考数据源

- LTA DataMall Dynamic Datasets: `https://datamall.lta.gov.sg/content/datamall/en/dynamic-data.html`
- LTA DataMall API User Guide: `https://datamall.lta.gov.sg/content/dam/datamall/datasets/LTA_DataMall_API_User_Guide.pdf`

# 一、算法总体流程

本项目最终目标是推荐体验更好的公交出行方案。核心算法分为三部分：

```text
ETA 预测
    ↓
客载状态预测
    ↓
出行体验评价与推荐
```

整体流程如下：

```text
整理 LTA DataMall 风格数据集或仿真数据集
        ↓
构造训练样本和测试样本
        ↓
训练 ETA 预测模型
        ↓
训练客载状态预测模型
        ↓
对候选公交方案预测 predicted_eta_minutes 和 predicted_load
        ↓
计算时间得分、客载体验得分、步行得分、换乘得分
        ↓
输出最快方案、最少换乘方案、车内更宽松方案、综合推荐方案
```

一句话说明：

```text
ETA 预测负责判断“车多久到”
客载状态预测负责判断“车内体验如何”
推荐算法负责判断“哪条路线更适合用户”
```

# 二、数据集使用方案

本项目参考 LTA DataMall 的公开数据结构。真实数据可用时直接采集；真实数据不足时，按相同字段结构生成仿真数据，保证后续可以替换为真实数据源

| **数据集** | **使用方式** | **对应算法环节** |
|---|---|---|
| `Bus Arrival` | 提供 `EstimatedArrival`、车辆位置、`Load` | 构造 ETA 标签和客载状态标签 |
| `Bus Stops` | 提供站点编号、站点名称、经纬度 | 站点匹配、步行距离、地图展示 |
| `Bus Routes` | 提供线路经过站点、方向、站点顺序、距离 | ETA 特征、车内行驶时间估计 |
| `Bus Services` | 提供线路起终点、发车频率、运营信息 | ETA 特征、客载状态特征 |
| `Passenger Volume by Bus Stops` | 提供站点按小时 tap in / tap out 客流 | 客载状态预测特征 |
| `Passenger Volume by Origin Destination Bus Stops` | 提供公交站点间 OD 出行量 | 客载状态预测特征 |
| `weather` | 获取或仿真天气、温度、降雨情况 | ETA 预测特征、客载状态预测特征 |

## (一) `Bus Arrival` 的关键作用

`Bus Arrival` 中两个字段最重要：

| **字段** | **用途** |
|---|---|
| `EstimatedArrival` | 构造 ETA 预测标签 |
| `Load` | 构造客载状态预测标签 |

注意：

```text
EstimatedArrival 不作为最终 ETA 算法结果，而是用来构造 eta_label
Load 不表示精确车上人数，而是客载状态分类标签
```

ETA 标签构造：

```text
eta_label = EstimatedArrival - query_time
```

含义：

```text
eta_label 表示当前查询时刻距离公交预计到站还剩多少分钟
```

客载状态标签：

```text
load_label = Load
```

`Load` 编码说明：

| **Load 编码** | **含义** | **项目展示** |
|---|---|---|
| `SEA` | Seats Available | 有座位，较宽松 |
| `SDA` | Standing Available | 可站立，适中 |
| `LSD` | Limited Standing | 站立空间有限，较拥挤 |

## (二) 站点客流和 OD 客流的作用

`Passenger Volume by Bus Stops` 提供站点上下车客流：

```text
YEAR_MONTH
DAY_TYPE
TIME_PER_HOUR
PT_CODE
TOTAL_TAP_IN_VOLUME
TOTAL_TAP_OUT_VOLUME
```

用于构造：

```text
站点上车热度
站点下车热度
小时客流规律
工作日 / 周末客流差异
```

`Passenger Volume by Origin Destination Bus Stops` 提供站点间 OD 出行量：

```text
ORIGIN_PT_CODE
DESTINATION_PT_CODE
TIME_PER_HOUR
TOTAL_TRIPS
```

用于构造：

```text
起终点需求强度
线路方向客流压力
候选路线潜在客载压力
```

# 三、训练集与测试集划分

本项目采用时间切分方式构造训练集和测试集

```text
第 1-30 天：训练集
第 31 天：测试集
```

训练集用于：

```text
训练 ETA 预测模型
训练客载状态预测模型
计算 baseline
```

测试集用于：

```text
验证 ETA 预测效果
验证客载状态预测效果
验证推荐结果是否合理
```

重要原则：

```text
第 31 天数据不能参与模型训练
第 31 天只作为 ground truth
预测结果必须和第 31 天真实或仿真标签对比
```

如果暂时无法获取真实连续 31 天 LTA 数据，可以先生成 LTA 字段风格的 31 天仿真数据。评价时仍然按照相同方式切分

# 四、ETA 预测模型

## (一) 实现目标

ETA 预测模型用于预测公交到达用户上车站点还需要多少分钟

预测目标：

```text
predicted_eta_minutes
```

训练标签：

```text
eta_label = EstimatedArrival - query_time
```

如果真实 `EstimatedArrival` 暂时不可用，则在仿真数据中生成对应的 `eta_label`，用于训练和验证

## (二) 特征设计

| **特征** | **来源** | **说明** |
|---|---|---|
| `service_no` | `Bus Arrival` / `Bus Routes` | 公交线路 |
| `bus_stop_code` | `Bus Arrival` / `Bus Stops` | 上车站点 |
| `direction` | `Bus Routes` | 行驶方向 |
| `stop_sequence` | `Bus Routes` | 站点顺序 |
| `distance_to_stop` | `Bus Routes` / 车辆位置 | 车辆到目标站点距离 |
| `hour` | 查询时间 | 小时 |
| `day_type` | 时间字段 | 工作日 / 周末 |
| `service_frequency` | `Bus Services` | 发车频率 |
| `weather` | 天气数据 | 天气情况 |
| `previous_eta` | 历史到站数据 | 上一时间片 ETA |
| `previous_load` | `Bus Arrival Load` | 上一时间片客载状态 |

## (三) 模型选择

ETA 预测是回归任务

可选模型：

```text
Linear Regression
Decision Tree Regressor
Random Forest Regressor
XGBoost Regressor
```

第一阶段建议：

```text
先实现 Linear Regression 作为简单模型
再使用 Random Forest Regressor 或 XGBoost Regressor 作为主要模型
```

原因：

```text
ETA 特征多为表格数据
树模型适合处理非线性关系
实现成本低，适合课程项目展示
```

## (四) ETA baseline

Baseline 用于评价模型效果，不作为最终算法亮点

建议设置两个 baseline：

| **Baseline** | **方法** | **作用** |
|---|---|---|
| 平均值 baseline | 永远预测训练集 ETA 平均值 | 弱 baseline |
| 历史均值 baseline | 预测同线路、同站点、同小时的历史平均 ETA | 业务 baseline |

历史均值 baseline：

```text
baseline_eta =
avg(eta_label | service_no, bus_stop_code, hour, day_type)
```

如果样本不足，按以下顺序降级：

```text
service_no + bus_stop_code + hour + day_type
service_no + bus_stop_code + hour
service_no + hour
bus_stop_code + hour
全局平均 ETA
```

## (五) ETA 评价指标

在测试集上比较：

```text
predicted_eta_minutes
eta_label
```

评价指标：

```text
MAE = avg(|predicted_eta_minutes - eta_label|)
RMSE = sqrt(avg((predicted_eta_minutes - eta_label)^2))
```

重点展示：

```text
模型 MAE 是否低于 baseline MAE
模型 RMSE 是否低于 baseline RMSE
```

# 五、客载状态预测模型

## (一) 实现目标

客载状态预测模型用于预测候选公交在用户乘坐时的 `Load` 状态

预测目标：

```text
predicted_load
```

训练标签：

```text
load_label = Load
```

分类标签：

```text
SEA / SDA / LSD
```

## (二) 特征设计

| **特征** | **来源** | **说明** |
|---|---|---|
| `service_no` | `Bus Arrival` / `Bus Routes` | 公交线路 |
| `bus_stop_code` | `Bus Arrival` / `Bus Stops` | 查询站点 |
| `direction` | `Bus Routes` | 行驶方向 |
| `stop_sequence` | `Bus Routes` | 站点在线路中的顺序 |
| `hour` | 查询时间 | 小时 |
| `day_type` | 客流数据 | 工作日 / 周末 |
| `tap_in_volume` | `Passenger Volume by Bus Stops` | 站点上车客流 |
| `tap_out_volume` | `Passenger Volume by Bus Stops` | 站点下车客流 |
| `od_trips` | `Passenger Volume by OD Bus Stops` | 起终点需求 |
| `service_frequency` | `Bus Services` | 发车频率 |
| `weather` | 外部天气或仿真 | 天气情况 |
| `previous_load` | 历史 `Bus Arrival` | 上一时间片客载状态 |

## (三) 模型选择

客载状态预测是三分类任务

可选模型：

```text
Logistic Regression
Decision Tree
Random Forest
XGBoost
```

第一阶段建议：

```text
Logistic Regression 作为简单模型
Random Forest 或 XGBoost 作为主要模型
```

原因：

```text
适合表格特征
能处理非线性关系
对特征尺度不敏感
比深度学习更容易解释和落地
```

## (四) 客载状态 baseline

建议设置两个 baseline：

| **Baseline** | **方法** | **作用** |
|---|---|---|
| 多数类 baseline | 永远预测训练集中出现最多的 `Load` 类别 | 弱 baseline |
| 历史众数 baseline | 预测同线路、同站点、同小时最常见的 `Load` | 业务 baseline |

历史众数 baseline：

```text
baseline_load =
mode(Load | service_no, bus_stop_code, hour, day_type)
```

如果样本不足，按以下顺序降级：

```text
service_no + bus_stop_code + hour + day_type
service_no + bus_stop_code + hour
service_no + hour
bus_stop_code + hour
全局最多 Load 类别
```

## (五) 客载状态评价指标

在测试集上比较：

```text
predicted_load
load_label
```

评价指标：

```text
Accuracy：整体分类准确率
Macro-F1：三类 Load 的 F1 平均值
Confusion Matrix：查看不同等级之间的误判情况
```

重点关注：

```text
LSD 被预测成 SEA：严重低估拥挤，风险最高
SEA 被预测成 LSD：高估拥挤，会降低推荐体验
```

## (六) 输出字段

| **字段** | **说明** |
|---|---|
| `predicted_load` | 预测客载状态，取值 `SEA`、`SDA`、`LSD` |
| `load_score` | 客载体验得分 |
| `model_name` | 使用的模型，例如 `random_forest`、`xgboost` |
| `confidence` | 预测置信度 |

客载体验得分：

```text
SEA = 100
SDA = 70
LSD = 35
```

# 六、出行体验评价与推荐

## (一) 实现目标

出行体验评价与推荐模块综合 ETA、客载状态、步行时间和换乘次数，为用户输出不同类型的推荐方案

## (二) 候选方案字段

| **字段** | **说明** |
|---|---|
| `route_id` | 推荐方案编号 |
| `service_no` | 公交线路编号 |
| `line_name` | 公交线路名称 |
| `start_stop_code` | 上车站点 |
| `end_stop_code` | 下车站点 |
| `predicted_eta_minutes` | 预测等车时间 |
| `walk_time` | 到站步行时间 |
| `ride_time` | 车内行驶时间 |
| `time_cost` | 总时间成本 |
| `predicted_load` | 预测客载状态 |
| `load_score` | 客载体验得分 |
| `transfer_count` | 换乘次数 |
| `experience_score` | 出行体验综合分 |
| `recommend_type` | 推荐类型 |
| `reason` | 推荐原因 |

## (三) 指标归一化

时间成本：

```text
time_cost = walk_time + predicted_eta_minutes + ride_time
```

时间成本得分：

```text
time_score = max(0, 100 - time_cost * 3)
```

客载体验得分：

```text
SEA = 100
SDA = 70
LSD = 35
```

步行体验得分：

```text
walk_score = max(0, 100 - walk_time * 8)
```

换乘体验得分：

```text
transfer_score = max(0, 100 - transfer_count * 30)
```

## (四) 综合评分

出行体验评分采用加权综合评价方式。四个权重可以根据用户偏好动态调整，通用公式如下：

```text
experience_score =
w_load * load_score
+ w_time * time_score
+ w_walk * walk_score
+ w_transfer * transfer_score
```

其中：

```text
w_load + w_time + w_walk + w_transfer = 1
```

默认综合推荐权重：

| **指标** | **默认权重** | **含义** |
|---|---:|---|
| `load_score` | `0.35` | 车内体验 |
| `time_score` | `0.30` | ETA 和总耗时 |
| `walk_score` | `0.20` | 步行负担 |
| `transfer_score` | `0.15` | 换乘复杂度 |

默认权重偏向车内体验，同时兼顾时间成本、步行负担和换乘复杂度

用户偏好权重建议：

| **用户偏好** | **`w_load`** | **`w_time`** | **`w_walk`** | **`w_transfer`** | **说明** |
|---|---:|---:|---:|---:|---|
| `balanced` | `0.35` | `0.30` | `0.20` | `0.15` | 综合推荐，各指标相对均衡 |
| `comfort` | `0.55` | `0.20` | `0.15` | `0.10` | 更重视车内不拥挤 |
| `fastest` | `0.20` | `0.55` | `0.15` | `0.10` | 更重视总耗时 |
| `less_walk` | `0.25` | `0.25` | `0.40` | `0.10` | 更重视少走路 |
| `less_transfer` | `0.25` | `0.25` | `0.15` | `0.35` | 更重视少换乘 |

## (五) 推荐类型

| **推荐类型** | **选择规则** |
|---|---|
| `fastest` | `time_cost` 最小 |
| `best_experience` | 当前用户偏好下 `experience_score` 最高 |
| `least_transfer` | `transfer_count` 最少，若相同则体验分更高 |
| `least_crowded` | `load_score` 最高，若相同则时间成本更低 |

## (六) 推荐理由

第一阶段使用模板生成推荐理由

```text
推荐 {line_name}，预计 {predicted_eta_minutes} 分钟后到达上车站，
预测客载状态为 {load_text}，步行约 {walk_time} 分钟，
换乘 {transfer_count} 次。
该方案在时间成本和车内体验之间更均衡，因此作为综合推荐
```

# 八、第一阶段实现重点

第一阶段优先完成以下内容：

```text
1. 整理 LTA 风格字段结构
2. 构造 ETA 预测训练样本，标签为 eta_label
3. 构造客载状态预测训练样本，标签为 Load
4. 分别训练 ETA 回归模型和 Load 分类模型
5. 为 ETA 设置平均值 baseline 和历史均值 baseline
6. 为 Load 设置多数类 baseline 和历史众数 baseline
7. 用第 31 天测试集评价 MAE、RMSE、Accuracy、Macro-F1
8. 将 predicted_eta_minutes 和 predicted_load 接入出行体验评分
```

一句话总结：

> 训练集用于训练 ETA 预测模型和客载状态预测模型，测试集用于验证预测效果，推荐算法使用两个模型的预测结果进行出行体验评分
