> 当前方案判断：BusMind 第一阶段不再把 ETA 和 Load 作为机器学习预测模型重点，而是将 LTA 实时到站、实时客载、历史客流、道路热力和用户偏好融合起来，构建可解释的多目标公交出行推荐算法

# 一、算法设想

## (一) 算法主线

早期算法主线是：

```text
ETA 预测
        ↓
客载状态预测
        ↓
出行体验评价与推荐
```

该方案默认需要先训练 ETA 回归模型和 Load 三分类模型，再将两个预测结果输入推荐模块

经过 LTA DataMall 数据源调研后，当前主线调整为：

```text
实时公交状态提取
        ↓
历史客流与道路热力特征整理
        ↓
候选公交路线生成
        ↓
多目标出行体验评价
        ↓
用户偏好动态权重推荐
        ↓
结构化推荐结果
        ↓
大模型自然语言解释
```

一句话说明：

```text
ETA 和 Load 是输入数据，出行体验评价和个性化路线推荐才是当前算法核心
```

算法名称可以定义为：

```text
基于多源数据融合的个性化公交出行推荐算法
```

更完整的表述是：

```text
融合实时公交状态、道路热力、历史客流与用户偏好的多目标公交出行推荐算法
```

## (二) 算法目标

本算法不是单纯推荐最快路线，而是在多条可达公交方案中综合考虑：

- 时间效率
- 车内舒适度
- 站点和线路客流压力
- 道路拥堵情况
- 步行负担
- 换乘复杂度
- 数据可靠性
- 用户偏好

算法需要回答这些问题：

```text
1. 当前有哪些公交方案可以到达目的地
2. 每条方案的实时 ETA 是多少
3. 当前车辆客载状态如何
4. 该站点和该线路当前时段是否属于高客流
5. 途经道路是否拥堵
6. 用户更重视快、舒适、少走路，还是少换乘
7. 哪条路线综合体验最好
8. 为什么推荐这条路线
```

## (三) 核心数据来源

| **数据源** | **关键字段** | **使用方式** | **算法环节** |
|---|---|---|---|
| Bus Arrival | `EstimatedArrival`、`Load`、`Latitude`、`Longitude`、`Monitored` | 获取实时 ETA、车辆位置、客载状态和可信度 | 实时公交状态提取 |
| Bus Stops | `BusStopCode`、`RoadName`、`Description`、`Latitude`、`Longitude` | 站点匹配、附近站点查询、地图展示 | 候选路线生成 |
| Bus Routes | `ServiceNo`、`Direction`、`StopSequence`、`BusStopCode`、`Distance` | 构建线路和站点顺序关系 | 候选路线生成 |
| Bus Services | `ServiceNo`、`Operator`、`OriginCode`、`DestinationCode`、发车频率 | 获取线路起终点、发车频率、运营信息 | 时间估算和缺失降级 |
| Passenger Volume by Bus Stops | `PT_CODE`、`TIME_PER_HOUR`、上下车量 | 计算站点历史客流压力 | 历史客流分析 |
| Passenger Volume by OD Bus Stops | `ORIGIN_PT_CODE`、`DESTINATION_PT_CODE`、`TOTAL_TRIPS` | 计算起终点需求强度 | 可选扩展 |
| Traffic Speed Bands | `link_id`、`speed_band`、`road_name`、坐标 | 计算道路拥堵热力 | 路况压力分析 |
| Weather API | 天气状态、降雨、温度、湿度 | 修正步行和换乘权重 | 可选扩展 |
| 用户行为数据 | 历史选择、收藏路线、常用站点、偏好设置 | 动态调整推荐权重 | 可选扩展 |

第一阶段建议优先使用 LTA 核心数据：

```text
Bus Arrival
Bus Stops
Bus Routes
Bus Services
Passenger Volume by Bus Stops
Traffic Speed Bands
```

天气和用户行为可以作为第二阶段增强项，不阻塞第一阶段实现

## (四) 技术亮点

该方案即使不训练 ETA/Load 模型，也有足够技术含量：

- 多源数据融合：实时到站、实时客载、历史客流、道路速度热力共同参与推荐
- 多目标评价：不只看最快，还看舒适、步行、换乘和可靠性
- 动态偏好权重：不同用户偏好会得到不同排序结果
- 可解释推荐：输出每条路线的分项得分和推荐原因
- 数据缺失降级：实时数据缺失时用发车频率、历史客流或默认值兜底
- 可扩展路径搜索：第一阶段可简化实现，后续扩展改进 A*、TOPSIS 或 Pareto 多目标优化

# 二、算法实现

## (一) 总体流程

系统完整算法流程如下：

```text
用户输入起点、终点和出行偏好
        ↓
根据坐标匹配附近公交站点
        ↓
基于 Bus Routes / Bus Services 生成候选公交路线
        ↓
调用 Bus Arrival 获取实时 ETA、Load、车辆位置、Monitored
        ↓
读取 Passenger Volume 计算站点与线路客流压力
        ↓
读取 Traffic Speed Bands 计算道路拥堵压力
        ↓
计算每条候选路线的分项得分
        ↓
根据用户偏好动态调整权重
        ↓
计算 recommend_score
        ↓
Top-K 排序输出推荐路线
        ↓
生成结构化推荐原因
        ↓
大模型将结构化结果转为自然语言解释
```

## (二) 输入与输出

算法输入：

```text
用户起点坐标或起点站
用户终点坐标或终点站
出行时间
用户偏好
实时 Bus Arrival 数据
静态线路站点数据
历史站点客流数据
道路速度热力数据
```

算法输出：

```text
候选路线列表
每条路线的分项得分
每条路线的综合推荐分
最快路线
车内更宽松路线
少换乘路线
少步行路线
综合推荐路线
结构化推荐原因
自然语言解释上下文
```

## (三) 实时公交状态提取

实时公交状态提取模块负责从 LTA Bus Arrival API 获取当前公交运行状态，并转换为推荐特征

字段转换关系：

| **原始字段** | **推荐特征** | **说明** |
|---|---|---|
| `EstimatedArrival` | `eta_minutes` | 当前查询时刻到预计到站时间的分钟差 |
| `Load` | `load_code`、`load_score` | 实时客载状态和舒适度分数 |
| `Latitude`、`Longitude` | `vehicle_location` | 车辆实时位置 |
| `Monitored` | `monitored_score` | 实时监控可信度 |
| `query_time` | `data_freshness_seconds` | 数据新鲜度 |

ETA 计算：

```text
eta_minutes = EstimatedArrival - query_time
```

注意：

```text
eta_minutes 当前来自实时接口解析，不是机器学习模型预测结果
```

Load 映射：

| **Load 编码** | **含义** | **项目展示** | **load_score** |
|---|---|---|---:|
| `SEA` | Seats Available | 有座位，较宽松 | 100 |
| `SDA` | Standing Available | 可站立，适中 | 70 |
| `LSD` | Limited Standing | 站立空间有限，较拥挤 | 35 |
| `Unknown` | 暂无数据 | 客载状态未知 | 60 |

## (四) 候选公交路线生成

候选路线生成模块用于根据用户起点和终点生成若干条可达公交方案

第一阶段建议采用简化路径搜索：

```text
1. 根据用户起点匹配附近公交站点
2. 根据用户终点匹配附近公交站点
3. 根据 Bus Routes 查询经过起点附近站和终点附近站的线路
4. 优先生成直达路线
5. 若无直达路线，再生成一次换乘路线
6. 输出 3-5 条候选路线
```

每条候选路线应包含：

```text
route_id
service_no
start_stop_code
end_stop_code
passed_stops
walk_distance_meters
walk_time_minutes
ride_time_minutes
transfer_count
total_time_minutes
```

第二阶段可扩展为改进 A* 候选路径搜索

传统 A*：

```text
f(n) = g(n) + h(n)
```

BusMind 改进形式：

```text
f(n) =
α × TimeCost(n)
+ β × CrowdCost(n)
+ γ × WalkCost(n)
+ δ × TransferCost(n)
+ ε × CongestionCost(n)
+ λ × PreferencePenalty(n)
+ h(n)
```

各成本项含义：

| **成本项** | **含义** |
|---|---|
| `TimeCost` | ETA、车内行驶时间、总耗时 |
| `CrowdCost` | 实时客载状态和历史客流压力 |
| `WalkCost` | 步行距离和步行时间 |
| `TransferCost` | 换乘次数和换乘复杂度 |
| `CongestionCost` | 道路热力和速度拥堵压力 |
| `PreferencePenalty` | 与用户偏好不匹配的惩罚 |
| `h(n)` | 当前节点到终点的预估剩余距离或时间 |

## (五) 历史客流压力分析

历史客流压力分析用于判断站点、线路或 OD 方向在当前时段是否容易人多

它不预测某辆车上的具体人数，而是判断：

```text
当前站点 / 当前线路 / 当前方向 / 当前时段是否容易出现高客流
```

站点客流：

```text
station_flow = TOTAL_TAP_IN_VOLUME + TOTAL_TAP_OUT_VOLUME
```

按照同一日期类型和同一小时分位数划分：

```text
低客流：0% - 33%
中客流：33% - 66%
高客流：66% - 100%
```

客流等级映射：

| **flow_level** | **含义** | **flow_score** |
|---|---|---:|
| `low` | 低客流 | 100 |
| `medium` | 中客流 | 70 |
| `high` | 高客流 | 40 |

路线历史客流得分：

```text
route_flow_score = avg(flow_score of passed_stops)
```

如果有 OD 数据：

```text
history_flow_score =
0.7 × route_flow_score
+ 0.3 × od_score
```

如果没有 OD 数据：

```text
history_flow_score = route_flow_score
```

## (六) 道路热力与拥堵分析

道路热力模块用于判断候选路线途经道路是否拥堵，从而辅助计算车内行驶时间和推荐分数

使用数据：

```text
Traffic Speed Bands
```

主要字段：

```text
road_segment_id
road_name
speed_band
coordinates
updated_time
```

拥堵得分映射：

| **路况状态** | **含义** | **congestion_score** |
|---|---|---:|
| 畅通 | 速度较高 | 100 |
| 轻微拥堵 | 中等速度 | 75 |
| 明显拥堵 | 低速 | 50 |
| 严重拥堵 | 极低速 | 25 |
| Unknown | 未知 | 60 |

路线拥堵得分：

```text
route_congestion_score = avg(congestion_score of matched_road_segments)
```

如果无法精确匹配道路段：

```text
route_congestion_score = 60
```

该字段主要影响：

```text
ride_time_minutes
time_score
comfort_score
reliability_score
recommend_score
```

## (七) 推荐特征整理

每条候选路线最终整理为统一特征对象：

| **特征类别** | **字段** | **来源** | **用途** |
|---|---|---|---|
| 时间效率 | `eta_minutes` | Bus Arrival | 等车时间 |
| 时间效率 | `ride_time_minutes` | Bus Routes / 道路速度 | 车内行驶时间 |
| 时间效率 | `total_time_minutes` | 计算字段 | 最快方案排序 |
| 服务稳定性 | `avg_service_frequency` | Bus Services | 发车频率评价 |
| 舒适度 | `load_code` | Bus Arrival | 车辆客载状态 |
| 舒适度 | `load_score` | Load 映射 | 舒适评分 |
| 客流压力 | `tap_in_volume`、`tap_out_volume` | Passenger Volume | 站点客流趋势 |
| 客流压力 | `history_flow_score` | 计算字段 | 历史客流压力 |
| 路况压力 | `speed_band` | Traffic Speed Bands | 道路速度热力 |
| 路况压力 | `congestion_score` | speed_band 映射 | 路况惩罚 |
| 可达性 | `walk_time_minutes` | 坐标距离或地图 API | 步行负担 |
| 可达性 | `transfer_count` | 候选路线结构 | 换乘复杂度 |
| 可靠性 | `data_freshness_seconds` | query_time | 实时数据新鲜度 |
| 可靠性 | `monitored` | Bus Arrival | 数据可信度 |

## (八) 多目标出行体验评价

推荐评价模型将不同维度统一转化为 0-100 分

分项得分包括：

```text
time_score
comfort_score
walk_score
transfer_score
reliability_score
```

时间成本：

```text
total_time_minutes = walk_time_minutes + eta_minutes + ride_time_minutes
```

时间得分可采用线性扣分：

```text
time_score = max(0, 100 - total_time_minutes × 2)
```

也可在候选路线内部做归一化：

```text
time_score = 100 × (max_time - current_time) / (max_time - min_time)
```

舒适度得分：

```text
comfort_score =
0.5 × load_score
+ 0.3 × history_flow_score
+ 0.2 × congestion_score
```

步行得分：

```text
walk_score = max(0, 100 - walk_time_minutes × 8)
```

换乘得分：

```text
0 次换乘：100
1 次换乘：75
2 次换乘：50
3 次及以上：30
```

可靠性得分：

```text
reliability_score =
0.5 × freshness_score
+ 0.3 × monitored_score
+ 0.2 × completeness_score
```

数据新鲜度示例：

```text
data_freshness_seconds ≤ 60：freshness_score = 100
60-180 秒：freshness_score = 80
180-300 秒：freshness_score = 60
> 300 秒：freshness_score = 40
```

## (九) 综合推荐评分

基础公式：

```text
recommend_score =
w_time × time_score
+ w_comfort × comfort_score
+ w_walk × walk_score
+ w_transfer × transfer_score
+ w_reliability × reliability_score
```

约束：

```text
w_time + w_comfort + w_walk + w_transfer + w_reliability = 1
```

默认权重：

| **指标** | **默认权重** | **说明** |
|---|---:|---|
| `time_score` | 0.30 | 时间效率 |
| `comfort_score` | 0.30 | 车内体验、客流压力、路况压力 |
| `walk_score` | 0.15 | 步行负担 |
| `transfer_score` | 0.15 | 换乘复杂度 |
| `reliability_score` | 0.10 | 数据可靠性 |

## (十) 用户偏好动态权重

用户偏好权重：

| **用户偏好** | **w_time** | **w_comfort** | **w_walk** | **w_transfer** | **w_reliability** | **说明** |
|---|---:|---:|---:|---:|---:|---|
| `balanced` | 0.30 | 0.30 | 0.15 | 0.15 | 0.10 | 综合推荐 |
| `fastest` | 0.55 | 0.15 | 0.10 | 0.10 | 0.10 | 更重视最快到达 |
| `comfort` | 0.20 | 0.55 | 0.10 | 0.05 | 0.10 | 更重视车内舒适 |
| `less_walk` | 0.25 | 0.20 | 0.40 | 0.05 | 0.10 | 更重视少步行 |
| `less_transfer` | 0.25 | 0.20 | 0.10 | 0.35 | 0.10 | 更重视少换乘 |

历史行为反馈可作为第二阶段扩展：

```text
如果用户连续多次选择最快路线：w_time += 0.05
如果用户连续多次选择低客载路线：w_comfort += 0.05
如果用户连续多次选择少步行路线：w_walk += 0.05
如果用户连续多次选择少换乘路线：w_transfer += 0.05
```

调整后统一归一化：

```text
normalized_weight_i = weight_i / sum(all_weights)
```

天气修正也作为第二阶段扩展

例如中雨时：

```text
w_walk += 0.05
w_transfer += 0.03
```

最后同样进行权重归一化

## (十一) 推荐类型输出

系统不只输出一个路线，而是输出多个推荐标签，方便用户比较

| **推荐类型** | **选择规则** | **展示名称** |
|---|---|---|
| `best_experience` | 当前用户偏好下 `recommend_score` 最高 | 综合推荐 |
| `fastest` | `total_time_minutes` 最小 | 最快到达 |
| `least_crowded` | `comfort_score` 最高 | 车内更宽松 |
| `least_transfer` | `transfer_count` 最少，相同则 `recommend_score` 更高 | 换乘更少 |
| `least_walking` | `walk_time_minutes` 最短，相同则 `recommend_score` 更高 | 步行更少 |
| `most_reliable` | `reliability_score` 最高 | 数据更可靠 |

推荐规则示例：

```text
best_experience_route = max(routes, key=recommend_score)
fastest_route = min(routes, key=total_time_minutes)
least_crowded_route = max(routes, key=comfort_score)
least_transfer_route = min(routes, key=(transfer_count, -recommend_score))
least_walking_route = min(routes, key=(walk_time_minutes, -recommend_score))
```

## (十二) 数据缺失降级

真实接口可能出现数据缺失，因此需要明确降级策略

| **缺失数据** | **降级策略** |
|---|---|
| 缺少 `EstimatedArrival` | 使用线路平均发车间隔估算等待时间 |
| 缺少 `Load` | 使用站点历史客流、高峰时段和线路热度估算客载压力，并降低可靠性得分 |
| 缺少 Traffic Speed Bands | 使用线路平均速度或道路段平均速度 |
| 缺少 OD 客流 | 不阻塞推荐，仅使用站点客流 |
| 缺少 Passenger Volume | 使用默认 `flow_score = 60` |
| 缺少天气数据 | 按普通天气处理 |
| 缺少步行路径 | 使用经纬度直线距离乘绕行系数估算 |
| 缺少 `Monitored` 字段 | `monitored_score = 60` |
| 实时数据过旧 | 降低 `reliability_score` |

发车频率估算等待时间：

```text
estimated_wait_time = avg_service_frequency / 2
```

缺少 Load 时的舒适度兜底：

```text
if peak_hour and flow_level == "high":
    load_score = 40
elif flow_level == "medium":
    load_score = 70
else:
    load_score = 60
```

## (十三) 推荐理由生成

推荐模块应先生成结构化原因，再交给大模型生成自然语言

结构化推荐原因示例：

```json
{
  "main_reason": "该路线综合体验评分最高",
  "eta_reason": "预计 6 分钟后到站，等待时间较短",
  "comfort_reason": "当前车辆客载状态为有座位，历史客流压力较低",
  "traffic_reason": "途经道路拥堵程度较低",
  "walk_reason": "步行距离较短",
  "transfer_reason": "无需换乘",
  "reliability_reason": "实时数据更新时间较近，可信度较高"
}
```

自然语言模板：

```text
推荐乘坐 {service_no} 路。该路线预计 {eta_minutes} 分钟后到达上车站，
当前车辆客载状态为 {load_text}，历史客流压力为 {flow_level}，
途经道路拥堵程度为 {congestion_level}，步行约 {walk_time_minutes} 分钟，
换乘 {transfer_count} 次。综合考虑你的 {preference_text} 偏好，
该路线的推荐得分最高
```

大模型边界：

```text
大模型不直接计算 ETA
大模型不预测 Load
大模型不修改 recommend_score
大模型只读取结构化推荐结果并生成自然语言解释
```

## (十四) 推荐接口输出示例

推荐接口可以返回如下结构：


## (十六) 第一阶段实现重点

第一阶段优先完成：

```text
1. 导入 Bus Stops、Bus Routes、Bus Services 基础数据
2. 根据用户起终点生成 3-5 条候选公交路线
3. 调用或模拟 Bus Arrival，获取 eta_minutes、Load、车辆位置
4. 将 Load 映射为 load_score
5. 使用 Passenger Volume by Bus Stops 计算 history_flow_score
6. 接入或模拟 Traffic Speed Bands，计算 congestion_score
7. 计算 time_score、comfort_score、walk_score、transfer_score、reliability_score
8. 根据用户偏好设置推荐权重
9. 计算 recommend_score
10. 对候选路线进行 Top-K 排序
11. 输出最快、最舒适、少换乘、少步行、综合推荐等结果
12. 生成结构化推荐原因
13. 由大模型生成自然语言解释
```

第一阶段交付内容：

| **输出内容** | **说明** |
|---|---|
| 候选路线列表 | 至少 3 条可比较路线 |
| 实时公交状态 | `eta_minutes`、`Load`、车辆位置 |
| 历史客流压力 | `history_flow_score`、`flow_level` |
| 道路热力结果 | `congestion_score`、`congestion_level` |
| 出行体验分项得分 | time、comfort、walk、transfer、reliability |
| 个性化推荐结果 | `recommend_score`、Top-K routes |
| 推荐理由 | 结构化原因 + 自然语言解释 |

## (十七) 第二阶段扩展方向

第二阶段可以增强以下能力：

- 完整改进 A* 路径搜索
- TOPSIS 多指标决策
- Pareto 前沿筛选
- 用户历史行为权重学习
- 天气环境权重修正
- OD 客流压力分析
- 轻量级站点客流预测模型

其中，客流预测模型可以作为后续增强，不作为第一阶段核心

如果真的要训练模型，可以考虑这些：

| **可训练任务**   | **训练数据来源**                 | **标签怎么来**                 |
| ---------------- | -------------------------------- | ------------------------------ |
| 站点客流压力预测 | Passenger Volume by Bus Stops    | tap_in + tap_out 或 flow_level |
| OD 需求强度预测  | Passenger Volume by OD Bus Stops | TOTAL_TRIPS 或 od_level        |
| 道路拥堵趋势预测 | Traffic Speed Bands 连续采样     | speed_band 或 congestion_score |
| 用户偏好权重学习 | 用户最终选择路线记录             | 用户选了哪条路线               |
| 推荐排序优化     | 推荐曝光和点击/选择日志          | 是否被用户选择                 |

# 三、总结

BusMind 2.1 核心算法可以总结为：

```text
基于多源数据融合的个性化公交出行推荐算法
```

完整说明：

```text
系统首先基于公交站点和线路数据生成候选路线，并通过 LTA Bus Arrival 获取实时 ETA、车辆位置和客载状态；随后结合 Passenger Volume 历史客流数据和 Traffic Speed Bands 道路热力数据，计算站点客流压力、道路拥堵压力和数据可靠性；再将时间效率、舒适度、步行负担、换乘复杂度和可靠性转化为多目标评价指标；最后结合用户偏好动态调整权重，对候选路线进行 Top-K 排序，并通过大模型生成自然语言推荐解释
```
