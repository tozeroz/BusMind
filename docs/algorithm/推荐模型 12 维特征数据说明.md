# 推荐模型 12 维特征数据说明

## 1. 总体结论

当前推荐模型评分层固定使用 12 个数值特征。模型不负责采集数据、不查库、不调 LTA API；但模型侧包含预处理层，会把后端传入的硬性业务字段映射成 12 维数值特征。后端对外调用 `predict_recommendation` 时不要传 `load_score`、`history_flow_score`、`congestion_score`、`data_freshness_seconds`、`monitored_score`、`completeness_score`。

离线训练阶段：

```text
data/raw/lta
    ↓
data/processed/process_scripts/build_lta_processed_data.py
    ↓
data/processed/*.csv
    ↓
algorithm/dataset/build_features.py
    ↓
algorithm/dataset/recommendation/v1/features.csv
```

线上调用阶段：

```text
后端读取 MySQL / 缓存 / LTA 实时刷新结果
    ↓
后端组装候选路线业务字段
    ↓
algorithm/model/preprocessing.py 映射为 12 维路线特征
    ↓
algorithm.model.predictor.predict_recommendation(payload)
```

当前仓库没有提交真实 `data/processed/*.csv`，只有处理脚本和 raw 样本。因此离线脚本的读取顺序是：优先读 `data/processed`，如果缺失则从 `data/raw/lta` 按同一口径重建。

`algorithm/dataset/recommendation/v1/features.csv` 是推荐模型 v1 的训练特征文件。真实来源字段、估算字段和降级字段必须通过 `feature_sources`、`degraded_fields`、`is_synthetic` 区分。

## 2. 12 维特征来源表

| 序号 | 模型字段 | 离线训练来源 | 线上调用来源 | 当前处理口径 | 当前状态 |
|---:|---|---|---|---|---|
| 1 | `eta_minutes` | `lta_bus_arrival.csv.eta_minutes`；若无 processed，则读 `bus_arrival_samples/**/*.jsonl` 的 `EstimatedArrival` 和 `query_time` | 后端从 LTA Bus Arrival 实时结果计算 | `eta_minutes = EstimatedArrival - query_time`，单位分钟 | 已有真实来源 |
| 2 | `ride_time_minutes` | `line_station.csv.route_distance_km` 或 `map_road_segment.csv.ride_time_minutes` | 后端候选路线生成时按线路站序、距离或地图路径聚合 | 当前训练脚本用剩余线路距离按 18km/h 估算 | 可用但偏估算 |
| 3 | `walk_time_minutes` | 当前离线没有真实步行路径，训练脚本按候选组做模拟 | 后端路径规划或站点距离计算结果 | 当前默认 0；若候选组无差异，则生成 0/3/6/9 分钟模拟值 | 需要后端补真实值 |
| 4 | `walk_distance_meters` | 由 `walk_time_minutes * 80` 估算 | 后端路径规划或站点距离计算结果 | 当前按步行速度约 80m/min 反推 | 需要后端补真实值 |
| 5 | `transfer_count` | 当前离线候选多为单站候选，训练脚本按候选组模拟 | 后端候选路线生成时直接给出 | 当前默认 0；若候选组无差异，则生成 0/1/2 模拟值 | 需要后端补真实值 |
| 6 | `load_score` | `lta_bus_arrival.csv.load_score`；若无 processed，则读 `Load` | 后端必须传 `load_code` | `SEA=100, SDA=70, LSD=35, UNKNOWN=60` | 已有真实来源，映射在模型侧 |
| 7 | `history_flow_score` | `passenger_flow_trend.csv.total_flow / flow_level`；若无 processed，则读 `passenger_volume_stop/<month>/original.zip` | 后端必须传 `station_flow_level` | `low=90, medium=70, high=45` | 已有真实来源，映射在模型侧 |
| 8 | `congestion_score` | `traffic_speed_bands.csv.speed_band / congestion_score`；若无 processed，则读 `traffic_speed_bands/**/*.json` | 后端必须传路线聚合后的 `route_speed_band` | LTA `SpeedBand=1..8` 映射为 30..100 通畅分 | 已有真实来源，路线匹配较粗，映射在模型侧 |
| 9 | `data_freshness_seconds` | 当前离线训练脚本固定为 60 | 后端必须传 `source_updated_at` | `now - source_updated_at`，单位秒 | 离线为默认值，线上映射在模型侧 |
| 10 | `monitored_score` | `lta_bus_arrival.csv.monitored` | 后端必须传 `monitored` | `Monitored=1 -> 100`，否则为 75 | 已统一，映射在模型侧 |
| 11 | `completeness_score` | 训练脚本根据 `degraded_fields` 计算 | 后端必须传 `degraded_fields` | `100 - 降级字段数 * 6`，最低 55 | 已统一，映射在模型侧 |
| 12 | `avg_service_frequency_minutes` | `bus_line.csv.avg_service_frequency`；若无 processed，则读 `BusServices_full.json` 的四个频率字段 | 后端查 `bus_line.avg_service_frequency` | 解析 `AM/PM Peak/Offpeak Freq` 区间均值后再求平均 | 已有真实来源 |

12 个字段可以这样理解：

| 字段                            | 含义                                         | 越大代表                    |
| ------------------------------- | -------------------------------------------- | --------------------------- |
| `eta_minutes`                   | 公交预计多久到站                             | 等车更久，不一定好          |
| `ride_time_minutes`             | 上车后的乘车时间                             | 坐车更久，不一定好          |
| `walk_time_minutes`             | 从当前位置到站点、或下车后到目的地的步行时间 | 步行更久，不好              |
| `walk_distance_meters`          | 步行距离                                     | 走得更远，不好              |
| `transfer_count`                | 换乘次数                                     | 换乘更多，不好              |
| `load_score`                    | 车内客载舒适度评分                           | 越高越宽松，越好            |
| `history_flow_score`            | 站点历史客流压力评分                         | 越高代表客流压力越低，越好  |
| `congestion_score`              | 道路通畅评分                                 | 越高代表越不堵，越好        |
| `data_freshness_seconds`        | 实时数据距离现在过去了多久                   | 越大代表数据越旧，不好      |
| `monitored_score`               | LTA 实时监控可信度                           | 越高越可信，越好            |
| `completeness_score`            | 当前路线特征完整度                           | 越高代表缺失/估算越少，越好 |
| `avg_service_frequency_minutes` | 线路平均发车间隔                             | 越大代表班次越稀疏，不好    |

## 3. 原始数据字段对应关系

### 3.1 静态公交网络

来自：

```text
data/raw/lta/api_response/BusStops_full.json
data/raw/lta/api_response/BusRoutes_full.json
data/raw/lta/api_response/BusServices_full.json
```

关键字段：

| 原始文件 | 原始字段 | 处理后用途 |
|---|---|---|
| `BusStops_full.json` | `BusStopCode`、`RoadName`、`Latitude`、`Longitude` | 站点 ID、道路名、站点坐标 |
| `BusRoutes_full.json` | `ServiceNo`、`Direction`、`StopSequence`、`BusStopCode`、`Distance` | 线路站序、剩余距离、车内时间估算 |
| `BusServices_full.json` | `AM_Peak_Freq`、`AM_Offpeak_Freq`、`PM_Peak_Freq`、`PM_Offpeak_Freq` | 平均发车间隔 |

### 3.2 实时到站和客载

来自：

```text
data/raw/lta/bus_arrival_samples/YYYY-MM-DD/bus_arrival_sample.jsonl
```

关键字段：

| 原始字段 | 处理后字段 | 模型用途 |
|---|---|---|
| `query_time` | `query_time` | ETA 和数据新鲜度计算 |
| `NextBus/NextBus2/NextBus3.EstimatedArrival` | `eta_minutes` | 实时 ETA |
| `NextBus/NextBus2/NextBus3.Load` | `load_code`、`load_score` | 客载舒适度 |
| `NextBus/NextBus2/NextBus3.Monitored` | `monitored_score` | 实时可信度 |
| `NextBus/NextBus2/NextBus3.OriginCode/DestinationCode` | `line_id` 匹配 | 匹配具体线路方向 |

ETA 和 Load 不作为训练目标，只作为推荐模型输入特征。

### 3.3 历史客流

来自：

```text
data/raw/lta/passenger_volume_stop/<month>/original.zip
```

关键字段：

| 原始字段 | 处理后字段 | 模型用途 |
|---|---|---|
| `PT_CODE` | `station_id` / `bus_stop_code` | 匹配公交站点 |
| `DAY_TYPE` | `day_type` | 区分 weekday / weekend |
| `TIME_PER_HOUR` | `hour` | 同小时聚合 |
| `TOTAL_TAP_IN_VOLUME` | `tap_in_volume` | 进站客流 |
| `TOTAL_TAP_OUT_VOLUME` | `tap_out_volume` | 出站客流 |
| tap in + tap out | `total_flow` | 历史客流压力 |

当前推荐模型不做客流时间序列预测，先使用上个月同站点、同小时、同 day_type 的 `total_flow` 均值，并映射成 `history_flow_score`。

### 3.4 道路速度热力

来自：

```text
data/raw/lta/traffic_speed_bands/YYYY-MM-DD/*.json
```

关键字段：

| 原始字段 | 处理后字段 | 模型用途 |
|---|---|---|
| `LinkID` | `link_id` | 道路段编号 |
| `RoadName` | `road_name` | 与站点道路名粗匹配 |
| `SpeedBand` | `congestion_score` | 道路拥堵/通畅特征 |
| `MinimumSpeed` / `MaximumSpeed` | `minimum_speed_kmh` / `maximum_speed_kmh` | 可用于后续更细的速度估算 |
| `StartLat/StartLon/EndLat/EndLon` | 道路段坐标 | 地图展示和路线匹配 |

当前脚本先按道路名把线路站点和 Traffic Speed Bands 做粗匹配，再按线路求均值。后续可以升级为基于坐标和线路 polyline 的空间匹配。

## 4. 当前脚本里的降级和模拟

当前 `algorithm/dataset/recommendation_data.py` 中仍有几项是为了打通离线训练链路而设置的临时口径：

| 字段 | 当前离线处理 | 风险 |
|---|---|---|
| `walk_time_minutes` | 默认 0；无差异时模拟 0/3/6/9 | 不代表真实步行路径 |
| `walk_distance_meters` | `walk_time_minutes * 80` | 依赖模拟步行时间 |
| `transfer_count` | 默认 0；无差异时模拟 0/1/2 | 不代表真实换乘路线 |
| `data_freshness_seconds` | 固定 60 | 没有反映真实缓存年龄 |
| `completeness_score` | 按 `degraded_fields` 数量计算 | 当前为线性扣分规则 |
| `congestion_score` | 按道路名粗匹配后线路均值 | 不能精确表示公交实际经过道路 |

这些字段可以用于第一版训练链路，但答辩或文档中需要说明：静态数据、ETA、Load、客流、Traffic Speed Bands 有真实来源；步行、换乘、数据新鲜度、完整度在离线阶段先用规则估算，线上由后端补齐。

真实来源字段：

```text
eta_minutes
load_score
history_flow_score
congestion_score
avg_service_frequency_minutes
```

估算或过渡字段：

```text
ride_time_minutes
walk_time_minutes
walk_distance_meters
transfer_count
data_freshness_seconds
completeness_score
```

辅助追踪字段：

```text
feature_sources
degraded_fields
is_synthetic
```
