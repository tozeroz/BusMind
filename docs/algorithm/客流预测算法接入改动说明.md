# 客流预测算法接入改动说明

## 目标

在不改前端页面展示、不修改 `algorithm/` 目录代码的前提下，为现有管理员客流页面补齐站点级客流预测能力，让 `/api/v1/history/passenger-flow/prediction` 能返回真实预测数据。

## 本次改动

1. 新增后端预测服务 `backend/app/services/passenger_flow_forecast_service.py`
   - 读取 `passenger_flow_trend`
   - 生成未来 7 天、按小时的站点客流预测
   - 写入 `passenger_flow_prediction`
   - 默认优先使用 Ridge 思路的多元线性回归
   - 为了避免新增 `scikit-learn` 依赖，后端直接用纯 Python 实现了小规模 Ridge 线性方程求解
   - 当单站历史样本过少或矩阵不可解时，自动回退到历史均值模型，保证接口尽量有数据可用

2. 增强既有历史查询服务 `backend/app/services/history_service.py`
   - 当现有前端请求某个站点的预测数据，且未传时间范围时
   - 后端会先检查该站点未来 7 天预测是否已存在
   - 若不存在或已过期，则即时生成并落库
   - 之后继续走原有 DTO 和接口返回逻辑

3. 新增批量脚本 `backend/scripts/forecast_passenger_flow.py`
   - 支持一次性为全部有历史数据的站点生成预测
   - 也支持通过 `--station-id` 指定单个或多个站点

4. 更新 `backend/scripts/README.md`
   - 补充新脚本用途说明

## 未改动内容

- 未修改任何前端页面代码
- 未修改 `algorithm/` 目录
- 未改动现有 `/api/v1/history/passenger-flow/prediction` 接口入参和返回结构
- 未改动 `PassengerFlowPrediction` ORM 结构

## 算法说明

### 输入数据

- 表：`passenger_flow_trend`
- 过滤：`target_type = 'station'`
- 训练窗口：默认回看最近 90 天历史

### 特征

- `hour`
- `weekday`
- `hour^2`
- `sin(2π*hour/24)`
- `cos(2π*hour/24)`
- `sin(2π*weekday/7)`
- `cos(2π*weekday/7)`
- 常数项

### 输出数据

- 表：`passenger_flow_prediction`
- 粒度：未来 7 天、每小时 1 条
- `target_type = 'station'`
- `target_id = station_id`
- `prediction_time = 本次生成批次时间`
- `predict_time = 被预测的未来小时`

### 拥挤等级

- `predicted_flow >= 100` -> `high`
- `predicted_flow >= 50` -> `medium`
- 其他 -> `low`

## 运行方式

### 方式一：页面懒生成

管理员页面选择站点时，后端会自动补生成预测，无需改前端。

### 方式二：手动批量预热

在项目根目录执行：

```powershell
python backend\scripts\forecast_passenger_flow.py
```

只生成指定站点：

```powershell
python backend\scripts\forecast_passenger_flow.py --station-id 1001 --station-id 1002
```

## 自测结果

本次已补充自动化测试 `backend/tests/test_passenger_flow_forecast.py`，覆盖：

1. 单站预测生成后应写入 168 条未来 7 天小时级记录
2. 同一站点重复请求时不应重复插入
3. 历史查询接口在预测表为空时可自动补数并返回结果

建议上线前再执行一次批量脚本，提前把高频站点的预测数据写入库中，避免管理员首次点击站点时需要等待即时生成。
