# 离线训练数据需求清单

> 适用范围：`ly` 分支当前阶段只做**离线训练**，不依赖后端实时链路完全跑通。训练优先级按“客流趋势预测 -> 道路拥堵趋势预测 -> 推荐排序伪标签训练 -> 用户偏好学习”推进。ETA 和 Load 不再作为第一阶段训练任务

## 一、当前训练原则

- **不训练 ETA 模型**
  - `eta_minutes` 直接来自 `LTA Bus Arrival.EstimatedArrival - query_time`
- **不训练 Load 三分类模型**
  - `load_code` 直接来自 `LTA Bus Arrival.Load`
- **优先训练可增强推荐质量的离线模型**
  - 站点客流趋势预测
  - 道路拥堵趋势预测
  - 推荐排序权重学习
  - 用户偏好学习

## 二、现在能从 `data` 目录直接拿到的原始数据

### 1. 静态公交基础数据

路径：

```text
data/raw/lta/api_response/BusStops_full.json
data/raw/lta/api_response/BusRoutes_full.json
data/raw/lta/api_response/BusServices_full.json
```

用途：

- 站点基础属性
- 线路与站序关系
- 发车频率和起终点信息
- 候选路线生成的静态底表

### 2. 站点客流月度数据

路径：

```text
data/raw/lta/passenger_volume_stop/202605/original.zip
data/raw/lta/api_response/PassengerVolumeByBusStops_202605.json
```

用途：

- 训练站点客流趋势模型
- 计算站点历史拥挤压力
- 给 Load 缺失时做舒适度兜底特征

### 3. OD 客流月度数据

路径：

```text
data/raw/lta/passenger_volume_od/202605/original.zip
data/raw/lta/api_response/PassengerVolumeByOriginDestinationBusStops_202605.json
```

用途：

- 训练 OD 需求强度模型
- 后续增强候选路线生成和起终点需求估计

### 4. Bus Arrival 动态样本

路径：

```text
data/raw/lta/bus_arrival_samples/2026-07-08/bus_arrival_sample.jsonl
```

用途：

- 提取 `eta_minutes`
- 提取 `load_code` / `load_score`
- 提取车辆位置和 `monitored` 可信度
- 为推荐排序样本构造实时状态特征

### 5. Traffic Speed Bands 动态样本

路径：

```text
data/raw/lta/traffic_speed_bands/2026-07-09/traffic_speed_bands_20260709_154158.json
```

用途：

- 训练道路拥堵趋势模型
- 生成 `congestion_score`
- 为推荐排序样本构造路况压力特征

## 三、processed 数据口径

当前 `ly` 分支工作区里 **`data/processed` 实际 CSV 可能还没生成出来**，但已经有对应处理脚本：

```text
data/processed/process_scripts/build_lta_processed_data.py
```

处理脚本目标输出：

```text
data/processed/bus_station.csv
data/processed/bus_line.csv
data/processed/line_station.csv
data/processed/bus_vehicle.csv
data/processed/lta_bus_arrival.csv
data/processed/bus_eta_status.csv
data/processed/bus_load_status.csv
data/processed/map_station.csv
data/processed/map_road_segment.csv
data/processed/passenger_flow_trend.csv
data/processed/traffic_speed_bands.csv
```

结论：

- **训练脚本优先读 `data/processed`**
- 如果 `processed` 不存在，先用 `build_lta_processed_data.py` 从 `raw` 重建
- 如果某类动态数据样本不足，就用模拟数据先跑通训练管道

## 四、各训练任务需要什么数据

### 1. 站点客流趋势预测

推荐优先级：**P0**

主数据来源：

```text
data/processed/passenger_flow_trend.csv
data/processed/map_station.csv
data/processed/line_station.csv
data/processed/bus_line.csv
```

如果 `processed` 不存在，可直接从：

```text
data/raw/lta/passenger_volume_stop/<month>/original.zip
```

训练目标：

- 回归：`total_flow`
- 分类：`flow_level`

建议特征：

- `target_id` / `bus_stop_code`
- `record_time`
- 小时、星期、是否工作日
- `tap_in_volume`
- `tap_out_volume`
- 历史 lag 特征：`t-1`、`t-2`、`t-24`
- 滚动均值：3 小时、6 小时、24 小时
- 站点线路数、服务数
- 周边线路平均发车频率

当前缺口：

- 现在只有 `202605` 一个月的站点客流，能做第一版训练，但时间跨度偏短

缺数据时的处理：

- 先做单月小时级时间序列 baseline
- 如果样本不够，先用规则阈值或滚动均值模型做伪基线

### 2. OD 需求强度预测

推荐优先级：**P2**

主数据来源：

```text
data/raw/lta/passenger_volume_od/202605/original.zip
```

训练目标：

- 回归：`TOTAL_TRIPS`
- 分类：`od_level`

建议特征：

- `ORIGIN_PT_CODE`
- `DESTINATION_PT_CODE`
- 小时、日期类型
- 起点站和终点站的线路数
- 两站静态距离

当前缺口：

- 当前项目主链路还没强依赖 OD 模型
- 只有一个月数据，优先级可以放后

缺数据时的处理：

- 暂不单独训练
- 先把它作为后续增强项保留

### 3. 道路拥堵趋势预测

推荐优先级：**P0**

主数据来源：

```text
data/processed/traffic_speed_bands.csv
data/processed/map_road_segment.csv
data/processed/bus_line.csv
```

如果 `processed` 不存在，可直接从：

```text
data/raw/lta/traffic_speed_bands/YYYY-MM-DD/*.json
```

训练目标：

- 分类：`speed_band`
- 回归：`congestion_score`

建议特征：

- `link_id`
- `road_name`
- `road_category`
- `query_time`
- 小时、星期
- `minimum_speed_kmh`
- `maximum_speed_kmh`
- 历史 lag 特征：最近 1 次、3 次、6 次采样
- 路段邻近 segment 的平均拥堵分

当前缺口：

- 当前仓库只有**极少量** Traffic Speed Bands 样本
- 不足以训练可靠的时序模型

缺数据时的处理：

- 先做“训练管道验证版”模拟数据
- 用现有单次或少量快照复制成多时间片，再加入轻微噪声，验证特征工程、训练、评估和模型落盘流程
- 这类模拟数据只用于**打通训练流程**，不能当正式模型结论

### 4. 推荐排序权重学习

推荐优先级：**P1**

主数据来源：

```text
data/processed/bus_line.csv
data/processed/line_station.csv
data/processed/lta_bus_arrival.csv
data/processed/passenger_flow_trend.csv
data/processed/traffic_speed_bands.csv
data/processed/map_road_segment.csv
```

训练样本的核心特征字典：

- `eta_minutes`
- `load_score`
- `walk_time`
- `transfer_count`
- `avg_service_frequency`
- `station_flow_score`
- `congestion_score`
- `reliability_score`
- `confidence`
- `data_source`

标签来源：

- 真实理想标签：用户最终选择了哪条路线
- 当前阶段没有真实曝光/点击/选择日志

缺数据时的处理：

- 先构造**伪标签排序数据集**
- 规则做法：用当前算法文档定义的多目标评分公式，对候选路线打分，把分数最高路线当正样本
- 再生成不同用户偏好模板：
  - `fastest`
  - `balanced`
  - `low_load`
  - `less_walking`
  - `less_transfer`

这样可以先训练：

- 逻辑回归 / GBDT 排序基线
- LambdaMART 风格排序样本

### 5. 用户偏好学习

推荐优先级：**P2**

真实所需数据：

- 推荐曝光日志
- 用户点击日志
- 用户最终路线选择日志
- 用户偏好设置历史

当前状态：

- 这些数据在 `data` 目录里**还没有**

缺数据时的处理：

- 先模拟用户画像
- 为每种画像定义权重偏好
- 用规则评分函数自动生成“该用户会选哪条路线”的伪标签

适合先产出的内容：

- `recommend_weight_config.json`
- 用户偏好模板配置
- 伪用户训练样本生成器

## 五、现在最适合先做的训练任务

按当前数据成熟度，建议顺序：

1. **站点客流趋势预测**
   - 数据最完整
   - 不依赖后端联调
   - 能直接增强 `station_flow_score`
2. **道路拥堵趋势预测训练管道**
   - 先用少量真实样本 + 模拟时间片把流程打通
   - 后续补采样后替换真实训练集
3. **推荐排序伪标签数据集生成**
   - 把推荐模型先从纯规则推进到“可训练”

## 六、暂时不建议做的事

- 不要再生成 ETA 训练集
- 不要再生成 Load 三分类训练集
- 不要把 `data/import` 下的小样例 CSV 当正式训练数据
- 不要等后端全跑通再开始训练准备

## 七、下一步落地建议

建议直接开三个离线脚本：

```text
algorithm/data_processing/build_flow_training_dataset.py
algorithm/data_processing/build_congestion_training_dataset.py
algorithm/data_processing/build_recommendation_training_dataset.py
```

它们的职责分别是：

- 从 `data/raw` 或 `data/processed` 读取原始数据
- 输出标准训练集到 `algorithm/model_files` 或 `data/processed`
- 支持“真实数据不足时自动模拟”模式

## 八、一句话结论

当前离线训练最现实的路线是：

```text
先用 Passenger Volume 做站点客流趋势模型
再用 Traffic Speed Bands 做拥堵趋势训练管道
最后用规则伪标签生成推荐排序训练集
ETA / Load 不进入第一阶段训练
```
