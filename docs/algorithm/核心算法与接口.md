# 核心算法与接口

## 1. 总体链路

```text
后端生成候选路线
    ↓
后端聚合每条候选路线的业务字段
    ↓
后端传入 payload
    ↓
模型预处理层 preprocessing
    - load_code -> load_score
    - speed_band / congestion_pressure -> congestion_score
    - station_flow_level -> history_flow_score
    - monitored -> monitored_score
    - query_time / source_updated_at -> data_freshness_seconds
    - degraded_fields -> completeness_score
    ↓
模型评分层 scorer
    - 12 维数值特征标准化
    - 五维评分头
    - 用户偏好融合
    ↓
模型返回每条候选路线的五维体验分 + 综合推荐分
    ↓
后端按分数排序、派生推荐类型、组织公开响应
```

模型和后端分工：

```text
后端负责：
  - 生成候选路线
  - 调 LTA API / 读缓存 / 查数据库
  - 聚合 ETA、历史客流、路况快照、步行、换乘、发车频率等业务字段
  - 调用模型并接收评分结果
  - 按 recommend_score 排序并组织最终接口响应

模型负责：
  - 校验后端传入的候选路线字段
  - 把可映射业务字段统一转换为 12 维数值特征
  - 对每条候选路线输出五维体验分
  - 根据用户偏好输出综合推荐分
  - 返回 feature_contributions 供解释层使用
```

一句话口径：

```text
后端给我候选路线和业务字段，我在模型预处理层转成 12 维特征，再给后端每条路线的评分结果。
```

模型侧不负责：

- 生成候选路线；
- 调 LTA API；
- 查数据库；
- 预测 ETA；
- 预测 Load；
- 生成最终自然语言解释。

## 2. 对外调用入口

```python
from algorithm.model.predictor import predict_recommendation

result = predict_recommendation(payload)
```

函数签名：

```python
predict_recommendation(payload: dict) -> dict
```

顶层 payload：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `contract_version` | `str` | 否 | 当前固定为 `1.0.0`，不传时使用默认值 |
| `request_id` | `str` | 是 | 后端请求 ID 或一次推荐请求的唯一 ID |
| `preference` | `str` | 否 | 用户偏好，不传默认为 `balanced` |
| `routes` | `list[dict]` | 是 | 候选路线业务字段列表，当前最多 10 条 |

`preference` 枚举：

```text
balanced
fastest
comfort
less_walking
less_transfer
```

兼容旧口径：

```text
low_load -> comfort
```

模型不接受后端传入 `weights`。后端只传 `preference`，模型内部根据偏好选择融合模板。

## 3. routes[] 后端 payload 硬性字段

后端调用 `predict_recommendation(payload)` 时，每条 `routes[]` 必须按下表传字段。后端不要传模型内部的分数字段。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `route_id` | `str` | 是 | 候选路线唯一 ID |
| `service_nos` | `list[str]` | 是 | 公交线路编号列表 |
| `eta_minutes` | `float` | 是 | 实时 ETA，越小越好 |
| `ride_time_minutes` | `float` | 是 | 车内时间，越小越好 |
| `walk_time_minutes` | `float` | 是 | 步行时间，越小越好 |
| `walk_distance_meters` | `float` | 是 | 步行距离，越小越好 |
| `transfer_count` | `int` | 是 | 换乘次数，越小越好 |
| `avg_service_frequency_minutes` | `float` | 是 | 平均发车间隔，越小越好 |
| `load_code` | `str` | 是 | `SEA` / `SDA` / `LSD` / `UNKNOWN` |
| `station_flow_level` | `str` | 是 | `low` / `medium` / `high` |
| `route_speed_band` | `int` | 是 | 后端按候选路线聚合后的 LTA SpeedBand，范围 1-8 |
| `source_updated_at` | `str` | 是 | 实时数据更新时间，ISO datetime 字符串 |
| `monitored` | `int` | 是 | LTA Bus Arrival `Monitored`，取值 0/1 |
| `degraded_fields` | `list[str]` | 是 | 缺失、估算或降级字段列表；没有降级时传空列表 |

后端禁止传以下模型内部字段：

```text
load_score
history_flow_score
congestion_score
data_freshness_seconds
monitored_score
completeness_score
```

这些字段由 `algorithm/model/preprocessing.py` 根据后端传入的原始业务字段统一生成。

预处理映射口径：

| 模型内部特征 | 后端字段 | 映射口径 |
|---|---|---|
| `load_score` | `load_code` | `SEA=100`，`SDA=70`，`LSD=35`，`UNKNOWN=60` |
| `history_flow_score` | `station_flow_level` | `low=90`，`medium=70`，`high=45` |
| `congestion_score` | `route_speed_band` | `SpeedBand=1..8` 映射为 30..100 通畅分 |
| `data_freshness_seconds` | `source_updated_at` | 当前时间减数据时间戳 |
| `monitored_score` | `monitored` | `1 -> 100`，`0 -> 75` |
| `completeness_score` | `degraded_fields` | `100 - 降级字段数 * 6`，最低 55 |

模型评分层内部固定使用以下 12 个数值特征：

```text
eta_minutes
ride_time_minutes
walk_time_minutes
walk_distance_meters
transfer_count
load_score
history_flow_score
congestion_score
data_freshness_seconds
monitored_score
completeness_score
avg_service_frequency_minutes
```

如果后端缺少任一硬性字段，或者传入了禁止字段，模型会抛出 `ModelContractError`。

## 5. 模型输出

模型原始输出：

| 字段 | 类型 | 说明 |
|---|---|---|
| `contract_version` | `str` | 契约版本 |
| `request_id` | `str` | 原样返回请求 ID |
| `model_name` | `str` | 当前为 `busmind-route-scorer` |
| `model_version` | `str` | 当前为 `0.1.0` |
| `results` | `list[dict]` | 每条路线的评分结果 |
| `warnings` | `list[str]` | 预警信息 |
| `duration_ms` | `int` | 模型侧耗时 |
| `generated_at` | `str` | UTC 生成时间 |

`results[]` 字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `route_id` | `str` | 候选路线 ID |
| `time_score` | `float` | 时间效率分，越高越好 |
| `comfort_score` | `float` | 舒适度分，越高越好 |
| `walk_score` | `float` | 步行体验分，越高越好 |
| `transfer_score` | `float` | 换乘体验分，越高越好 |
| `reliability_score` | `float` | 数据可靠性分，越高越好 |
| `recommend_score` | `float` | 当前偏好下的综合推荐分，越高越推荐 |
| `feature_contributions` | `dict[str, float]` | 12 维特征对综合推荐分的贡献，用于解释 |

注意：模型当前保持输入顺序返回 `results`，不替后端排序。后端需要按 `recommend_score` 降序排序。

## 6. 后端派生输出

后端拿到模型输出后，建议派生以下 ID：

| 字段 | 派生规则 |
|---|---|
| `best_experience_route_id` | `recommend_score` 最高 |
| `fastest_route_id` | `time_score` 最高 |
| `least_crowded_route_id` | `comfort_score` 最高 |
| `least_walking_route_id` | `walk_score` 最高 |
| `least_transfer_route_id` | `transfer_score` 最高 |

这些字段属于后端公开响应层，不是模型函数当前直接返回的字段。

## 7. 错误和校验规则

模型会在输入不符合契约时抛出 `ModelContractError`：

- `routes` 必须是非空列表，最多 10 条；
- 模型内部 12 个数值特征必须能由输入直接得到或经预处理映射得到；
- `transfer_count` 必须是非负整数；
- `service_nos` 必须是非空列表；
- `load_code` 若传入，必须是 `SEA` / `SDA` / `LSD` / `UNKNOWN`；
- `feature_sources` 若传入，必须是对象，来源值必须属于允许枚举；
- 不允许传入 `weights`。

后端建议捕获 `ModelContractError`，转成明确的接口错误或降级到规则排序。

## 8. 模型结构

当前实现：

```text
raw route payload
    ↓
preprocessing
    ↓
x ∈ R^12
z = (x - mean) / std
s = sigmoid(C_sub · z + b_sub) × 100
recommend_score = preference_mix(preference) · s
```

其中：

- `x` 是 12 维路线特征；
- `z` 是标准化后的特征；
- `C_sub` 是 `5 x 12` 的评分头系数矩阵；
- `b_sub` 是 5 维偏置；
- `s` 是五维体验分；
- `preference_mix` 是模型内置偏好模板系数。

若 `algorithm/model/artifacts/route_scorer_v1.json` 存在且可读，模型使用训练得到的 `mean/std/subscore_coefficients/subscore_bias`。若模型文件缺失或异常，自动回退到内置默认参数。

## 10. 对外说明建议

> BusMind 推荐模型是候选路线评分器。后端负责生成候选路线，并为每条候选路线聚合 ETA、客载、步行、换乘、发车频率、历史客流、道路拥堵和数据可靠性等业务字段；模型预处理层统一将业务字段转换为 12 维数值特征，评分层再通过五维评分头输出时间效率、舒适度、步行体验、换乘体验、可靠性五类分数，最后根据用户偏好生成综合推荐分。第一版使用规则基线生成伪标签训练，后续可以接入真实用户反馈继续优化排序。
