# BusMind 后端与推荐模型数据契约

> 契约版本：`1.0.0`  
> 模型入口：`algorithm.model.predictor.predict_recommendation(payload)`  
> 模型名称：`busmind-route-scorer`  
> 模型版本：`0.1.0`

## 1. 责任边界

模型只负责给后端生成好的候选路线打分，不负责生成候选路线。

后端负责：

- 根据起点、终点生成候选路线；
- 聚合每条候选路线的 ETA、Load、历史客流、路况、步行、换乘、发车频率等业务字段；
- 标记估算、缺失或降级字段；
- 调用推荐模型；
- 根据模型分数排序、分组、打标签，并组织最终返回给前端的响应。

模型负责：

- 校验后端传入的 route payload；
- 把业务字段预处理成 12 维数值特征；
- 输出每条候选路线的 5 个分项评分和综合推荐分；
- 返回 `feature_contributions`，供后续解释层使用。

模型不负责：

- 调用 LTA API；
- 查询 MySQL 或缓存；
- 预测 ETA；
- 预测 Load；
- 生成最终自然语言解释；
- 决定公开 HTTP API 响应格式。

## 2. 顶层请求

```json
{
  "contract_version": "1.0.0",
  "request_id": "rec_20260713_001",
  "preference": "balanced",
  "routes": []
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `contract_version` | string | 否 | 当前固定为 `1.0.0`，不传时使用默认值 |
| `request_id` | string | 是 | 一次推荐请求的唯一 ID |
| `preference` | string | 否 | 用户偏好，不传默认为 `balanced` |
| `routes` | object[] | 是 | 后端生成的候选路线，当前最多 10 条 |

支持的偏好：

```text
balanced
fastest
comfort
less_walking
less_transfer
```

兼容别名：

```text
low_load -> comfort
```

模型不接受 `weights`。如果 payload 中出现 `weights`，模型会返回契约错误。用户偏好只通过 `preference` 表达，偏好权重模板由模型内部维护。

## 3. routes[] 输入字段

后端调用模型时，每条候选路线必须传下面这些业务字段。后端不要传 `load_score`、`history_flow_score` 这类模型内部分。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `route_id` | string | 是 | 候选路线唯一 ID |
| `service_nos` | string[] | 是 | 候选路线包含的公交线路号，例如 `["105"]` 或 `["105","88"]` |
| `eta_minutes` | number | 是 | 首段公交预计到站时间，来自 LTA Bus Arrival |
| `ride_time_minutes` | number | 是 | 车内时间，后端寻路或距离估算得到 |
| `walk_time_minutes` | number | 是 | 起终点接驳步行时间 |
| `walk_distance_meters` | number | 是 | 起终点接驳步行距离 |
| `transfer_count` | integer | 是 | 换乘次数 |
| `avg_service_frequency_minutes` | number | 是 | 候选线路平均发车间隔 |
| `load_code` | string | 是 | LTA Load 原始状态：`SEA` / `SDA` / `LSD` / `UNKNOWN` |
| `station_flow_level` | string | 是 | 历史站点客流等级：`low` / `medium` / `high` |
| `route_speed_band` | integer | 是 | 后端按候选路线聚合后的 LTA SpeedBand，范围 1-8 |
| `source_updated_at` | string | 是 | 实时数据更新时间，ISO datetime 字符串 |
| `monitored` | integer | 是 | LTA Bus Arrival `Monitored`，取值 0/1 |
| `degraded_fields` | string[] | 是 | 缺失、估算或降级字段列表；没有降级时传空数组 |
| `feature_sources` | object | 是 | 各输入字段的数据来源 |

`feature_sources` 的值建议使用以下来源名：

```text
lta_realtime
lta_line_fallback
cache
database
processed_csv
historical
traffic_speed_bands
backend_graph
backend_graph_estimate
rule_estimate
model
default
```

禁止后端传入以下模型内部字段：

```text
load_score
history_flow_score
congestion_score
data_freshness_seconds
monitored_score
completeness_score
```

这些字段统一由 `algorithm/model/preprocessing.py` 映射得到。

## 4. 预处理后的 12 维数值特征

模型训练和推理真正使用的 12 维数值特征固定如下：

```text
eta_minutes
ride_time_minutes
walk_time_minutes
walk_distance_meters
transfer_count
load_score
history_flow_score
congestion_score
data_freshness_seconds
monitored_score
completeness_score
avg_service_frequency_minutes
```

其中 6 个字段由模型预处理层映射：

| 模型内部特征 | 后端字段 | 映射口径 |
|---|---|---|
| `load_score` | `load_code` | `SEA=100`，`SDA=70`，`LSD=35`，`UNKNOWN=60` |
| `history_flow_score` | `station_flow_level` | `low=90`，`medium=70`，`high=45` |
| `congestion_score` | `route_speed_band` | `SpeedBand=1..8` 映射为 30..100 通畅分 |
| `data_freshness_seconds` | `source_updated_at` | 当前时间减数据更新时间 |
| `monitored_score` | `monitored` | `1 -> 100`，`0 -> 75` |
| `completeness_score` | `degraded_fields` | `100 - 降级字段数 * 6`，最低 55 |

## 6. 模型响应

```json
{
  "contract_version": "1.0.0",
  "request_id": "rec_20260713_001",
  "model_name": "busmind-route-scorer",
  "model_version": "0.1.0",
  "results": [
    {
      "route_id": "route_001",
      "time_score": 76.0,
      "comfort_score": 72.5,
      "walk_score": 60.0,
      "transfer_score": 100.0,
      "reliability_score": 88.0,
      "recommend_score": 79.8,
      "feature_contributions": {
        "eta_minutes": -0.08,
        "load_score": 0.19
      }
    }
  ],
  "warnings": [],
  "duration_ms": 12,
  "generated_at": "2026-07-13T00:01:30+00:00"
}
```

`results[]` 字段说明：

| 字段 | 类型 | 说明 |
|---|---|---|
| `route_id` | string | 原样返回候选路线 ID |
| `time_score` | number | 时间效率分 |
| `comfort_score` | number | 舒适度分 |
| `walk_score` | number | 步行体验分 |
| `transfer_score` | number | 换乘体验分 |
| `reliability_score` | number | 数据可靠性分 |
| `recommend_score` | number | 当前偏好下的综合推荐分 |
| `feature_contributions` | object | 12 维特征对综合分的贡献 |

模型保持输入顺序返回结果，不替后端排序。后端需要按 `recommend_score` 降序排序。

## 7. 后端最终派生字段

后端拿到模型响应后，建议在公开推荐接口中派生：

| 字段 | 派生规则 |
|---|---|
| `best_experience_route_id` | `recommend_score` 最高 |
| `fastest_route_id` | `time_score` 最高 |
| `least_crowded_route_id` | `comfort_score` 最高 |
| `least_walking_route_id` | `walk_score` 最高 |
| `least_transfer_route_id` | `transfer_score` 最高 |

这些字段属于后端公开响应层，不是模型函数当前直接返回的字段。

## 8. 错误和降级

模型契约错误包括：

- 缺少必填字段；
- `routes` 为空或超过 10 条；
- 数值字段不是有限数；
- `contract_version` 不匹配；
- `load_code`、`station_flow_level` 等枚举不合法；
- `feature_sources` 不是对象或来源值不在允许枚举内；
- payload 中出现 `weights`；
- 后端传入了禁止的模型内部分数字段。

后端接入时，如果模型不可用或响应不合法，应整批回退规则评分，不能部分采用模型结果。

## 9. 离线训练对齐

离线训练的 `algorithm/dataset/features.csv` 与线上 `routes[]` 业务字段对齐，只额外包含：

```text
candidate_group_id
is_synthetic
```

训练时由 `build_labels.py` 把同一 `candidate_group_id` 下的路线组装成 `predict_recommendation(payload)` 同款结构，再生成伪标签。`train_ranker.py` 训练前也必须走同一套 preprocessing，不能直接依赖 `features.csv` 中不存在的内部字段。
