# BusMind 项目最终数据库表设计文档

> 适用项目：BusMind 公交舒适出行系统  
> 技术栈：FastAPI + MySQL + Vue3  
> 文档版本：v1.2（2026-07-12 代码核对版）  
> 核对基线：`backend/app/models/**`、`backend/app/db/schema_check.py`、`backend/tests/test_database_schema_alignment.py`  
> 目标：覆盖用户、线路、站点、车辆、地图、实时 ETA、实时客载、历史客流分析、保留的客流预测数据、路线推荐、AI 助手、管理员统计及数据更新接口。

本文档描述目标数据库结构。发生冲突时，判断顺序为：实际运行数据库结构、当前 SQLAlchemy ORM、结构一致性测试、本文档。实际数据库应通过只读结构检查器再次验证；仅通过 ORM 测试不能证明部署数据库已经同步。

## 1. 最终表清单

### 1.1 必须建立

```text
user_account
user_query_history
location_poi
bus_station
bus_line
line_station
bus_vehicle
bus_eta_status
```

其中 `passenger_flow_prediction` 是保留的历史/实验数据表。第一阶段 Passenger Flow 定位为历史客流分析，不作为第一阶段核心预测模型；核心算法是多源数据融合、多目标评分和个性化推荐。

### 1.2 建议保留

```text
lta_bus_arrival
traffic_speed_bands
```

### 1.3 建议建立为视图

```text
v_map_station
```

### 1.4 可选扩展

```text
ai_conversation
ai_message
user_location
route_recommendation_history
```

# 3. 必须建立的数据表

## 3.1 `user_account` 用户账号表

### 用途

用于注册、登录、身份判断、用户信息查询和管理员鉴权。

### 对应接口

```text
POST /api/v1/users/register
POST /api/v1/users/login
GET  /api/v1/users/me
POST /api/user/login
```

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `user_id` | BIGINT | PK, AUTO_INCREMENT | 用户主键 |
| `username` | VARCHAR(50) | NOT NULL, UNIQUE | 登录用户名 |
| `password_hash` | VARCHAR(255) | NOT NULL | 密码哈希，禁止保存明文 |
| `nickname` | VARCHAR(50) | NULL | 用户昵称 |
| `role` | VARCHAR(20) | NOT NULL, DEFAULT `passenger` | `passenger`、`admin` |
| `status` | VARCHAR(20) | NOT NULL, DEFAULT `active` | `active`、`disabled` |
| `created_at` | DATETIME | NOT NULL | 注册时间 |
| `updated_at` | DATETIME | NOT NULL | 更新时间 |
| `last_login_at` | DATETIME | NULL | 最后登录时间 |

### 索引

```sql
UNIQUE KEY uk_user_username (username);
KEY idx_user_role (role);
KEY idx_user_status (status);
```

## 3.5 `bus_line` 公交线路表

### 用途

线路列表、线路详情、运营信息及管理员线路统计。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `line_id` | BIGINT | PK | 线路主键，使用数据脚本生成的整数 ID |
| `service_no` | VARCHAR(30) | NOT NULL | 原始线路编号 |
| `line_name` | VARCHAR(100) | NOT NULL | 展示名称 |
| `operator` | VARCHAR(100) | NULL | 运营方 |
| `direction` | TINYINT | NOT NULL, DEFAULT 1 | 线路方向 |
| `category` | VARCHAR(50) | NULL | LTA 线路类别或服务类别 |
| `origin_station_id` | BIGINT | FK, NULL | 起点站 |
| `origin_stop_code` | VARCHAR(30) | NULL | 起点站 LTA 原始编号；ORM 兼容属性名为 `start_station` |
| `destination_station_id` | BIGINT | FK, NULL | 终点站 |
| `destination_stop_code` | VARCHAR(30) | NULL | 终点站 LTA 原始编号；ORM 兼容属性名为 `end_station` |
| `am_peak_freq` | VARCHAR(20) | NULL | 早高峰发车间隔范围，保留 LTA 原始值，例如 `06-08` |
| `am_offpeak_freq` | VARCHAR(20) | NULL | 日间非高峰发车间隔范围，保留 LTA 原始值 |
| `pm_peak_freq` | VARCHAR(20) | NULL | 晚高峰发车间隔范围，保留 LTA 原始值 |
| `pm_offpeak_freq` | VARCHAR(20) | NULL | 晚间非高峰发车间隔范围，保留 LTA 原始值 |
| `avg_service_frequency` | DECIMAL(6,2) | NULL | 综合平均发车间隔，分钟，用于排序、推荐和展示 |
| `loop_desc` | VARCHAR(255) | NULL | 环线或循环线路说明 |
| `status` | VARCHAR(20) | NOT NULL, DEFAULT `running` | `running`、`suspended`、`offline` |
| `created_at` | DATETIME | NOT NULL | 创建时间 |
| `updated_at` | DATETIME | NOT NULL | 更新时间 |

### 索引

```sql
UNIQUE KEY uk_line_service_direction (service_no, direction);
KEY idx_line_service_no (service_no);
KEY idx_line_name (line_name);
KEY idx_line_status (status);
```

说明：`service_no` 单独不保证唯一，同一线路编号可按不同 `direction` 存储；物理唯一性由 `(service_no, direction)` 保证。

## 3.6 `line_station` 线路站序表

### 用途

维护线路与站点的多对多关系、站序和累计距离。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `line_station_id` | VARCHAR(50) | PK | 线路站序主键，例如 `1_001` |
| `line_id` | BIGINT | FK, NOT NULL | 线路 ID |
| `service_no` | VARCHAR(30) | NULL | 原始线路编号快照，便于导入和排查 |
| `line_name` | VARCHAR(100) | NULL | 线路名称快照 |
| `operator` | VARCHAR(100) | NULL | 运营方快照 |
| `direction` | TINYINT | NULL | 线路方向快照 |
| `station_id` | BIGINT | FK, NOT NULL | 站点 ID |
| `bus_stop_code` | VARCHAR(30) | NULL | 原始站点编号快照 |
| `stop_sequence` | INT | NOT NULL | 站点顺序 |
| `route_distance_km` | DECIMAL(8,3) | NULL | 从线路起点累计距离 |
| `wd_first_bus` | TIME | NULL | 工作日首班车 |
| `wd_last_bus` | TIME | NULL | 工作日末班车 |
| `sat_first_bus` | TIME | NULL | 周六首班车 |
| `sat_last_bus` | TIME | NULL | 周六末班车 |
| `sun_first_bus` | TIME | NULL | 周日首班车 |
| `sun_last_bus` | TIME | NULL | 周日末班车 |
| `created_at` | DATETIME | NOT NULL | 创建时间 |
| `updated_at` | DATETIME | NOT NULL | 更新时间 |

### 约束与索引

```sql
UNIQUE KEY uk_line_station_sequence (line_id, stop_sequence);
KEY idx_line_station_pair (line_id, station_id);
KEY idx_line_station_station (station_id);
```

说明：不对 `(line_id, station_id)` 做唯一约束。部分环线或特殊线路可能在同一方向中重复经过同一站点，真正唯一的是 `(line_id, stop_sequence)`。

## 3.8 `bus_eta_status` ETA 实时到站状态表

### 用途

存储 LTA Bus Arrival API 返回并解析得到的车辆到站状态。当前第一阶段 ETA 来自实时接口，不作为机器学习预测结果。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `eta_status_id` | BIGINT | PK, AUTO_INCREMENT | 主键 |
| `vehicle_id` | BIGINT | NOT NULL | 车辆 ID；LTA 不提供稳定车牌时由脚本生成 |
| `line_id` | BIGINT | FK, NOT NULL | 线路 ID |
| `target_station_id` | BIGINT | FK, NOT NULL | 目标站点 |
| `bus_stop_code` | VARCHAR(30) | NULL | LTA 原始站点编号，便于追溯 |
| `query_time` | DATETIME | NOT NULL | API 查询时间 |
| `eta_minutes` | DECIMAL(8,2) | NOT NULL | 到站分钟数 |
| `arrival_time` | DATETIME | NULL | 预计到站时间 |
| `vehicle_to_stop_distance_m` | DECIMAL(10,2) | NULL | 车辆到站距离 |
| `speed_kph` | DECIMAL(6,2) | NULL | 根据车辆位置和 ETA 估算速度 |
| `confidence` | DECIMAL(5,4) | NULL | 置信度 |
| `data_source` | VARCHAR(100) | NOT NULL | 数据来源，默认 `lta_bus_arrival` |
| `created_at` | DATETIME | NOT NULL | 入库时间 |
| `updated_at` | DATETIME | NOT NULL | 更新时间 |

### 索引

```sql
UNIQUE KEY uk_eta_vehicle_station_time (vehicle_id, target_station_id, query_time);
KEY idx_eta_vehicle_time (vehicle_id, query_time);
KEY idx_eta_line_station (line_id, target_station_id);
KEY idx_eta_arrival_time (arrival_time);
```

说明：`vehicle_id` 用于关联同一次到站样本和车辆状态，但当前不对 `bus_vehicle` 建外键，避免 LTA 样本中临时生成的车辆 ID 影响实时数据入库。

## 3.11 `passenger_flow_prediction` 客流预测结果表

### 用途

保存历史遗留或实验性的未来站点/路段客流结果。该表仍由当前 ORM 映射并保持可运行，但第一阶段页面和核心算法不依赖该表；Passenger Flow 的正式第一阶段能力是基于 `passenger_flow_trend` 的历史客流分析。该表不能用 `bus_load_status` 替代，后者表示 LTA 实时车辆客载。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `prediction_id` | BIGINT | PK, AUTO_INCREMENT | 主键 |
| `target_type` | VARCHAR(20) | NOT NULL | `station` 或 `segment` |
| `target_id` | VARCHAR(50) | NOT NULL | 目标站点或路段；站点可写入 `station_id` 字符串，路段对应 `segment_id` |
| `prediction_time` | DATETIME | NOT NULL | 本次预测生成时间 |
| `predict_time` | DATETIME | NOT NULL | 被预测的未来时间 |
| `predicted_flow` | INT | NOT NULL | 预测客流量 |
| `crowd_level` | VARCHAR(20) | NOT NULL | 拥挤等级 |
| `confidence` | DECIMAL(5,4) | NULL | 置信度 |
| `model_version` | VARCHAR(100) | NULL | 模型版本 |
| `created_at` | DATETIME | NOT NULL | 创建时间 |

### 索引

```sql
KEY idx_flow_prediction_target_time (target_type, target_id, predict_time);
KEY idx_flow_prediction_created (prediction_time);
```

## 3.12 `map_road_segment` 地图路段表

### 用途

支持地图线路绘制、路段点击、路段详情和客流展示。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `segment_id` | VARCHAR(50) | PK | 路段 ID，使用脚本生成的字符串编号，例如 `1_001` |

# 5. 建议建立的视图

## 5.1 `v_map_station` 地图站点视图

不建议长期维护独立的 `map_station` 实体表。站点坐标来自 `bus_station`，线路列表来自 `line_station` 和 `bus_line`。

### 建议字段

```text
station_id
bus_stop_code
station_name
road_name
longitude
latitude
line_count
service_count
line_ids
line_names
service_nos
```

### 示例 SQL

```sql
CREATE VIEW v_map_station AS
SELECT
    s.station_id,
    s.bus_stop_code,
    s.station_name,
    s.road_name,
    s.longitude,
    s.latitude,
    COUNT(DISTINCT ls.line_id) AS line_count,
    COUNT(DISTINCT l.service_no) AS service_count,
    GROUP_CONCAT(DISTINCT l.line_id ORDER BY l.line_id SEPARATOR '|') AS line_ids,
    GROUP_CONCAT(DISTINCT l.line_name ORDER BY l.line_id SEPARATOR '|') AS line_names,
    GROUP_CONCAT(DISTINCT l.service_no ORDER BY l.service_no SEPARATOR '|') AS service_nos
FROM bus_station s
LEFT JOIN line_station ls ON ls.station_id = s.station_id
LEFT JOIN bus_line l ON l.line_id = ls.line_id
GROUP BY
    s.station_id,
    s.bus_stop_code,
    s.station_name,
    s.road_name,
    s.longitude,
    s.latitude;
```

# 11. 数据库实施注意事项

1. 密码只保存哈希值，禁止保存明文密码。
2. 经度和纬度统一使用 `DECIMAL(10,7)`。
3. 时间统一使用 `DATETIME`，接口层再格式化为统一字符串。
4. 保留的实验预测表 `passenger_flow_prediction` 必须包含 `prediction_time`、`model_version` 和必要的置信度字段；该表的存在不代表 Passenger Flow 已被列为第一阶段预测模型。
5. `bus_load_status` 表示车内客载实时状态，不能替代站点或路段客流预测表。
6. `bus_vehicle.operation_status` 与 `load_level` 必须分开。
7. `map_station` 建议改为视图，避免与 `bus_station` 重复存储。
8. 路线推荐可以实时计算，不强制建立推荐结果表。
9. AI 助手不保存聊天历史时，AI 相关表可以暂不建立。
10. 数据更新接口写数据库时应使用事务，并同步更新 `updated_at`。
11. CSV 导入前应统一 ID 类型、枚举值、时间格式和空值规则；基础数据表导入时保留脚本生成的 `station_id`、`line_id`、`vehicle_id`，不重新自增生成。
12. 表名与后端 ORM 模型名应保持一一对应，避免同一含义出现多种命名。

# 12. 数据流转、缓存与算法调用边界

## 12.1 总体数据流转链路

BusMind 的数据流转建议按“原始采集、清洗加工、数据库/缓存、后端编排、算法计算、前端展示”分层处理

```text
LTA DataMall 数据源
        ↓
采集脚本保存原始 JSON / ZIP / JSONL
        ↓
原始数据进入 data/raw/lta
        ↓
处理脚本清洗、映射 ID、统一字段、计算派生特征
        ↓
生成 data/processed 标准 CSV
        ↓
静态与历史数据导入 MySQL
        ↓
实时 Bus Arrival 和 Traffic Speed Bands 写入内存缓存，后续可替换为 Redis
        ↓
后端接口读取 MySQL + 缓存
        ↓
推荐算法计算分项得分和综合推荐分
        ↓
后端返回结构化推荐结果
        ↓
大模型基于结构化结果生成自然语言解释
        ↓
前端展示地图、路线卡片、热力图和推荐理由
```

一句话边界：

```text
数据处理工程师负责把 LTA 原始数据变成稳定、干净、可导入、可计算的 processed CSV 和实时缓存数据；
后端负责把这些数据接入数据库、缓存、接口和推荐服务
```

## 12.2 数据分层与存储位置

| **数据层** | **存放位置** | **数据类型** | **主要用途** |
|---|---|---|---|
| 原始采集层 | `data/raw/lta` | LTA 原始 JSON、ZIP、JSONL | 保留原始响应，便于追溯和重新处理 |
| 清洗加工层 | `data/processed` | 标准 CSV、处理摘要 | 数据库初始化、后端导入、算法特征准备 |
| 静态主数据层 | MySQL | 站点、线路、站序、车辆、地图路段 | 支持基础查询和候选路线生成 |
| 历史统计层 | MySQL | 站点客流、道路速度采样、到站采样 | 支持趋势展示、推荐特征和后续训练 |
| 实时缓存层 | Redis / 内存缓存 | ETA、Load、车辆位置、道路热力最新状态 | 支持推荐接口快速读取 |
| 服务编排层 | FastAPI 后端 | 结构化候选路线特征 | 组装算法输入，调用推荐模块 |
| 算法计算层 | `algorithm/` | 纯计算函数、模型文件、规则参数 | 计算分数、排序、推荐原因 |

## 12.3 采集到 processed CSV 的流转

原始数据采集脚本放在：

```text
data/raw/collect_scripts
```

原始数据建议保存为：

```text
data/raw/lta/api_response/BusStops_full.json
data/raw/lta/api_response/BusRoutes_full.json
data/raw/lta/api_response/BusServices_full.json
data/raw/lta/bus_arrival_samples/YYYY-MM-DD/bus_arrival_sample.jsonl
data/raw/lta/passenger_volume_stop/YYYYMM/original.zip
data/raw/lta/passenger_volume_od/YYYYMM/original.zip
data/raw/lta/traffic_speed_bands/YYYY-MM-DD/*.json
```

处理脚本放在：

```text
data/processed/process_scripts
```

标准 CSV 输出到：

```text
data/processed
```

建议输出：

```text
bus_station.csv
bus_line.csv
line_station.csv
bus_vehicle.csv
lta_bus_arrival.csv
bus_eta_status.csv
bus_load_status.csv
passenger_flow_trend.csv
map_road_segment.csv
traffic_speed_bands.csv
processing_summary.csv
```

处理阶段主要完成：

| **处理任务** | **说明** |
|---|---|
| ID 映射 | 将 LTA 的 `BusStopCode`、`ServiceNo` 映射为项目内部 `station_id`、`line_id` |
| 字段统一 | 转为数据库和接口使用的 snake_case 字段 |
| 时间统一 | 统一 `query_time`、`estimated_arrival`、`record_time` |
| ETA 派生 | 使用 `EstimatedArrival - query_time` 生成 `eta_minutes` |
| Load 映射 | 将 `SEA/SDA/LSD` 映射为后端枚举和 `load_score` |
| 客流分级 | 根据 Passenger Volume 生成 `flow_level`、`total_flow` |
| 路况分级 | 根据 Traffic Speed Bands 生成 `congestion_score`、热力颜色 |
| 质量报告 | 生成 `processing_summary.csv`，记录数据量、缺失值和处理时间 |

## 12.4 MySQL 与缓存的分工

静态数据、低频数据和历史统计数据进入 MySQL

实时数据和准实时数据第一阶段优先进入程序内存缓存，后续可替换为 Redis，避免用户请求反复调用 LTA API，也避免数据库写入过于频繁

| **数据类型** | **推荐存储** | **更新频率建议** | **说明** |
|---|---|---:|---|
| Bus Stops | MySQL | 低频更新 | 站点基础数据，完整入库 |
| Bus Routes | MySQL | 低频更新 | 线路站序数据，完整入库 |
| Bus Services | MySQL | 低频更新 | 运营信息和发车频率，完整入库 |
| Passenger Volume by Bus Stops | MySQL | 月度或批处理 | 历史客流压力特征 |
| Passenger Volume by OD Bus Stops | MySQL / 可选 | 月度或批处理 | OD 客流压力，第一阶段非核心 |
| Bus Arrival 最新状态 | Redis / 内存缓存 | 30-60 秒 | 实时 ETA、Load、车辆位置 |
| Traffic Speed Bands 最新状态 | Redis / 内存缓存 | 1-5 分钟 | 准实时道路热力 |
| Bus Arrival 采样快照 | MySQL / CSV | 5 分钟或更长 | 追溯、演示趋势、后续分析 |
| Traffic Speed Bands 采样快照 | MySQL / CSV | 5 分钟或更长 | 道路热力历史趋势 |
| 推荐结果历史 | MySQL 可选 | 按用户请求 | 用于用户历史、个性化学习 |

## 12.5 缓存 Key 设计

Bus Arrival 最新状态：

```text
bus_arrival:{bus_stop_code}
bus_arrival:{bus_stop_code}:{service_no}
```

示例：

```text
bus_arrival:01012
bus_arrival:01012:36
```

缓存内容建议包含：

```json
{
  "bus_stop_code": "01012",
  "service_no": "36",
  "query_time": "2026-07-09T14:00:00+08:00",
  "estimated_arrival": "2026-07-09T14:08:00+08:00",
  "eta_minutes": 8,
  "load_code": "SDA",
  "load_score": 70,
  "latitude": 1.2903,
  "longitude": 103.8519,
  "monitored": true,
  "data_source": "lta_bus_arrival"
}
```

Traffic Speed Bands 最新状态：

```text
traffic_speed_bands:latest
traffic_speed_bands:{link_id}
```

推荐结果短缓存：

```text
recommend:{start_station_id}:{end_station_id}:{preference}:{time_bucket}
```

其中 `time_bucket` 可按 5 分钟取整，减少同一时间段重复计算

## 12.6 缓存 TTL 与刷新策略

| **缓存内容** | **建议 TTL** | **刷新方式** | **过期处理** |
|---|---:|---|---|
| `bus_arrival:{bus_stop_code}` | 45-90 秒 | 定时任务或请求触发刷新 | 返回旧值并降低可靠性，或使用发车频率降级 |
| `bus_arrival:{bus_stop_code}:{service_no}` | 45-90 秒 | 定时任务或请求触发刷新 | 同上 |
| `traffic_speed_bands:latest` | 2-5 分钟 | 定时任务刷新 | 使用最近一次采样或默认拥堵分 |
| `traffic_speed_bands:{link_id}` | 2-5 分钟 | 定时任务刷新 | 使用默认 `congestion_score = 60` |
| `recommend:*` | 1-5 分钟 | 用户请求后写入 | 过期后重新计算 |

实时数据缓存建议采用“先读缓存、再读数据库、最后降级”的顺序：

```text
Redis / 内存缓存
        ↓
MySQL 最新状态或最近采样
        ↓
LTA 实时接口
        ↓
发车频率、历史客流、默认路况降级
```

## 12.7 请求合并与防重复采集

当多个用户同时查询同一站点或同一路线时，不应重复调用 LTA API

建议使用请求合并锁：

```text
1. 先查缓存
2. 缓存命中且未过期，直接返回
3. 缓存过期时，只有第一个请求触发刷新
4. 其他请求等待刷新结果，或返回旧缓存并标记 stale
5. 刷新完成后更新缓存和过期时间
```

伪流程：

```text
if cache exists and not expired:
    return cache

if refresh_lock exists:
    return stale_cache or wait

acquire refresh_lock
fetch LTA
update cache
release refresh_lock
return data
```

锁 Key 示例：

```text
lock:bus_arrival:{bus_stop_code}:{service_no}
lock:traffic_speed_bands:latest
```

## 12.8 实时数据落库原则

实时数据不建议每次请求都完整入库，避免数据库膨胀

建议分三类保存：

| **保存方式** | **说明** | **对应表或文件** |
|---|---|---|
| 最新状态 | 只保留最新一份，支持接口快速读取 | `bus_vehicle`、`bus_eta_status`、`bus_load_status` |
| 采样快照 | 按固定间隔保存，例如每 5 分钟一次 | `lta_bus_arrival`、`traffic_speed_bands` |
| 聚合统计 | 将分钟级数据聚合为小时级或日级 | 后续可扩展趋势统计表 |

原则：

```text
静态数据全量入库
实时数据短期缓存
历史数据按时间片采样保存
统计分析数据按小时或天聚合
```

## 12.9 离线训练阶段的数据流转

如果后续训练客流预测、道路拥堵趋势或用户偏好模型，训练阶段可以由数据处理脚本或算法脚本直接读取 CSV 或数据库历史表

离线训练链路：

```text
data/raw
        ↓
data/processed
        ↓
MySQL 历史表 / CSV
        ↓
训练或统计脚本
        ↓
模型文件 / 特征统计 / 训练样本
        ↓
algorithm/model/artifacts 或 algorithm/dataset/recommendation/v1
```

可训练任务和数据来源：

| **任务** | **训练数据来源** | **标签或目标** |
|---|---|---|
| 站点客流压力预测 | `passenger_flow_trend`、Passenger Volume CSV | `total_flow` 或 `flow_level` |
| OD 需求强度预测 | OD Passenger Volume CSV | `TOTAL_TRIPS` 或 `od_level` |
| 道路拥堵趋势预测 | Traffic Speed Bands 历史采样 | `speed_band` 或 `congestion_score` |
| 用户偏好学习 | 推荐曝光、点击、最终选择记录 | 用户最终选择的路线 |
| 推荐排序优化 | 推荐结果历史和用户反馈 | 是否被点击或选择 |

离线训练输出示例：

```text
algorithm/dataset/recommendation/v1/features.csv
algorithm/dataset/recommendation/v1/pseudo_labels.csv
algorithm/model/artifacts/route_scorer_v1.json
```

## 12.10 线上调用阶段的数据流转

线上推荐接口被用户调用时，推荐算法模块不建议直接读取 MySQL、Redis 或 LTA API

推荐边界是：

```text
后端负责读数据库和缓存
后端负责组装候选路线特征
算法模块只负责计算分数、排序和推荐原因
```

线上调用链路：

```text
前端请求推荐接口
        ↓
FastAPI 后端读取 MySQL 静态数据
        ↓
FastAPI 后端读取内存缓存 / Redis 中的实时 ETA、Load、道路热力
        ↓
FastAPI 后端读取 MySQL 历史客流趋势
        ↓
后端组装 CandidateRouteFeature 列表
        ↓
调用 algorithm.model.predictor.predict_recommendation 纯计算函数
        ↓
算法返回评分、排序、推荐标签和结构化原因
        ↓
后端返回前端，并可交给大模型解释
```

## 12.11 后端、数据处理与算法边界

| **阶段** | **谁读取数据库/缓存** | **算法职责** | **输出** |
|---|---|---|---|
| 数据采集 | 数据处理脚本读取 LTA API | 不参与实时推荐计算 | 原始 JSON、ZIP、JSONL |
| 数据清洗 | 数据处理脚本读取 raw | 生成特征字段和标准 CSV | processed CSV |
| 离线训练/统计 | 模型数据脚本读取 CSV / MySQL | 生成 12 维特征、伪标签、训练评分头 | features、pseudo_labels、模型 JSON |
| 线上调用 | 后端读取 MySQL + 缓存 | 只计算分数，不直接读库或调 API | 路线评分结果 |
| AI 解释 | 后端传结构化推荐结果 | 不重新计算推荐分 | 自然语言解释上下文 |

最终建议：

```text
训练阶段，算法/数据脚本可以直接读 CSV 或数据库；
线上调用阶段，算法不直接读库和缓存，而是由后端传入已经组装好的结构化特征
```
