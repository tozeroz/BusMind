# BusMind 数据流转链路实现

> 文档版本：v1.3（2026-07-14 代码核对版）。
>
> 当前口径：线上实时服务、离线数据加工和推荐模型训练彼此解耦。ETA 来自 LTA Bus Arrival 的 `EstimatedArrival`，Load 来自同一响应的 `Load`；接口中的 `predicted_*`、`model_version` 是兼容命名，不表示项目训练了 ETA/客载预测模型。Passenger Flow 当前用于历史客流分析与推荐特征，不是第一阶段的独立预测模型。

## 2. 三条实际链路

系统不是一条从 `data/raw` 串到实时接口的单链路，而是三条可独立运行的链路。

### 2.1 离线基础数据

```text
LTA 静态接口 / Passenger Volume / 历史 Bus Arrival / Traffic Speed Bands
  → data/raw/lta
  → build_lta_processed_data.py
  → data/processed/*.csv
  → import_processed_to_mysql.py
  → MySQL 基础表、状态表、历史表
```

加工脚本主要输出：`bus_station.csv`、`bus_line.csv`、`line_station.csv`、`bus_vehicle.csv`、`lta_bus_arrival.csv`、`bus_eta_status.csv`、`bus_load_status.csv`、`map_station.csv`、`map_road_segment.csv`、`passenger_flow_trend.csv`、`traffic_speed_bands.csv`（可选）和 `processing_summary.csv`。

这些是数据库/业务中间数据，不是最终训练样本。`eta_prediction.csv`、`load_prediction.csv` 不是当前加工脚本输出，也不应作为现阶段训练集或主链路输入。

### 2.2 在线实时数据

```text
LTA v3/BusArrival 或 v4/TrafficSpeedBands
  → LtaDataMallClient
  → LtaCollectorService
  → MemoryCacheProvider
  → CacheSyncService（需要持久化的调用路径）
  → MySQL
```

| 数据 | 缓存 key | TTL | 同步表 |
|---|---|---:|---|
| Bus Arrival | `bus_arrival:{stop_code}`、`bus_arrival:{stop_code}:{service_no}` | 75 秒 | `bus_vehicle`、`bus_eta_status`、`bus_load_status`、`lta_bus_arrival` |
| Traffic Speed Bands | `traffic_speed_bands:latest`、`traffic_speed_bands:{link_id}` | 180 秒 | `traffic_speed_bands` |

缓存是全局单例 `MemoryCacheProvider`，仅当前后端进程可见；多进程/多实例不共享，重启后清空。目前没有 Redis Provider。

### 2.3 在线推荐与 AI

```text
POST /api/v1/recommend-routes
  → RecommendationService
  → TransitGraphService 生成候选路线
  → MySQLTransitGateway + EtaService + PassengerLoadService 聚合多源数据
  → 组装模型原始业务字段
  → algorithm.model.predictor.predict_recommendation
  → 12 维特征、5 个子分数、recommend_score
  → 排序/标签/前端
  → POST /api/v1/ai/travel 可解释结构化结果
```

DeepSeek 只负责自然语言生成，不生成候选路线、不改写模型分数；未配置或调用失败时使用本地模板降级。

## 3. 自动刷新与手动刷新

Bus Arrival 自动刷新已经接入，不是待实现项：

- FastAPI `lifespan` 启动/停止 `BusArrivalRefreshScheduler`。
- 调度器用 `asyncio.create_task` 自旋，不依赖 APScheduler。
- 配置 `LTA_ACCOUNT_KEY` 且刷新间隔大于 0 时启用；否则安全禁用。
- 生产工厂默认每 60 秒从 `line_station` 选取前 10 条线路、每线前 2 个真实站点，组成有效 `(ServiceNo, BusStopCode)` 任务。
- 默认并发 4，单任务截止时间 6 秒；失败隔离，不中断后续 tick。
- 每个任务先刷新内存缓存，再同步并提交 MySQL。
- `POST /api/v1/users/me/refresh-arrivals` 可非阻塞触发一次刷新，有 60 秒冷却。
- `GET /api/v1/admin/lta/bus-arrival/scheduler-status` 返回调度状态。

管理员接口：

| 接口 | 行为 |
|---|---|
| `POST /api/v1/admin/lta/bus-arrival/refresh` | 刷新指定站点/线路；`sync_to_db=true` 时同步四张实时表 |
| `POST /api/v1/admin/lta/traffic-speed-bands/refresh` | 刷新全量路况；`sync_to_db=true` 时入库 |

Traffic Speed Bands 当前没有生命周期周期调度器，主要依赖管理员触发。`GET /api/v1/map/bus-arrival` 在缓存 miss 且配置了 LTA key 时会直连 LTA 回填缓存，但不执行数据库同步。

## 4. ETA 与 Load 的实际读取顺序

```text
bus_arrival 进程缓存
  → miss：bus_eta_status / bus_load_status
  → miss：lta_bus_arrival 最近记录
  → miss：服务层规则或外部适配器兜底
```

仿真状态可在服务层覆盖实时结果，用于演示和测试。命中 MySQL 实时状态时使用 `eta_mysql_realtime_v1`、`load_mysql_realtime_v1` 标识。因此：

- `GET /api/v1/eta` 返回 LTA 实时到站状态或降级结果，不是自研 ETA 模型推理。
- `POST /api/v1/realtime-passenger-load` 是推荐的实时客载接口。
- `POST /api/v1/passenger-load-prediction` 是兼容路径，内部仍调用同一 `PassengerLoadService.predict()`。

## 5. 推荐在线推理

实际步骤：

1. 将站点 ID 或坐标解析为起终点站。
2. 从数据库公交图生成候选路线，并绑定首段线路的最新车辆。
3. 最多取前 6 条候选；再按 `max_walk_minutes` 过滤。
4. 为每条路线查询 ETA、Load、平均班次、上车站历史客流、分段路况。
5. 先生成规则体验分，再调用推荐模型；模型异常时保留规则分继续返回。
6. 按模型分和偏好排序，标记最佳体验、最快、最不拥挤、最少步行、最少换乘。
7. 未指定 `max_walk_minutes` 时，推荐结果在进程内缓存 300 秒。

| 推荐数据 | 来源 |
|---|---|
| 候选线路、行驶/步行时间、换乘数 | `bus_line`、`line_station` 构建的图快照 |
| ETA、Load | Bus Arrival 缓存/状态表/事实表，必要时规则兜底 |
| 平均班次 | `bus_line.avg_service_frequency` |
| 历史客流 | `passenger_flow_trend`，按站点和小时查询 |
| 路况 | 路线分段匹配 `traffic_speed_bands`，按行驶时间加权 |

### 5.1 后端传给模型的原始字段

```text
route_id, service_nos,
eta_minutes, ride_time_minutes, walk_time_minutes, walk_distance_meters,
transfer_count, avg_service_frequency_minutes,
load_code, station_flow_level, route_speed_band,
source_updated_at, monitored, degraded_fields, feature_sources
```

当前降级规则：步行距离按 `walk_time_minutes × 60 × 1.2m/s` 估算；无班次用 12 分钟；无客流用 `medium`；无路况用 SpeedBand 5；规则 ETA/Load 会记入 `degraded_fields`。另一个已确认的契约债务是：在线 `service_nos` 当前填入候选 `line_id` 的字符串，而不是真实 `service_no`。

### 5.2 模型侧派生的 12 维特征

`predict_recommendation()` 以 `strict_backend=True` 校验原始字段，再由 `preprocessing.py` 派生：

| 特征 | 来源/换算 |
|---|---|
| `eta_minutes`、`ride_time_minutes`、`walk_time_minutes`、`walk_distance_meters`、`transfer_count`、`avg_service_frequency_minutes` | 后端原值 |
| `load_score` | `SEA=100`、`SDA=70`、`LSD=35`、未知=60 |
| `history_flow_score` | `low=90`、`medium=70`、`high=45` |
| `congestion_score` | SpeedBand 1..8 映射为通畅度 30..100 |
| `data_freshness_seconds` | 当前时间减 `source_updated_at` |
| `monitored_score` | 真=100，假=75 |
| `completeness_score` | 100 起，每个不同降级字段扣 6，最低 55 |

输出包含 `time_score`、`comfort_score`、`walk_score`、`transfer_score`、`reliability_score`、`recommend_score` 和 `feature_contributions`。`BUSMIND_SCORING_MODEL` 默认选择 `linear`，也注册了 `xgboost`、`tabpfn`。

## 6. 推荐模型离线训练

离线训练不调用后端 HTTP、不访问线上缓存，也不直接查询 MySQL：

```text
data/raw/lta + data/processed/*.csv
  → algorithm/dataset/scripts/build_features.py
  → algorithm/dataset/features.csv
  → algorithm/dataset/scripts/build_rule_pseudo_labels.py
  → algorithm/dataset/rule_pseudo_labels.csv
  → algorithm/model/linear_scoring/train.py
  → algorithm/model/linear_scoring/artifacts/linear_route_scoring.json
```

`build_features.py` 用本地 processed CSV 构建公交图和 OD 候选组，保留 `is_synthetic`、`feature_sources`、`degraded_fields`。`build_rule_pseudo_labels.py` 用独立规则生成五个子分数、排序、软标签和置信度，不调用当前 scorer，避免模型自举。`train.py` 训练标准化层加 5 个线性 sigmoid head，再按偏好混合为 `recommend_score`。

## 8. 当前边界与待完善项

边界已固定：`data/raw` 保存原始快照，`data/processed` 保存通用 CSV；`backend` 负责实时采集、缓存、同步、候选路线和在线 payload；`algorithm/dataset` 负责训练样本/伪标签；`algorithm/model` 负责预处理、训练和推理。`passenger_flow_prediction` 表及兼容接口属于保留/实验能力，不改变 Passenger Flow 第一阶段的历史分析定位。

待完善：

1. 引入共享缓存，解决多实例一致性与重启丢失。
2. 为 Traffic Speed Bands 增加周期刷新。
3. 将 Bus Arrival 的稳定前 N 池升级为真实热度/用户范围刷新策略。
4. 补齐统一重试、退避、限流和指标。
5. 修正 `service_nos` 与 `line_id` 的语义错位。
6. 用真实步行路径距离替代时间估算。

后续继续保持：训练统一从 `algorithm/dataset` 读取冻结数据集，在线推理统一由后端传入结构化 payload；不要让训练脚本直接依赖线上接口、进程缓存或散落 CSV。
