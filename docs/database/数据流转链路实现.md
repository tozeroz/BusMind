# 一、当前数据流转总览

```text
LTA DataMall
        ↓
data/raw/lta 保存原始 JSON / ZIP / JSONL
        ↓
build_lta_processed_data.py 生成 processed CSV
        ↓
init_busmind.sql 建表
        ↓
import_processed_to_mysql.py 导入 MySQL
        ↓
FastAPI 后端读取内存缓存 / MySQL
        ↓
ETA / Load / Recommend 服务组装结构化结果
        ↓
前端展示，AI 助手只解释结构化推荐结果
```

第一阶段已经跑通的主链路是：

```text
raw -> processed -> MySQL
```

后端实时链路已经有基础代码：

```text
LTA Bus Arrival -> MemoryCacheProvider -> CacheSyncService -> MySQL
```

但接口层目前主要仍是：

```text
DemoIntelligenceGateway / SimulationStateStore / 规则降级
```

## (四) 采集和同步触发入口

当前 `LtaCollectorService` 和 `CacheSyncService` 是可调用服务，但还没有统一触发入口

可选方案：

| **方案** | **说明** | **适用阶段** |
|---|---|---|
| 管理接口触发 | 提供后台接口手动刷新某站某线 | 第一阶段推荐 |
| 应用启动后台任务 | FastAPI 启动后用轻量 loop 定时刷新 | 小规模演示可用 |
| APScheduler | 进程内定时任务 | 第二阶段 |
| Celery / RQ | 独立 worker 和任务队列 | 后续扩展 |

第一阶段建议先做管理接口或服务函数调用，不急着引入复杂任务框架

## (五) 命名兼容问题

当前接口层仍有 `predicted_*` 字段，例如：

```text
predicted_eta_minutes
predicted_load_rate
predicted_load_level
```

数据库层已经统一成：

```text
eta_minutes
load_rate
load_level
load_code
```

建议处理方式：

| **层级** | **建议** |
|---|---|
| 数据库 / CSV | 使用 `eta_minutes`、`load_level`、`load_code`、`load_rate` |
| 后端内部 Repository | 使用数据库字段名 |
| 对外接口 | 可暂时保留 `predicted_*`，避免和接口文档冲突 |
| 后续重构 | 再统一接口命名为 `eta_minutes`、`load_status` 等更准确语义 |

另外，后端还需要检查 `simulation` 相关命名：

| **当前命名** | **问题** | **建议** |
|---|---|---|
| `SimulationService.refresh_from_lta(...)` | 实际读取真实 LTA 数据，不是模拟数据 | 迁移到 `LtaCollectorService.refresh_bus_arrival(...)` 或改名 |
| `SimulationStateStore` | 适合作为演示 / 手动覆盖缓存，不适合作为正式实时缓存名称 | 正式链路统一使用 `CacheProvider` / `MemoryCacheProvider` |
| `/api/v1/simulation` | 如果暴露真实 LTA 刷新能力，会误导调用方 | 真实采集接口后续应放到 admin / collector / data refresh 语义下 |
| `PredictionResultUpdate*` | ETA / Load 第一阶段来自 LTA 实时状态，不是模型预测 | 对外可暂时兼容，内部应区分 `status` 和 `prediction` |

推荐边界：

```text
simulation_service
    只保留演示数据、手动覆盖、测试入口

collector_service
    负责真实 LTA API 采集

sync_service
    负责缓存同步 MySQL

cache
    负责统一内存缓存，后续可替换 Redis
```
