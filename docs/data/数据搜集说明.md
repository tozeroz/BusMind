> 当前项目的数据方案采用“半真实基础数据 + 本地仿真运行数据 + 模拟历史客流数据”，不追求真实公交企业内部数据，优先保证系统能查询、能展示、能预测、能推荐

# 一、数据收集目标

本项目要支撑的核心功能是：

```text
线路查询
站点查询
地图展示
车辆状态展示
ETA 到站时间计算
拥挤程度预测
舒适出行推荐
历史趋势图表
AI 助手推荐解释
```

因此数据收集不是单纯找一批真实公交数据，而是要构建一套能贯通前端、后端、数据库、算法和 AI 助手的项目数据底座

推荐总方案：

```text
真实/半真实线路站点数据
        +
模拟车辆运行数据
        +
模拟历史客流数据
        +
天气、时段、星期等辅助特征
        ↓
ETA 计算、拥挤预测、舒适推荐、地图展示、AI 解释
```

# 二、数据边界

## (一) 需要尽量真实的数据

这些数据直接影响地图展示和线路可信度，建议优先从地图或校园地图中整理：

| **数据** | **推荐来源** | **作用** |
|---|---|---|
| 线路名称 | 高德地图、百度地图、校园地图、手工设计 | 线路查询、车辆绑定 |
| 站点名称 | 地图搜索、校园地图、手工整理 | 站点查询、线路详情 |
| 站点顺序 | 地图公交线路、手工配置 | 车辆运行方向、ETA 计算 |
| 站点经纬度 | 地图 API、坐标拾取器 | 地图展示、站间距离估算 |
| 首末班时间 | 地图线路详情、手工设定 | 线路详情展示 |
| 线路轨迹 | 地图 API、前端绘线、站点连线 | 地图可视化 |

## (二) 可以模拟生成的数据

这些数据一般不公开，短期实训不建议硬找真实数据：

| **数据** | **生成方式** | **作用** |
|---|---|---|
| 车辆数量 | 人工设定 | 车辆状态展示 |
| 车辆容量 | 人工设定 | 拥挤率计算 |
| 车辆当前位置 | 仿真脚本生成 | 地图展示、ETA 计算 |
| 当前载客人数 | 仿真脚本生成 | 当前拥挤等级 |
| 历史上下车人数 | 规则模拟生成 | 拥挤预测训练、趋势图表 |
| 拥挤等级标签 | 根据载客率生成 | 规则模型和机器学习模型标签 |
| 推荐结果样例 | 推荐算法生成 | 前端演示、AI 解释 |

## (三) 不建议作为第一阶段目标的数据

这些数据通常难以获取，容易拖慢项目进度：

```text
真实公交 GPS 实时流
真实车内载客人数
真实刷卡或扫码客流
真实公交调度数据
第三方地图 App 内部实时公交 ETA
```

# 三、推荐收集方式

## (一) 第一优先：校园或区域模拟线路

这是最适合当前项目的方式，成本低，数据完全可控，适合快速完成演示闭环

建议做 3 条线路：

```text
校园 1 号线：北门 -> 宿舍区 -> 食堂 -> 教学楼 -> 图书馆 -> 体育馆 -> 南门
校园 2 号线：地铁站 -> 西门 -> 实验楼 -> 教学楼 -> 行政楼 -> 东门
校园 3 号线：学生公寓 -> 医务室 -> 食堂 -> 图书馆 -> 创新楼 -> 地铁站
```

收集步骤：

```text
1. 先确定 3 条线路和每条线路的站点顺序
2. 用高德地图、百度地图或坐标拾取器查每个站点经纬度
3. 根据站点顺序估算站间距离和平均站间耗时
4. 人工配置车辆数量、车辆容量、首末班时间
5. 用脚本生成车辆运行数据和历史客流数据
```

## (二) 第二优先：地图 API 半真实线路

如果时间允许，可以调用高德地图或百度地图开放平台，获取真实公交线路和站点

地图 API 适合获取：

```text
公交线路名称
起点站和终点站
途经站点
站点顺序
站点经纬度
首末班时间
线路轨迹
```

注意事项：

```text
API Key 不要提交到 Git
地图 API 只负责基础线路和坐标
系统运行时优先读取本地 CSV 或数据库，不要每次实时请求地图 API
具体接口参数以高德或百度开放平台最新文档为准
```

地图 API 不负责解决：

```text
真实车上人数
真实拥挤等级
真实历史客流
真实车辆实时位置
项目内的拥挤预测标签
```

## (三) ETA 数据说明

本项目中的 ETA 指“公交车到达用户选择的起始站点还需要多久”

如果没有官方实时公交接口，不能直接依赖普通地图 App 的内部 ETA。项目第一阶段建议自己计算：

```text
ETA =
车辆当前位置到目标站点的剩余站间时间
+ 经过站点的停靠时间
+ 高峰、天气、拥挤带来的延迟
```

因此需要准备：

```text
站点顺序
站间距离
平均站间耗时
车辆当前所在站点
车辆下一站
车辆在线路中的行驶进度
站点停靠时间
时段系数
天气系数
拥挤延迟系数
```

# 六、拥挤等级规则

项目计划书中拥挤程度分为 4 档，数据中建议使用英文编码，前端展示时再映射成中文

| **编码** | **中文展示** | **规则** | **含义** |
|---|---|---|---|
| `free` | 空闲 | `0.00 <= crowd_rate < 0.40` | 车内人数较少 |
| `comfortable` | 适中 | `0.40 <= crowd_rate < 0.70` | 正常乘坐，体验较好 |
| `crowded` | 拥挤 | `0.70 <= crowd_rate < 0.90` | 可能需要站立 |
| `very_crowded` | 非常拥挤 | `0.90 <= crowd_rate` | 不推荐乘坐 |

计算方式：

```text
crowd_rate = onboard_count / capacity
```

# 七、模拟数据生成规则

## (一) 时间规则

```text
早高峰：7:00-9:00
午高峰：11:30-13:30
晚高峰：17:00-19:00
普通时段：其他运营时间
周末：教学楼客流下降，地铁站、商业区客流上升
```

## (二) 站点规则

```text
宿舍区、学生公寓：早高峰上车人数高
教学楼、实验楼：早高峰下车人数高，晚高峰上车人数高
食堂：午高峰上下车人数高
图书馆：下午和晚间客流较高
地铁站、校门：早晚高峰换乘客流高
```

## (三) 天气规则

```text
晴天：正常速度和正常客流
雨天：公交需求上升，车辆速度下降
高温或低温：公交需求略有上升
```

## (四) 车辆运行规则

```text
每辆车绑定一条线路
车辆按站点顺序循环移动
每隔 5 秒或 10 秒更新一次位置
到站后停靠一定时间
高峰期速度降低
拥挤时停靠时间增加
```

# 八、第一阶段数据规模

第一阶段不要追求大规模，先保证功能闭环

建议规模：

```text
3 条线路
20-30 个站点
6-10 辆车
30 天历史客流
每天 6:00-22:00 按小时生成记录
运行时模拟车辆每 5 秒或 10 秒更新一次
```

完成后应该能支持：

```text
线路和站点查询
地图展示站点和线路
车辆当前位置展示
ETA 到站时间计算
拥挤等级展示

AI 助手解释推荐原因
```

# 十、验收标准

数据收集任务完成后，至少要满足：

```text
前端能在地图上展示站点和线路
后端能查询线路、站点、车辆和拥挤数据
ETA 模块能根据车辆位置计算到站时间
拥挤预测模块有可训练或可规则判断的数据
推荐模块能拿到 ETA、拥挤等级、换乘次数、步行距离等字段
AI 助手能拿到结构化上下文并生成解释
```

一句话总结：

> 地图数据负责“看起来真实”，模拟数据负责“系统能跑通”，历史客流负责“模型能预测”，推荐样例负责“答辩能演示”
