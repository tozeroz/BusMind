# 三、数据与算法方案

## 六、数据来源方案

由于公交企业真实 GPS 数据和车内客流数据通常不公开，本项目采用“半真实数据 + 仿真数据 + 模拟历史数据”的方式构建数据集。

整体数据来源方案如下：

```text
真实/半真实线路站点数据
        +
仿真车辆运行数据
        +
模拟历史客流数据
        +
天气、时段、星期等辅助特征
        ↓
用于 ETA 计算、拥挤预测、舒适推荐和可视化展示
```

### 6.1 线路与站点数据

线路与站点数据可以参考校园公交或城市公交场景进行设计。

| 数据         | 来源                    | 说明                                     |
| ------------ | ----------------------- | ---------------------------------------- |
| 公交线路名称 | 手工设计 / 地图参考     | 例如校园 1 号线、校园 2 号线、地铁接驳线 |
| 站点名称     | 手工整理                | 宿舍区、教学楼、图书馆、食堂、地铁站等   |
| 站点经纬度   | 高德地图 / 百度地图拾取 | 用于地图展示                             |
| 站点顺序     | 手工配置                | 用于计算车辆运行方向                     |
| 站间距离     | 手工估算 / 经纬度计算   | 用于 ETA 计算                            |

示例线路：

```text
校园 1 号线：
北门 → 宿舍区 → 食堂 → 教学楼 → 图书馆 → 体育馆 → 南门

校园 2 号线：
地铁站 → 西门 → 实验楼 → 教学楼 → 行政楼 → 东门

校园 3 号线：
学生公寓 → 医务室 → 食堂 → 图书馆 → 创新楼 → 地铁站
```

### 6.2 实时车辆数据

实时车辆数据采用 Python 脚本定时模拟生成。

模拟规则如下：

| 模拟内容 | 规则说明                                   |
| -------- | ------------------------------------------ |
| 车辆运行 | 每辆车绑定一条线路，在线路站点之间循环移动 |
| 位置更新 | 每隔 5 秒或 10 秒更新一次车辆位置          |
| 到站停靠 | 车辆到站后停留一定时间                     |
| 高峰影响 | 早晚高峰车辆速度下降                       |
| 天气影响 | 雨天速度下降                               |
| 拥挤影响 | 拥挤车辆上下客时间增加                     |

核心字段：

| 字段               | 含义                      |
| ------------------ | ------------------------- |
| vehicle_id         | 车辆编号                  |
| line_id            | 所属线路                  |
| current_station_id | 当前站点                  |
| next_station_id    | 下一站点                  |
| progress           | 当前两站之间行驶进度      |
| speed              | 当前速度                  |
| onboard_count      | 当前车上人数              |
| capacity           | 车辆容量                  |
| status             | 正常 / 延误 / 拥挤 / 离线 |
| update_time        | 更新时间                  |

### 6.3 历史客流数据

历史客流数据由 Python 脚本生成，用于训练或验证拥挤预测模型。

生成规则如下：

| 场景               | 规则                                           |
| ------------------ | ---------------------------------------------- |
| 早高峰 7:00-9:00   | 宿舍区、地铁站上车人数增加，教学楼下车人数增加 |
| 午高峰 11:30-13:30 | 教学楼、食堂附近客流增加                       |
| 晚高峰 17:00-19:00 | 教学楼、图书馆上车人数增加，宿舍区下车人数增加 |
| 雨天               | 公交需求增加，车辆速度下降                     |
| 周末               | 教学楼客流下降，地铁站、商业区客流上升         |
| 特殊事件           | 某站点短时间客流异常升高，用于预警演示         |

核心字段：

| 字段           | 含义         |
| -------------- | ------------ |
| record_id      | 记录编号     |
| date           | 日期         |
| hour           | 小时         |
| weekday        | 星期         |
| line_id        | 线路编号     |
| station_id     | 站点编号     |
| weather        | 天气         |
| temperature    | 温度         |
| passenger_up   | 上车人数     |
| passenger_down | 下车人数     |
| onboard_count  | 当前车上人数 |
| capacity       | 车辆容量     |
| crowd_rate     | 拥挤率       |
| crowd_level    | 拥挤等级     |

拥挤率计算规则：

```text
拥挤率 = 当前车上人数 / 车辆容量
```

拥挤等级划分：

```text
0% - 40%：空闲
40% - 70%：适中
70% - 90%：拥挤
90% 以上：非常拥挤
```

## 七、核心算法方案

### 7.1 ETA 到站时间计算

ETA，即 Estimated Time of Arrival，表示预计到站时间。

本项目的 ETA 不强依赖真实公交公司的实时到站接口，而是基于车辆模拟位置、站间距离、平均速度、时段系数、天气系数和拥挤延迟进行计算。

计算公式：

```text
ETA =
剩余站间基础时间 × 时段系数 × 天气系数
+ 站点停靠时间
+ 拥挤延迟
```

其中：

| 因素             | 说明                                     |
| ---------------- | ---------------------------------------- |
| 剩余站间基础时间 | 根据车辆当前位置和目标站点之间的距离计算 |
| 时段系数         | 高峰期车辆速度较慢，因此系数增大         |
| 天气系数         | 雨天、恶劣天气导致速度下降               |
| 站点停靠时间     | 每经过一个站点需要停靠一定时间           |
| 拥挤延迟         | 拥挤车辆上下客时间更长                   |

示例：

```text
车辆距离目标站点还有 3 站；
平均每站需要 3 分钟；
早高峰系数为 1.3；
每站停靠 0.5 分钟。

ETA = 3 × 3 × 1.3 + 3 × 0.5 = 13.2 分钟
```

### 7.2 拥挤程度预测

拥挤程度预测是本项目的核心算法功能。

本项目计划采用两阶段实现：

#### 第一阶段：规则模型

先根据时段、站点热度、天气、当前载客率等因素构建规则模型。

```text
拥挤分数 =
当前载客率得分
+ 时段影响得分
+ 站点热度得分
+ 天气影响得分
+ 星期影响得分
```

根据拥挤分数划分为空闲、适中、拥挤、非常拥挤。

#### 第二阶段：机器学习模型

在规则模型基础上，使用模拟历史客流数据训练 XGBoost 或随机森林模型。

输入特征包括：

```text
line_id
station_id
hour
weekday
is_peak
weather
temperature
historical_avg_passenger
current_onboard_count
capacity
```

输出结果：

```text
crowd_level：空闲 / 适中 / 拥挤 / 非常拥挤
```

模型选择建议：

| 模型     | 说明                                                       |
| -------- | ---------------------------------------------------------- |
| 规则模型 | 实现简单，适合作为基础版本                                 |
| 随机森林 | 稳定性较好，适合分类任务                                   |
| XGBoost  | 效果较好，适合表格特征预测                                 |
| LSTM     | 更适合长时间序列，但实现成本较高，短期项目不建议作为主模型 |

本项目建议优先使用 **规则模型 + XGBoost**。规则模型保证系统可运行，XGBoost 作为算法亮点。

### 7.3 舒适出行推荐算法

舒适推荐模块综合考虑到站时间、拥挤程度、步行距离和换乘次数，为用户推荐更合适的出行方案。

推荐评分公式：

```text
推荐分数 =
ETA 权重 × ETA 得分
+ 舒适度权重 × 拥挤程度得分
+ 步行距离权重 × 步行得分
+ 换乘次数权重 × 换乘得分
```

默认权重：

```text
score = 0.35 × ETA得分
      + 0.40 × 舒适度得分
      + 0.15 × 步行距离得分
      + 0.10 × 换乘得分
```

不同用户偏好可以调整权重：

| 用户偏好   | 权重调整         |
| ---------- | ---------------- |
| 想最快到达 | 提高 ETA 权重    |
| 不想太挤   | 提高舒适度权重   |
| 不想走太远 | 提高步行距离权重 |
| 不想换乘   | 提高换乘次数权重 |
| 综合推荐   | 各指标均衡       |

推荐结果可以分为：

| 推荐类型     | 说明                                     |
| ------------ | ---------------------------------------- |
| 最快方案     | 总耗时最短                               |
| 最舒适方案   | 拥挤程度最低                             |
| 最少换乘方案 | 换乘次数最少                             |
| 综合推荐方案 | 时间、舒适度、步行距离、换乘次数综合最优 |

### 7.4 大模型助手方案

大模型助手主要用于自然语言交互、推荐解释和个性化建议生成。

工作流程：

```text
用户输入自然语言问题
        ↓
系统解析用户起点、终点、时间、偏好
        ↓
调用公交查询、ETA、拥挤预测、舒适推荐结果
        ↓
将结构化结果作为上下文传给大模型
        ↓
大模型生成自然语言出行建议
```

示例输入：

```text
我现在从宿舍区去教学楼，不想太挤，应该坐哪班车？
```

系统结构化数据：

```json
{
  "start": "宿舍区",
  "end": "教学楼",
  "preference": "comfort",
  "routes": [
    {
      "line": "校园1号线",
      "eta": 5,
      "crowd_level": "拥挤",
      "arrival_time": 12
    },
    {
      "line": "校园2号线",
      "eta": 9,
      "crowd_level": "适中",
      "arrival_time": 18
    }
  ]
}
```

大模型输出：

```text
如果你更重视舒适度，建议选择校园2号线。虽然它比校园1号线晚到约6分钟，但当前拥挤程度较低，乘坐体验更好。如果你比较赶时间，可以选择校园1号线，但可能需要站立。
```
