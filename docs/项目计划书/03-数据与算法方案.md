# 三、数据与算法方案

## 六、数据来源方案

本项目的算法方案以 LTA DataMall 的公开数据结构为参考。真实数据可用时优先采集真实数据；真实数据不足时，按照相同字段结构生成仿真数据，保证系统第一阶段可以训练、验证和演示，后续也能平滑替换为真实数据源。

整体数据方案如下：

```text
LTA DataMall 风格数据 / 仿真数据
        ↓
构造 ETA 标签和客载状态标签
        ↓
按时间切分训练集和测试集
        ↓
训练 ETA 预测模型和客载状态预测模型
        ↓
将 predicted_eta_minutes 和 predicted_load 输入推荐模块
        ↓
输出出行体验评分、推荐方案和推荐理由
```

### 6.1 主要数据集

| **数据集** | **关键字段** | **使用方式** |
|---|---|---|
| `Bus Arrival` | `EstimatedArrival`、车辆位置、`Load` | 构造 ETA 标签和客载状态标签 |
| `Bus Stops` | 站点编号、站点名称、经纬度 | 站点匹配、地图展示、步行距离计算 |
| `Bus Routes` | 线路、方向、站点顺序、距离 | 构造 ETA 特征、计算候选路线 |
| `Bus Services` | 线路起终点、运营方向、发车频率 | 构造 ETA 和客载状态特征 |
| `Passenger Volume by Bus Stops` | 站点小时级 tap in / tap out 客流 | 构造站点热度和时段客流特征 |
| `Passenger Volume by Origin Destination Bus Stops` | OD 站点对、小时级出行量 | 构造起终点需求和线路方向压力特征 |
| 天气数据 | 天气、温度、降雨状态 | 构造 ETA 和客载状态辅助特征 |

### 6.2 标签构造

`Bus Arrival` 中的 `EstimatedArrival` 不作为系统最终 ETA 输出，而是用于构造模型训练标签。

```text
eta_label = EstimatedArrival - query_time
```

其中 `query_time` 表示系统请求或采样 `Bus Arrival` 数据的时间，`eta_label` 表示查询时刻距离公交预计到站还剩多少分钟。

`Bus Arrival` 中的 `Load` 不表示精确车上人数，而是客载状态分类标签。

| **Load 标签** | **含义** | **项目展示** |
|---|---|---|
| `SEA` | Seats Available | 有座位，较宽松 |
| `SDA` | Standing Available | 可站立，适中 |
| `LSD` | Limited Standing | 站立空间有限，较拥挤 |

因此，本项目预测的是客载状态，而不是精确载客人数。

### 6.3 训练集与测试集划分

本项目采用时间切分方式构造训练集和测试集：

```text
第 1-30 天：训练集
第 31 天：测试集
```

训练集用于：

```text
训练 ETA 预测模型
训练客载状态预测模型
计算 baseline
```

测试集用于：

```text
验证 ETA 预测效果
验证客载状态预测效果
验证推荐结果是否合理
```

重要原则：

```text
第 31 天数据不能参与训练
第 31 天只作为 ground truth
预测结果必须和第 31 天真实或仿真标签对比
```

如果暂时无法获取真实连续 31 天数据，可以先生成 LTA 字段风格的 31 天仿真数据。评价方式保持一致，即用前 30 天训练，用第 31 天验证。

## 七、核心算法方案

当前核心算法由三部分组成：

```text
ETA 预测
    ↓
客载状态预测
    ↓
出行体验评价与推荐
```

一句话说明：

```text
ETA 预测负责判断“车多久到”
客载状态预测负责判断“车内体验如何”
推荐算法负责判断“哪条路线更适合用户”
```

### 7.1 ETA 预测模型

ETA 预测是回归任务，用于预测公交到达用户上车站点还需要多少分钟。

预测目标：

```text
predicted_eta_minutes
```

训练标签：

```text
eta_label = EstimatedArrival - query_time
```

特征设计：

| **特征** | **来源** | **说明** |
|---|---|---|
| `service_no` | `Bus Arrival` / `Bus Routes` | 公交线路 |
| `bus_stop_code` | `Bus Arrival` / `Bus Stops` | 上车站点 |
| `direction` | `Bus Routes` | 行驶方向 |
| `stop_sequence` | `Bus Routes` | 站点顺序 |
| `distance_to_stop` | `Bus Routes` / 车辆位置 | 车辆到目标站点距离 |
| `hour` | 查询时间 | 小时 |
| `day_type` | 时间字段 | 工作日 / 周末 |
| `service_frequency` | `Bus Services` | 发车频率 |
| `weather` | 天气数据 | 天气情况 |
| `previous_eta` | 历史到站数据 | 上一时间片 ETA |
| `previous_load` | `Bus Arrival Load` | 上一时间片客载状态 |

模型选择：

```text
Linear Regression
Decision Tree Regressor
Random Forest Regressor
XGBoost Regressor
```

第一阶段建议先实现 Linear Regression 作为简单模型，再使用 Random Forest Regressor 或 XGBoost Regressor 作为主要模型。

ETA baseline 用于评价模型效果：

| **Baseline** | **方法** | **作用** |
|---|---|---|
| 平均值 baseline | 永远预测训练集 ETA 平均值 | 弱 baseline |
| 历史均值 baseline | 预测同线路、同站点、同小时的历史平均 ETA | 业务 baseline |

评价指标：

```text
MAE = avg(|predicted_eta_minutes - eta_label|)
RMSE = sqrt(avg((predicted_eta_minutes - eta_label)^2))
```

重点展示模型 MAE、RMSE 是否低于 baseline。

### 7.2 客载状态预测模型

客载状态预测是三分类任务，用于预测候选公交在用户乘坐时的 `Load` 状态。

预测目标：

```text
predicted_load
```

训练标签：

```text
load_label = Load
```

分类标签：

```text
SEA / SDA / LSD
```

特征设计：

| **特征** | **来源** | **说明** |
|---|---|---|
| `service_no` | `Bus Arrival` / `Bus Routes` | 公交线路 |
| `bus_stop_code` | `Bus Arrival` / `Bus Stops` | 查询站点 |
| `direction` | `Bus Routes` | 行驶方向 |
| `stop_sequence` | `Bus Routes` | 站点在线路中的顺序 |
| `hour` | 查询时间 | 小时 |
| `day_type` | 客流数据 | 工作日 / 周末 |
| `tap_in_volume` | `Passenger Volume by Bus Stops` | 站点上车客流 |
| `tap_out_volume` | `Passenger Volume by Bus Stops` | 站点下车客流 |
| `od_trips` | `Passenger Volume by OD Bus Stops` | 起终点需求 |
| `service_frequency` | `Bus Services` | 发车频率 |
| `weather` | 外部天气或仿真 | 天气情况 |
| `previous_load` | 历史 `Bus Arrival` | 上一时间片客载状态 |

模型选择：

```text
Logistic Regression
Decision Tree
Random Forest
XGBoost
```

第一阶段建议使用 Logistic Regression 作为简单模型，使用 Random Forest 或 XGBoost 作为主要模型。

客载状态 baseline 用于评价模型效果：

| **Baseline** | **方法** | **作用** |
|---|---|---|
| 多数类 baseline | 永远预测训练集中出现最多的 `Load` 类别 | 弱 baseline |
| 历史众数 baseline | 预测同线路、同站点、同小时最常见的 `Load` | 业务 baseline |

评价指标：

```text
Accuracy：整体分类准确率
Macro-F1：三类 Load 的 F1 平均值
Confusion Matrix：查看不同等级之间的误判情况
```

重点关注：

```text
LSD 被预测成 SEA：严重低估拥挤，风险最高
SEA 被预测成 LSD：高估拥挤，会降低推荐体验
```

输出字段：

| **字段** | **说明** |
|---|---|
| `predicted_load` | 预测客载状态，取值 `SEA`、`SDA`、`LSD` |
| `load_score` | 客载体验得分 |
| `model_name` | 使用的模型，例如 `random_forest`、`xgboost` |
| `confidence` | 预测置信度 |

客载体验得分：

```text
SEA = 100
SDA = 70
LSD = 35
```

### 7.3 出行体验评价与推荐

出行体验评价与推荐模块综合 ETA、客载状态、步行时间和换乘次数，为用户输出不同类型的推荐方案。

候选方案字段：

| **字段** | **说明** |
|---|---|
| `route_id` | 推荐方案编号 |
| `service_no` | 公交线路编号 |
| `start_stop_code` | 上车站点 |
| `end_stop_code` | 下车站点 |
| `predicted_eta_minutes` | 预测等车时间 |
| `walk_time` | 到站步行时间 |
| `ride_time` | 车内行驶时间 |
| `time_cost` | 总时间成本 |
| `predicted_load` | 预测客载状态 |
| `load_score` | 客载体验得分 |
| `transfer_count` | 换乘次数 |
| `experience_score` | 出行体验综合分 |
| `recommend_type` | 推荐类型 |
| `reason` | 推荐原因 |

时间成本：

```text
time_cost = walk_time + predicted_eta_minutes + ride_time
```

评分公式：

```text
experience_score =
w_load * load_score
+ w_time * time_score
+ w_walk * walk_score
+ w_transfer * transfer_score
```

其中：

```text
w_load + w_time + w_walk + w_transfer = 1
```

默认综合推荐权重：

| **指标** | **默认权重** | **含义** |
|---|---:|---|
| `load_score` | `0.35` | 车内体验 |
| `time_score` | `0.30` | ETA 和总耗时 |
| `walk_score` | `0.20` | 步行负担 |
| `transfer_score` | `0.15` | 换乘复杂度 |

不同用户偏好可以调整权重：

| **用户偏好** | **`w_load`** | **`w_time`** | **`w_walk`** | **`w_transfer`** | **说明** |
|---|---:|---:|---:|---:|---|
| `balanced` | `0.35` | `0.30` | `0.20` | `0.15` | 综合推荐，各指标相对均衡 |
| `comfort` | `0.55` | `0.20` | `0.15` | `0.10` | 更重视车内不拥挤 |
| `fastest` | `0.20` | `0.55` | `0.15` | `0.10` | 更重视总耗时 |
| `less_walk` | `0.25` | `0.25` | `0.40` | `0.10` | 更重视少走路 |
| `less_transfer` | `0.25` | `0.25` | `0.15` | `0.35` | 更重视少换乘 |

推荐类型：

| **推荐类型** | **选择规则** |
|---|---|
| `fastest` | `time_cost` 最小 |
| `best_experience` | 当前用户偏好下 `experience_score` 最高 |
| `least_transfer` | `transfer_count` 最少，若相同则体验分更高 |
| `least_crowded` | `load_score` 最高，若相同则时间成本更低 |

### 7.4 大模型助手方案

大模型助手主要用于自然语言交互、推荐解释和个性化建议生成。大模型不直接训练 ETA 或客载状态模型，而是读取系统结构化结果，再把推荐方案解释成用户容易理解的自然语言。

工作流程：

```text
用户输入自然语言问题
        ↓
系统解析用户起点、终点、时间、偏好
        ↓
调用线路查询、ETA 预测、客载状态预测和推荐结果
        ↓
将结构化结果作为上下文传给大模型
        ↓
大模型生成自然语言出行建议
```

示例结构化数据：

```json
{
  "start": "宿舍区",
  "end": "教学楼",
  "preference": "comfort",
  "routes": [
    {
      "line": "校园1号线",
      "predicted_eta_minutes": 5,
      "predicted_load": "LSD",
      "experience_score": 68
    },
    {
      "line": "校园2号线",
      "predicted_eta_minutes": 9,
      "predicted_load": "SDA",
      "experience_score": 82
    }
  ]
}
```

示例输出：

```text
如果你更重视乘坐体验，建议选择校园2号线。虽然它预计晚到约4分钟，但预测客载状态为可站立，车内空间更宽松，综合体验分更高。
```

## 八、第一阶段实现重点

第一阶段优先完成以下内容：

```text
1. 整理 LTA 风格字段结构
2. 构造 ETA 预测训练样本，标签为 eta_label
3. 构造客载状态预测训练样本，标签为 Load
4. 分别训练 ETA 回归模型和 Load 分类模型
5. 为 ETA 设置平均值 baseline 和历史均值 baseline
6. 为 Load 设置多数类 baseline 和历史众数 baseline
7. 用第 31 天测试集评价 MAE、RMSE、Accuracy、Macro-F1
8. 将 predicted_eta_minutes 和 predicted_load 接入出行体验评分
```

第一阶段需要输出：

| **输出内容** | **说明** |
|---|---|
| LTA 风格训练数据 | 可来自真实数据，也可先用同字段仿真数据 |
| ETA 预测结果 | `predicted_eta_minutes` |
| 客载状态预测结果 | `predicted_load`、`load_score` |
| 模型评价结果 | ETA 的 MAE/RMSE，Load 的 Accuracy/Macro-F1 |
| 推荐结果 | 最快、车内更宽松、最少换乘、综合推荐方案 |

一句话总结：

> 训练集用于训练 ETA 预测模型和客载状态预测模型，测试集用于验证预测效果，推荐算法使用两个模型的预测结果进行出行体验评分。
