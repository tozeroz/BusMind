# 2 关键技术分析

本项目围绕公交出行过程中“路线是否可达、换乘是否方便、车辆是否拥挤、推荐是否可信、结果是否容易理解”等问题，综合采用地理空间数据处理、实时数据缓存与容错降级、大模型结构化解释以及个性化路线推荐等技术。Vue3、FastAPI、MySQL 等属于项目采用的开发框架或工具，本章重点分析其背后的数据处理、图搜索、系统优化和智能解释技术。

## 2.1 地理空间数据处理与多维可视化技术

地理空间数据处理是指对带有经纬度、空间位置及拓扑关系的数据进行组织、转换、匹配和分析的技术。在公交系统中，站点是空间点对象，线路是由多个有序站点构成的线对象，车辆位置是随时间变化的动态点对象，道路拥堵和站点客流则属于带有空间分布特征的状态数据。

本项目首先使用经纬度表示站点、车辆和道路位置，并通过空间距离计算匹配用户起点、终点附近的公交站点。对于两个坐标点，可使用 Haversine 公式估算其球面距离：

```text
d = 2R × arcsin(√(sin²(Δφ/2) + cosφ1 × cosφ2 × sin²(Δλ/2)))
```

其中，`R` 为地球平均半径，`φ` 表示纬度，`λ` 表示经度。系统可以按照距离由近到远筛选若干上车站和下车站，并在缺少真实步行导航数据时，利用直线距离乘以绕行系数估算步行距离和步行时间。

在数据表达方面，系统采用 GeoJSON 组织地图对象：公交站点和车辆位置使用 `Point` 表示，线路走向和推荐路径使用 `LineString` 表示，客流或道路拥堵区域可以使用带等级属性的点、线或面对象表示。前端读取 GeoJSON 后，根据对象类型和业务属性完成分层渲染：站点以图标或圆点显示，公交线路以不同颜色的线条显示，车辆位置以动态标记显示，客流压力和道路拥堵程度则通过颜色渐变、点大小或热力图展示。历史客流趋势和推荐五维得分采用统计图表展示，实现空间信息与数值信息的联合表达。

该技术的特点是能够把分散的线路、站点、车辆和客流数据统一映射到地图空间中，使用户直观理解公交运行状态和推荐路线。其优点是信息密度高、交互性强，并且有利于展示路线之间的空间差异；不足之处是地图对象较多时会增加浏览器渲染压力，坐标误差也可能造成站点和道路匹配偏差。此外，矢量瓦片加载、不同缩放级别下的要素聚合以及移动端显示空间有限，都是实际应用中需要处理的问题。本项目通过图层分级加载、视野范围过滤和按等级简化显示等方式降低渲染负担。

## 2.2 实时公交数据缓存与容错降级技术

公交到站时间、车辆位置和客载状态具有明显的实时性，但外部数据接口可能出现请求超时、更新延迟、字段缺失或临时不可用等情况。如果每次查询都直接依赖外部接口，不仅会增加响应时间，也可能因接口波动导致整个推荐服务失败。因此，本项目采用实时数据缓存、数据新鲜度判断和分级降级相结合的技术，提高系统的稳定性和可用性。

缓存技术的基本思想是把最近一次成功获取的实时数据暂存在内存或数据库中，并记录数据获取时间。用户发起查询时，系统优先判断缓存是否存在以及数据是否仍在有效期内：缓存新鲜时直接返回；缓存过期时尝试重新请求实时接口；实时接口失败时，在允许的时间范围内返回最近一次缓存，并在结果中标记数据来源和更新时间。公交网络有向图等变化频率较低的数据则构建为内存快照，避免每次搜索都重新读取线路站序并建图；当线路数据更新时，再主动刷新图快照。

系统通过 `data_freshness_seconds` 表示数据年龄，并将新鲜度、实时监控状态和字段完整度共同转换为可靠性得分：

```text
reliability_score =
0.5 × freshness_score
+ 0.3 × monitored_score
+ 0.2 × completeness_score
```

当实时数据无法获得时，系统按照数据类型采取不同的降级策略：缺少 ETA 时，根据线路平均发车间隔估算等待时间；缺少 Load 时，根据历史站点客流和高峰时段估算客载压力；缺少道路速度数据时，使用线路或道路历史平均速度；缺少步行路径时，使用坐标直线距离乘绕行系数估算；缺少历史客流时，使用中性默认值。发生降级的字段会降低完整度或可靠性得分，使用户能够区分实时结果和估算结果。

该技术的优点是能够减少重复接口调用、缩短响应时间，并保证部分数据源异常时系统仍可返回可用结果；缺点是缓存数据与真实状态之间可能存在时间差，默认值也可能降低推荐精度。主要挑战在于有效期设置：有效期过短会增加接口压力，过长则可能返回过时状态。因此，本项目根据数据变化速度设置不同缓存策略，并通过更新时间、数据来源和可靠性评分向后续推荐模块传递数据质量信息。

## 2.3 基于结构化数据增强的大模型推荐解释技术

大模型擅长理解自然语言和生成连贯文本，但如果直接要求其计算 ETA、判断车辆客载或选择路线，容易出现数据失真和不可验证的问题。本项目采用基于结构化数据增强的大模型推荐解释技术，即先由公交查询和推荐算法完成确定性计算，再将结构化结果作为上下文交给大模型生成面向用户的解释。

该技术与检索增强生成思想相似，核心是利用外部可信数据约束大模型输出。系统向大模型提供的上下文包括起点、终点、用户偏好、候选线路、ETA、客载状态、历史客流、道路拥堵、步行时间、换乘次数、五维得分、综合推荐分和结构化推荐原因。例如，算法可先生成“等待时间较短”“当前客载适中”“历史客流压力较低”“无需换乘”等原因字段，再由大模型将这些字段组织成自然、连贯的出行建议。

大模型在本项目中的处理流程如下：

```text
用户自然语言问题
→ 提取起点、终点和出行偏好
→ 调用公交查询与推荐算法
→ 生成结构化路线结果和推荐原因
→ 将受约束的上下文传入大模型
→ 输出自然语言解释
```

为了减少幻觉，提示词中需要明确大模型的能力边界：不得自行计算或修改 ETA、Load 和推荐分数；不得编造结构化上下文中不存在的线路和站点；信息缺失时应说明“数据暂缺”或“结果为估算值”。系统还可以限制输出格式和长度，并在返回前校验线路编号、时间和得分等关键字段是否与算法结果一致。

该技术的优点是兼顾算法结果的确定性与自然语言交互的灵活性，用户不仅能看到推荐结论，还能理解推荐依据；同时，结构化上下文便于追踪和校验。其不足是大模型仍可能出现表达偏差，并且调用外部模型会带来响应延迟和服务成本。主要挑战包括自然语言中的地点歧义、用户偏好识别错误以及模型输出稳定性。本项目通过结构化输入、规则边界、异常回退和固定解释模板降低这些风险；当大模型不可用时，系统仍可直接返回结构化推荐原因。

## 2.4 基于多源数据融合的多目标个性化路线推荐技术

该技术是本项目的核心。系统不是只寻找总时间最短的公交路线，而是先利用公交网络图生成若干可达候选路线，再融合实时到站、实时客载、历史客流、道路拥堵、步行距离、换乘次数和数据可靠性，对候选路线进行多目标评价，最后根据用户偏好输出 Top-K 推荐结果。

### 2.4.1 公交网络有向状态图建模

普通站点图通常只用站点作为节点，但这种表示难以准确判断乘客当前所在线路，也无法区分“沿原线路继续乘坐”和“在同一站换乘其他线路”。因此，本项目采用有向状态图描述公交网络，将一个节点定义为：

```text
GraphNode = (line_id, stop_sequence)
```

其中，`line_id` 表示公交线路，`stop_sequence` 表示该站在线路中的顺序。节点再映射到实际的 `station_id`。使用“线路 + 站序”作为状态还有一个重要作用：同一公交线路可能在环线中多次经过同一站点，这些位置拥有不同的站序，因此不会被错误地合并成一条捷径。

图中包含两类有向边：

- 乘车边：连接同一线路中相邻站序节点，只允许沿线路运行方向前进，边权为 0；
- 换乘边：连接同一实际站点上属于不同线路的状态节点，边权为 1。

这里的边权不表示行驶时间，而表示是否新增一次换乘。首次选择起始线路不计为换乘，同一线路继续乘坐也不增加换乘数，只有从当前线路切换到另一条线路时才增加 1。图结构由线路站序表构建，并保存线路名称、节点—站点映射、同站可换乘节点集合及乘车邻接边。由于公交线路变化频率远低于车辆实时状态，构建后的图可以以内存快照形式复用。

这种建模能够同时表达线路方向、站点顺序和换乘关系，适合最少换乘搜索。其不足是节点数量比普通站点图更多，换乘枢纽处可能产生较多换乘连接；如果基础站序或上下行方向数据错误，搜索结果也会受到直接影响。因此需要在建图前清洗重复站序、无效线路和错误站点关联。

### 2.4.2 基于 0-1 BFS 的最少换乘路径搜索

0-1 BFS 是专门处理边权只包含 0 和 1 的最短路算法。本项目的乘车边权为 0，换乘边权为 1，恰好满足其使用条件。算法使用双端队列维护待访问状态：沿同一线路前进时不增加换乘数，将新状态加入队首；发生换乘时换乘数增加 1，将新状态加入队尾。这样，换乘次数较少的状态会被优先搜索。

核心过程可以表示为：

```text
dist[start] = 0
将所有起点站对应的线路状态加入双端队列

while 队列非空:
    取出队首状态 u
    对每条乘车边 u → v:
        若 dist[v] > dist[u]:
            dist[v] = dist[u]
            v 加入队首
    对每条换乘边 u → v:
        若 dist[v] > dist[u] + 1:
            dist[v] = dist[u] + 1
            v 加入队尾
```

在邻接表表示下，其时间复杂度为 `O(V+E)`，空间复杂度为 `O(V)`，比对所有边统一使用 Dijkstra 更适合该二值边权场景。系统先计算起终点之间的最少换乘数，再在用户允许的最大换乘数范围内枚举有限数量的候选路线，并通过线路序列和换乘次数进行去重。搜索状态中记录已经使用过的线路，避免重复乘坐同一线路造成循环；同时通过 `max_transfer` 和 `max_candidates` 限制搜索范围。

该算法的优势是换乘目标明确、复杂度低，能够稳定生成直达、一次换乘或多次换乘路线；局限是它只优化换乘次数，不能单独保证总时间、步行距离或舒适度最优。因此，0-1 BFS 只负责生成以少换乘为重点的候选路线，最终选择仍由后续多目标评分完成。

### 2.4.3 多源公交数据特征融合

候选路线生成后，系统需要将不同来源的数据整理为统一的路线特征。主要数据包括 Bus Arrival 提供的 ETA、Load、车辆位置和监控状态，Bus Routes 与 Bus Services 提供的站序、线路和发车频率，Passenger Volume 提供的历史站点客流，以及 Traffic Speed Bands 提供的道路通畅程度。

由于不同指标的单位和方向不同，系统先进行字段解析、等级映射和缺失值处理。例如，Load 的 `SEA`、`SDA`、`LSD` 分别映射为 100、70、35 分；历史站点上下车量按照同类日期和小时分位数划分为低、中、高客流；道路速度等级映射为道路通畅分。每条候选路线最终形成时间、舒适、步行、换乘和可靠性等统一特征。

多源数据融合能够比单纯按时间推荐更全面地反映出行体验，但其难点在于数据更新频率不同、空间对象匹配不精确以及部分实时字段缺失。本项目通过统一字段契约、时间戳、新鲜度评分、来源标记和降级字段记录，使推荐模块能够同时利用数据内容和数据质量。

### 2.4.4 五维出行体验评价

系统将不同量纲的路线特征统一转换为 0～100 分，形成五维评价指标：时间效率、舒适度、步行体验、换乘体验和数据可靠性。

总时间由步行、等待和乘车时间组成：

```text
total_time_minutes = walk_time_minutes + eta_minutes + ride_time_minutes
time_score = max(0, 100 - 2 × total_time_minutes)
```

舒适度综合实时客载、历史客流和道路通畅程度：

```text
comfort_score =
0.5 × load_score
+ 0.3 × history_flow_score
+ 0.2 × congestion_score
```

步行得分随步行时间增加而降低；换乘得分按照 0 次、1 次、2 次和 3 次以上分别设置等级；可靠性得分则由数据新鲜度、监控可信度和字段完整度共同决定。五维评分既能用于综合推荐，也能直接展示给用户，说明一条路线在哪些方面占优。

该方法结构简单、含义明确且便于解释，但规则阈值和权重具有一定经验性，不一定适合所有城市和用户。后续可根据真实选择日志校准评分函数，或引入 TOPSIS、Pareto 前沿等多指标决策方法进行对比优化。

### 2.4.5 用户偏好动态权重与 Top-K 排序

完成五维评分后，系统按照用户偏好组合各项得分：

```text
recommend_score =
w_time × time_score
+ w_comfort × comfort_score
+ w_walk × walk_score
+ w_transfer × transfer_score
+ w_reliability × reliability_score
```

所有权重之和为 1。综合推荐使用较均衡的权重；“最快到达”提高时间权重；“舒适优先”提高舒适度权重；“少步行”和“少换乘”分别提高步行或换乘维度权重。系统按照综合推荐分对候选路线排序并输出 Top-K，同时分别标记综合推荐、最快到达、车内更宽松、换乘更少、步行更少和数据更可靠的路线，方便用户比较。

动态权重的优点是计算开销小、结果透明，同一组候选路线能够根据不同需求产生不同排序；不足是预设偏好无法完全刻画用户的真实选择。后续可以记录用户最终选择的路线，对权重进行持续校准，但需要防止短期偶然行为造成偏好漂移，并注意用户行为数据的隐私保护。

综上，本项目的推荐流程形成了“有向状态图建模—0-1 BFS 候选路线生成—多源特征融合—五维体验评价—偏好权重排序—大模型解释”的完整技术链路。各模块职责相互独立：图搜索保证路线可达并控制换乘，评分模块负责综合比较，大模型负责将结构化结论转化为自然语言，从而兼顾推荐效率、个性化和可解释性。
