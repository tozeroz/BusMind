# 七、组织分工与协作计划

> 本项目采用 5 人协作模式，围绕“公交信息查询、ETA、拥挤预测、舒适推荐和大模型助手”推进开发，不重点建设复杂管理员端

## (一) 团队角色总览

| **角色** | **人数** | **核心定位** | **主要产出** |
|---|---:|---|---|
| 项目经理 | 1 | 需求、进度、文档、汇报和答辩统筹 | 项目文档、进度表、PPT、演示脚本 |
| 服务端工程师 A | 1 | 后端业务接口负责人 | 用户、线路、站点、车辆等基础接口 |
| 服务端工程师 B | 1 | 算法接口与 AI 接口负责人 | ETA、拥挤预测、舒适推荐、AI 助手接口 |
| 客户端工程师 | 1 | 前端页面与交互负责人 | 用户端页面、地图展示、图表展示 |
| 数据处理工程师 | 1 | 数据集、模拟数据和预测模型负责人 | CSV 数据、SQL、拥挤预测模型、算法说明 |

## (二) 项目经理职责

项目经理负责整体推进，但不只做管理，需要参与需求设计、文档整理、AI 功能设计和答辩准备

| **模块** | **具体任务** |
|---|---|
| 需求分析 | 明确项目目标、用户角色、核心场景和功能边界 |
| 进度管理 | 制定每日任务表，跟踪完成情况，组织同步会议 |
| 功能统筹 | 协调公交查询、ETA、拥挤预测、舒适推荐、AI 助手之间的边界 |
| 文档汇总 | 汇总项目计划书、需求分析、数据库设计、接口文档和测试记录 |
| AI 设计 | 协助设计 AI 出行助手的 Prompt、输入上下文和输出格式 |
| 答辩准备 | 负责 PPT 主线、答辩稿、演示流程和项目亮点包装 |

交付物：

```text
项目计划书
需求分析文档
项目进度表
中期汇报 PPT
最终答辩 PPT
演示脚本
AI 助手 Prompt 初稿
```

## (三) 服务端工程师 A 职责

服务端工程师 A 负责 FastAPI 后端基础业务接口，保证前端能够稳定获取基础数据

| **模块** | **具体任务** |
|---|---|
| 后端框架 | 搭建 FastAPI 项目结构，配置路由、跨域和接口基础规范 |
| 用户功能 | 实现登录、注册、用户信息和历史查询记录接口 |
| 线路模块 | 实现线路列表、线路详情、站点列表查询接口 |
| 站点模块 | 实现站点信息、站点坐标、站点顺序查询接口 |
| 车辆模块 | 实现车辆状态、当前位置、下一站和运行状态查询接口 |
| 数据库对接 | 对接 MySQL 表结构，完成基础数据查询 |
| 接口文档 | 使用 Apifox / Postman 维护基础业务接口说明 |

交付物：

```text
用户登录注册接口
线路查询接口
站点查询接口
车辆状态接口
数据库查询代码
基础接口文档
```

## (四) 服务端工程师 B 职责

服务端工程师 B 负责算法接口、大模型接口和联调测试，是后端与数据算法模块之间的连接点

| **模块** | **具体任务** |
|---|---|
| ETA 接口 | 调用或实现公交到达起始站点的 ETA 计算逻辑 |
| 拥挤预测接口 | 对接数据处理工程师提供的规则模型或机器学习模型 |
| 舒适推荐接口 | 综合 ETA、拥挤程度、换乘次数和步行距离返回推荐方案 |
| AI 助手接口 | 接收用户自然语言问题，调用大模型生成出行建议 |
| 数据更新接口 | 配合模拟车辆状态和预测结果更新 |
| 接口测试 | 负责预测、推荐、AI 接口的联调和异常测试 |

交付物：

```text
ETA 查询接口
拥挤程度预测接口
舒适推荐接口
AI 出行助手接口
算法接口调用代码
接口测试记录
```

## (五) 客户端工程师职责

客户端工程师负责用户端页面、地图展示和图表展示。由于前端人手较少，页面优先保证核心流程可用

| **模块** | **具体任务** |
|---|---|
| 前端框架 | 搭建 Vue3 项目，配置路由、组件和接口请求 |
| 登录注册页 | 实现用户登录、注册和基础表单校验 |
| 首页 | 展示起终点输入、常用线路和 AI 助手入口 |
| 线路查询页 | 展示线路列表、站点列表、线路详情 |
| 公交详情页 | 展示车辆位置、ETA、拥挤等级和运行状态 |
| 舒适推荐页 | 展示最快、最舒适、少换乘和综合推荐方案 |
| AI 助手页 | 实现聊天式输入和自然语言结果展示 |
| 历史趋势页 | 使用 ECharts 展示线路或站点拥挤趋势 |

交付物：

```text
登录注册页面
首页和线路查询页面
公交详情页面
舒适推荐页面
AI 出行助手页面
历史趋势图表页面
前端接口联调记录
```

## (六) 数据处理工程师职责

数据处理工程师负责项目的数据底座和算法支撑，为后端接口、前端图表和 AI 助手提供结构化数据

| **模块** | **具体任务** |
|---|---|
| 数据结构设计 | 和后端一起确认线路、站点、车辆、客流、预测结果等表字段 |
| 基础数据构建 | 整理或生成线路、站点、车辆、线路站点关系数据 |
| 模拟数据生成 | 生成车辆运行数据、历史客流数据和预测样例数据 |
| ETA 参数支持 | 提供站间耗时、站间距离、时段系数、天气系数和拥挤延迟参数 |
| 拥挤预测 | 实现规则模型，后续可升级为随机森林或 XGBoost |
| 推荐算法支持 | 提供 ETA、拥挤等级、舒适度得分、推荐分数等字段 |
| 图表数据 | 生成线路拥挤趋势、站点客流趋势、拥挤等级分布数据 |
| 算法说明 | 编写特征字段、标签规则、模型流程和评价指标说明 |

交付物：

```text
bus_line.csv
bus_station.csv
line_station.csv
bus_vehicle.csv
bus_realtime.csv
crowd_history.csv
crowd_prediction.csv
数据库初始化 SQL
拥挤预测模型
舒适推荐样例数据
算法说明文档
```

## (八) 功能优先级

为了保证 10 天内可交付，功能按优先级推进

| **优先级** | **功能** | **说明** |
|---|---|---|
| P0 | 线路查询、站点查询、车辆状态、ETA、拥挤等级、舒适推荐 | 必须完成，形成项目核心闭环 |
| P1 | 地图展示、历史趋势、AI 出行助手、推荐理由解释 | 加分功能，增强展示效果 |
| P2 | 收藏路线、历史查询记录、复杂个性化偏好 | 可选功能，时间充足再做 |

核心闭环：

```text
用户输入起点终点
↓
系统查询可选线路
↓
计算公交到站 ETA
↓
预测拥挤程度
↓
输出舒适推荐方案
↓
AI 解释推荐原因
```

## (九) 每日协作机制

建议每天进行一次 15 分钟同步会，每个人只汇报三件事：

```text
今天完成了什么？
遇到了什么问题？
明天准备做什么？
```

项目经理维护问题清单：

| **问题** | **负责人** | **截止时间** | **状态** |
|---|---|---|---|
| 前端无法调用线路接口 | 服务端 A / 客户端 | 待定 | 待处理 |
| 拥挤预测字段与后端接口不一致 | 数据处理 / 服务端 B | 待定 | 待处理 |
| ETA 计算结果不稳定 | 服务端 B / 数据处理 | 待定 | 待处理 |
| AI 输出推荐理由不够准确 | 服务端 B / 项目经理 | 待定 | 待处理 |

## (十) 答辩分工建议

| **答辩内容** | **建议负责人** |
|---|---|
| 项目背景、目标和整体介绍 | 项目经理 |
| 系统架构和接口设计 | 服务端工程师 |
| 数据来源、拥挤预测和推荐算法 | 数据处理工程师 |
| 页面展示和交互流程 | 客户端工程师 |
| AI 助手和推荐解释 | 服务端 B / 项目经理 |
| 演示流程控制和总结 | 项目经理 |

最终对外分工表述：

> 本项目组共 5 人，设项目经理 1 名、服务端工程师 2 名、客户端工程师 1 名、数据处理工程师 1 名。项目经理负责需求分析、进度管理、文档汇总和答辩组织；服务端工程师负责 FastAPI 后端、业务接口、算法接口和 AI 接口；客户端工程师负责用户端页面、地图和图表展示；数据处理工程师负责数据集构建、数据预处理、ETA 参数支持、拥挤预测模型和舒适推荐算法支撑
