# 四、系统架构与接口设计

## 八、系统总体架构

本项目采用前后端分离架构，整体架构如下：

```text
用户浏览器
   ↓
Vue3 前端页面
   ↓ Axios 请求
FastAPI 后端服务
   ↓
业务服务层
   ├── 用户服务
   ├── 公交查询服务
   ├── 实时公交状态服务
   ├── 历史客流与道路热力服务
   ├── 出行体验推荐服务
   └── AI 助手服务
   ↓
数据访问层 SQLAlchemy
   ↓
MySQL 数据库

FastAPI 后端
   ↓
Python 算法模块
   ├── 数据处理
   ├── 候选路线生成
   ├── 实时 ETA / Load 解析
   ├── 历史客流压力计算
   ├── 道路拥堵压力计算
   ├── 多目标推荐评分
   └── 结构化推荐理由生成

FastAPI 后端
   ↓
大模型 API
   ↓
自然语言出行建议
```

核心数据流：

```text
线路和站点查询
        ↓
生成候选公交方案
        ↓
读取或模拟 Bus Arrival，得到 eta_minutes、Load、车辆位置和 Monitored
        ↓
读取历史站点客流和道路速度热力，计算 history_flow_score、congestion_score
        ↓
调用推荐模块，计算 time、comfort、walk、transfer、reliability 分项得分
        ↓
根据用户偏好计算 recommend_score 并排序
        ↓
返回推荐方案，并交给 AI 助手生成解释
```

## 九、技术选型

| 模块 | 技术选型 | 说明 |
|---|---|---|
| 前端框架 | Vue3 | 构建用户交互页面 |
| UI 组件 | Element Plus | 快速开发表单、表格、卡片等页面组件 |
| 地图渲染 | MapLibre GL JS + PMTiles + Protomaps | 前端使用 MapLibre GL JS 渲染地图，底图采用 PMTiles + Protomaps，支持离线或静态文件形式加载矢量瓦片 |
| 地图底图 | PMTiles + Protomaps | 使用 Protomaps 生成或提供地图样式与矢量瓦片，PMTiles 作为单文件瓦片数据源，便于本地部署和演示 |
| 站点展示 | GeoJSON Point + Layer | 公交站点使用 GeoJSON Point 表达，通过 Symbol Layer 或 Circle Layer 展示站点位置，支持点击站点查看详细信息 |
| 线路展示 | GeoJSON LineString + Line Layer | 公交线路使用 GeoJSON LineString 表达，通过 Line Layer 绘制线路走向、推荐路线和换乘路径 |
| 客流与路况可视化 | 颜色渐变 / 热力图 / 点大小 | 根据客流压力、客载状态或拥堵等级展示不同颜色和热力区域 |
| 图表展示 | ECharts | 展示客流趋势、推荐分项得分等图表 |
| 后端框架 | FastAPI | 提供 RESTful API，自动生成接口文档 |
| 数据库 | MySQL | 存储用户、线路、站点、车辆、实时采样、历史客流、道路热力和推荐结果 |
| ORM | SQLAlchemy | 操作数据库 |
| 数据处理 | Python + Pandas + NumPy | 处理 LTA 风格数据和仿真数据 |
| 推荐算法 | Python | 实现候选路线生成、多源特征融合、多目标评分和 Top-K 排序 |
| 可选机器学习 | Scikit-learn / XGBoost | 第二阶段用于站点客流、拥堵趋势或用户偏好学习 |
| 大模型接口 | DeepSeek / 通义千问 / OpenAI API | 实现 AI 出行助手 |
| 接口通信 | Axios + RESTful API | 前后端数据交互 |
| 接口测试 | Apifox / Postman | 接口调试和文档管理 |
| 项目管理 | Git / GitHub / Gitee | 代码版本管理 |
| 部署方式 | 本地部署 / 云服务器 + Nginx | 项目演示和答辩部署 |

## 十一、接口设计初稿

### 11.1 用户功能接口

| `/api/user/register` | POST | 用户注册 |

| `/api/user/favorite` | POST | 收藏路线 |
| `/api/user/history` | GET | 查询历史记录 |
| `/api/user/preference` | POST | 保存用户出行偏好 |

------

### 11.2 公交信息查询接口

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/bus/lines` | GET | 查询所有公交线路 |
| `/api/bus/lines/{line_id}` | GET | 查询线路详情 |
| `/api/bus/lines/{line_id}/stations` | GET | 查询线路站点 |
| `/api/bus/vehicles` | GET | 查询车辆实时状态 |
| `/api/bus/routes` | POST | 根据起终点查询候选公交方案 |
| `/api/bus/stops/nearby` | GET | 根据经纬度查询附近公交站点 |

------

### 11.3 实时状态与推荐接口

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/realtime/arrival` | GET | 获取站点线路的实时 ETA、Load、车辆位置和 Monitored |
| `/api/realtime/status` | POST | 批量获取候选路线的实时公交状态 |
| `/api/recommend/routes` | POST | 获取多目标出行体验推荐方案 |
| `/api/recommend/explain` | POST | 获取结构化推荐理由 |
| `/api/history/passenger-flow` | GET | 查询历史站点客流趋势 |
| `/api/history/traffic-speed` | GET | 查询道路速度热力或拥堵等级 |

推荐接口核心返回字段：

| 字段 | 说明 |
|---|---|
| `candidate_routes` | 候选路线列表 |
| `recommended_route` | 当前偏好下综合推荐路线 |
| `eta_minutes` | 实时预计到站时间 |
| `load` / `load_score` | 实时客载状态和舒适度分数 |
| `history_flow_score` | 历史客流压力得分 |
| `congestion_score` | 道路拥堵得分 |
| `time_score` | 时间效率得分 |
| `comfort_score` | 舒适度得分 |
| `walk_score` | 步行负担得分 |
| `transfer_score` | 换乘复杂度得分 |
| `reliability_score` | 数据可靠性得分 |
| `recommend_score` | 综合推荐分 |
| `reason` | 结构化推荐原因 |

## 十二、数据库设计初稿

建议至少设计以下数据表：

| 数据表 | 作用 |
|---|---|
| `user` | 存储用户信息 |
| `bus_line` | 存储公交线路信息 |
| `bus_station` | 存储站点信息 |
| `line_station` | 存储线路与站点的关联关系 |
| `bus_vehicle` | 存储车辆基础信息 |
| `bus_realtime` | 存储车辆实时位置、ETA、Load、Monitored 等采样数据 |
| `lta_bus_arrival` | 存储或模拟 `EstimatedArrival`、车辆位置、`Load` 等 LTA 到站数据 |
| `passenger_volume_stop` | 存储站点小时级 tap in / tap out 客流 |
| `passenger_volume_od` | 存储 OD 站点对小时级出行量，作为第二阶段扩展 |
| `traffic_speed_band` | 存储道路速度热力、道路名称、路段坐标和更新时间 |
| `route_recommendation` | 存储候选路线、分项得分、综合推荐分和推荐理由 |
| `user_preference` | 存储用户出行偏好权重 |
| `user_favorite` | 存储用户收藏路线 |
| `query_history` | 存储用户查询记录 |

第一阶段数据库重点保证以下链路可用：

```text
线路站点基础数据
        ↓
Bus Arrival 实时或仿真数据
        ↓
历史站点客流和道路热力数据
        ↓
候选路线生成
        ↓
多目标推荐评分
        ↓
出行体验推荐结果
```
