# 一、项目概述

## 一、项目简介

本项目拟设计并实现一套面向公交乘客的智能出行体验推荐系统，项目名称为：

**《智行公交：基于多源数据融合与大模型助手的公交出行体验推荐系统》**

系统围绕用户“出门乘坐公交”这一具体场景，提供公交线路查询、站点信息查看、实时 ETA 展示、实时客载状态展示、历史客流与道路热力分析、个性化出行体验推荐、地图可视化展示和大模型出行助手等功能。

与普通公交查询系统相比，本项目不仅关注“能不能到达”和“多久到达”，还进一步关注“乘坐体验是否更好”。系统通过融合公交线路、站点、车辆位置、实时到站、实时客载、历史客流、道路速度热力和用户偏好等数据，在多条可达公交方案中综合评估时间效率、舒适度、步行负担、换乘复杂度和数据可靠性，为用户推荐更适合的公交出行方案。

本项目采用前后端分离架构，前端使用 Vue3 构建用户交互界面，后端使用 FastAPI 提供接口服务，数据库采用 MySQL 存储基础数据、实时采样数据和推荐结果，算法模块使用 Python 实现候选路线生成、多源特征融合、多目标出行体验评分和个性化推荐，大模型模块用于生成自然语言出行建议和路线推荐解释。

------

## 二、项目背景

公交是城市和校园出行中使用频率较高的公共交通方式，具有价格低、覆盖范围广、绿色环保等优点。但在实际出行过程中，乘客仍然面临一些常见问题。

第一，用户往往不知道公交车什么时候到站。虽然部分地图软件可以提供路线规划，但对于具体车辆到站时间、等待时间和车辆运行状态的展示并不一定准确或充分。

第二，用户不知道即将到站的公交车内空间是否宽松。很多情况下，用户到站后才发现车辆已经非常拥挤，乘坐体验较差，甚至可能无法顺利上车。

第三，传统公交查询系统通常以“最快到达”为主要目标，较少考虑用户对出行体验的需求。例如，有些用户愿意多等几分钟，选择一辆车内更宽松的车；有些用户则更在意少换乘、少步行或数据更可靠。

第四，公交出行体验不仅受实时到站和客载影响，还会受到站点历史客流、途经道路拥堵和发车频率等因素影响。普通系统较少把这些数据合并到同一个可解释推荐框架中。

第五，普通系统的结果展示较为机械，用户需要自己理解路线、拥挤情况和推荐原因。对于不熟悉线路的用户来说，缺少一个能够用自然语言解释出行方案的智能助手。

因此，本项目希望在传统公交查询功能基础上，引入实时公交状态、历史客流、道路热力、多目标出行体验评价和大模型助手，为用户提供更加智能、个性化、体验友好的公交出行服务。

------

## 三、项目目标

本项目的总体目标是完成一套以公交乘客为主要服务对象的智能出行系统，实现“查公交、看 ETA、知客载、比体验、选方案、问助手”的完整功能闭环。

具体目标如下：

| 目标类别 | 具体目标 |
|---|---|
| 基础业务目标 | 实现用户登录注册、线路查询、站点查询、车辆状态查询、ETA 到站时间展示 |
| 数据处理目标 | 构建公交线路、站点、车辆、实时到站、实时客载、历史客流、道路热力等数据集 |
| 算法目标 | 融合实时公交状态、历史客流和道路热力，计算多目标出行体验评分 |
| 推荐目标 | 综合 ETA、客载状态、客流压力、道路拥堵、步行距离、换乘次数和用户偏好，为用户推荐体验更好的公交方案 |
| 可视化目标 | 使用地图和图表展示公交线路、站点、车辆位置、客载状态、道路热力和历史趋势 |
| 大模型目标 | 接入大模型，实现自然语言问答、路线推荐解释和个性化出行建议 |
| 工程目标 | 完成 Vue3 前端、FastAPI 后端、MySQL 数据库、Python 算法模块和大模型接口联调 |

本项目最终希望实现的效果是：用户输入起点和终点后，系统可以返回多种公交方案，并标注每种方案的实时 ETA、实时客载状态、历史客流压力、道路拥堵程度、换乘次数、步行距离、出行体验评分和推荐理由。用户也可以直接向 AI 助手提问，例如“现在去教学楼怎么坐比较舒服？”系统结合结构化推荐结果给出自然语言建议。

## 四、用户角色和使用场景

### 4.1 用户角色

本项目主要面向普通公交乘客，系统角色设计保持简洁。

| 用户角色 | 角色说明 | 主要功能 |
|---|---|---|
| 普通用户 | 使用系统进行公交出行查询的乘客 | 查询线路、查看 ETA、查看客载状态、获取出行体验推荐、使用 AI 助手 |
| 系统维护人员 | 负责维护基础数据和查看运行数据的人员 | 管理线路、站点、实时采样和历史客流数据，查看推荐结果和历史趋势 |

本项目重点聚焦普通用户侧，不做复杂的管理员端。系统维护人员相关功能主要用于数据管理和演示展示，不作为项目核心。

------

### 4.2 使用场景

#### 场景一：用户想知道怎么坐公交

用户输入起点和终点，例如“宿舍区到教学楼”，系统返回可选公交方案，包括可乘坐线路、经过站点、预计到站时间和预计到达时间。

#### 场景二：用户想知道下一班车什么时候来

用户选择某个站点或某条线路后，系统展示下一班车的实时 ETA、车辆当前位置、当前运行状态和实时客载状态。

#### 场景三：用户想避开拥挤车辆

早晚高峰时，系统根据实时 Load、历史站点客流、线路客流压力和途经道路拥堵程度，判断不同候选方案的乘坐体验。用户可以选择等待时间稍长但车内更宽松、整体体验更好的方案。

#### 场景四：用户希望系统直接推荐最合适方案

系统综合 ETA、客载状态、历史客流、道路热力、步行距离、换乘次数和用户偏好，给出“最快方案”“车内更宽松方案”“最少换乘方案”“步行更少方案”和“综合推荐方案”。

#### 场景五：用户通过 AI 助手自然语言提问

用户输入“我不想太挤，现在去图书馆怎么坐？”AI 助手结合系统推荐结果，生成自然语言建议，例如：

> 建议选择校园 2 号线。虽然预计到站时间比校园 1 号线晚约 4 分钟，但当前客载状态更宽松，历史客流压力较低，且途经道路拥堵程度较轻。如果你不赶时间，推荐选择该方案。
