# 大模型助手

## 1. 当前接口模式

当前 `POST /api/v1/ai/travel` 支持三种显式模式。模式不是由 AI 自动判断，而是调用方在请求体中设置 `mode`。

| 模式 | 必需参数 | 是否主动查询路线 | 用途 |
| --- | --- | --- | --- |
| `qa` | `question` | 否 | 基于调用方提供的上下文回答问题 |
| `suggest` | 建议提供起终点站 ID，或有效 `context` | 有起终点且上下文无路线时会查询 | 推荐并比较候选路线；缺参时返回结构化追问 |
| `explain` | 包含至少一条有效路线的 `context.items` | 否 | 解释已有推荐路线 |

请求校验位于 `backend/app/schemas/ai_travel.py`，执行分支位于 `backend/app/services/ai_service/service.py`。

## 2. `qa` 模式

```json
{
  "mode": "qa",
  "question": "下一班车多久到？"
}
```

`qa` 不会主动调用路线推荐、ETA、客载预测或 MySQL 路线查询。没有 `context.items` 时，模型得到的路线上下文是空数组，只能说明信息不足。

因此，不能把“下一班车多久到”“现在怎么去教学楼”等实时问题无条件发送成空上下文 `qa`。

## 3. `suggest` 模式

```json
{
  "mode": "suggest",
  "question": "帮我推荐一条比较舒适的路线",
  "start_station_id": 1,
  "end_station_id": 12,
  "preference": "balanced"
}
```

没有可复用路线上下文时，后端调用链为：

```text
RecommendationService
→ MySQLTransitGateway
→ 候选路线
→ ETA 预测
→ 客载预测
→ 步行与换乘数据
→ 体验分与排序
→ DeepSeek 解释
```

如果 `context.items` 已包含可解析的路线，服务会复用这些路线，不重复查询推荐服务。

## 4. `explain` 模式

当前可靠用法是同时传入 `route_id` 和包含完整路线的 `context`：

```json
{
  "mode": "explain",
  "question": "为什么推荐这条路线？",
  "route_id": "route-001",
  "context": { "items": ["recommend-routes 返回的完整路线对象"] }
}
```

P0 后端已收紧该契约：只传 `route_id` 或传空 `context.items` 会返回 HTTP 422。`route_id` 在上下文路线中不存在时，接口返回 `status: "needs_clarification"`，不会让模型错误解释其他路线。

## 5. 当前两个前端入口的实际状态

### 首页内嵌 AI

`frontend/src/modules/home/components/HomePage.vue` 已经包含：

- 地图右下角 AI 浮动按钮；
- AI 对话卡片；
- 起终点站解析和路线推荐；
- 推荐路线摘要；
- “地图查看”动作。

首页在已经解析出起终点时会发送 `suggest`，否则发送 `qa`，因此它比独立 AI 页面更接近可用链路。

首页针对“舒适/不挤”传入的旧值 `preference: "low_load"` 仍可使用。P0 后端会在 AI 请求边界将它规范化为 `comfort`；新代码应直接使用 `comfort`。

### 独立 AI 页面

`frontend/src/modules/ai-assistant/components/AiAssistantPage.vue` 固定发送 `qa`，而且没有传起终点或路线上下文。

对应的 `AiAssistantView.vue` 已存在，但当前 `frontend/src/router/index.js` 没有注册 AI 页面路由。因此该页面不是当前正式入口，不建议再围绕它建设第二套交互。

## 6. 当前限制

1. 后端没有意图识别路由器，调用方必须决定 `mode`。
2. 会话仅保存在组件内存，后端不知道上一轮推荐了什么。
3. P0 阶段的 `explain` 仍不能只依靠 `route_id` 恢复路线，必须携带有效上下文；推荐快照将在 P1 实现。
4. 首页与独立 AI 页存在两套不一致的交互实现。
5. `low_load` 仅作为兼容别名保留，新请求统一使用 `comfort`。
6. DeepSeek 只负责解释后端提供的数据，不能替代实时路线、ETA 和客载服务。
