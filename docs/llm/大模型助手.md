# 大模型助手

## 1. 当前接口模式

当前 `POST /api/v1/ai/travel` 支持三种模式。P1 起 `mode` 可以省略，由后端规则编排器识别；调用方显式传入的 `mode` 仍然优先，用于兼容 P0 客户端。

| 模式 | 必需参数 | 是否主动查询路线 | 用途 |
| --- | --- | --- | --- |
| `qa` | `question` | 否 | 基于调用方提供的上下文回答问题 |
| `suggest` | 建议提供起终点站 ID，或有效 `context` | 有起终点且上下文无路线时会查询 | 推荐并比较候选路线；缺参时返回结构化追问 |
| `explain` | 推荐快照，或有效 `context.items` | 否 | 解释已有推荐路线 |

请求校验位于 `backend/app/schemas/ai_travel.py`，执行分支位于 `backend/app/services/ai_service/service.py`。

## 2. `qa` 模式

```json
{
  "mode": "qa",
  "question": "下一班车多久到？"
}
```

`qa` 不会主动调用路线推荐、ETA、客载预测或 MySQL 路线查询。没有 `context.items` 时，模型得到的路线上下文是空数组，只能说明信息不足。

因此，不能把“下一班车多久到”“现在怎么去教学楼”等实时问题无条件发送成空上下文 `qa`。

## 3. `suggest` 模式

```json
{
  "mode": "suggest",
  "question": "帮我推荐一条比较舒适的路线",
  "start_station_id": 1,
  "end_station_id": 12,
  "preference": "balanced"
}
```

没有可复用路线上下文时，后端调用链为：

```text
RecommendationService
→ MySQLTransitGateway
→ 候选路线
→ ETA 预测
→ 客载预测
→ 步行与换乘数据
→ 体验分与排序
→ DeepSeek 解释
```

如果 `context.items` 已包含可解析的路线，服务会复用这些路线，不重复查询推荐服务。

## 4. `explain` 模式

首次无会话时可以同时传入 `route_id` 和包含完整路线的 `context`：

```json
{
  "mode": "explain",
  "question": "为什么推荐这条路线？",
  "route_id": "route-001",
  "context": { "items": ["recommend-routes 返回的完整路线对象"] }
}
```

P1 中推荐成功后，后续只需传首次响应的 `conversation_id` 和问题。后端会从推荐快照恢复路线；没有快照也没有有效上下文时返回 `status: "needs_clarification"`。`route_id` 在快照或上下文中不存在时同样结构化追问，不会让模型错误解释其他路线。

## 5. 当前两个前端入口的实际状态

### 首页内嵌 AI

`frontend/src/modules/home/components/HomePage.vue` 已经包含：

- 地图右下角 AI 浮动按钮；
- AI 对话卡片；
- 起终点站解析和路线推荐；
- 推荐路线摘要；
- “地图查看”动作。

首页在已经解析出起终点时会发送 `suggest`，否则发送 `qa`，因此它比独立 AI 页面更接近可用链路。

首页针对“舒适/不挤”传入的旧值 `preference: "low_load"` 仍可使用。P0 后端会在 AI 请求边界将它规范化为 `comfort`；新代码应直接使用 `comfort`。

### 独立 AI 页面

`frontend/src/modules/ai-assistant/components/AiAssistantPage.vue` 固定发送 `qa`，而且没有传起终点或路线上下文。

对应的 `AiAssistantView.vue` 已存在，但当前 `frontend/src/router/index.js` 没有注册 AI 页面路由。因此该页面不是当前正式入口，不建议再围绕它建设第二套交互。

## 6. 当前限制

1. 自然语言起终点目前可靠提取站点编号，不猜测未校验的站名。
2. 会话快照是单进程内存 LRU，服务重启或切换进程后不会保留。
3. 首页与独立 AI 页存在两套不一致的交互实现，尚待前端联调收敛。
4. `low_load` 仅作为兼容别名保留，新请求统一使用 `comfort`。
5. DeepSeek 只负责解释后端提供的数据，不能替代实时路线、ETA 和客载服务。
