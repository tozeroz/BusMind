# 大模型助手升级计划

## 1. 总体方向

```text
意图与参数识别
→ 结构化业务工具
→ 静态知识 RAG（后续）
→ 统一证据上下文
→ 大模型解释
→ 事实校验与降级
```

实时路线、ETA、客载和天气不进入静态向量知识库。RAG 只处理规则、公告、线路说明和 FAQ 等文档性知识。

## 2. 四阶段实施

### P0：接口与已有能力打通

目标：现有推荐、ETA、客载能力稳定进入 AI 回答链路。

状态：**后端部分已完成，等待前端联调。**

后端范围：

- 固化 `qa/suggest/explain` 请求与响应契约；
- 统一前后端偏好枚举；
- 修正 `explain` 的参数语义；
- 明确缺少起终点时的结构化返回；
- 补齐 API、服务和 fallback 测试；
- 保持现有接口向后兼容。

前端范围由前端负责人完成，参见《前端接入与组件交互说明》；本阶段不由当前后端任务修改 `frontend/`。

验收：

- `suggest` 必然使用真实推荐服务或明确的已有路线；
- 空上下文实时问题不会生成虚构答案；
- `explain` 不再出现参数校验通过但无路线可解释的状态；
- DeepSeek 失败仍能返回业务结果。

### P1：后端自动编排与会话

目标：前端不再承担自然语言意图判断。

- 增加规则优先的意图识别器；
- 提取起点、终点、偏好和目标路线等槽位；
- 参数不足时返回结构化追问；
- 增加 `conversation_id`；
- 保存最近一次推荐快照；
- 支持通过推荐快照和 `route_id` 解释路线；
- 保留显式 `mode` 作为兼容入口。

验收：连续完成“推荐路线 → 为什么 → 换一条”的多轮流程，前端不需要重复回传完整路线。

### P2：静态知识 RAG

目标：补充公交规则、线路说明、公告和 FAQ。

- 文档清洗、语义切块和元数据；
- BM25 与向量混合召回；
- 按线路、站点、文档类型和有效期过滤；
- reranker 重排；
- 返回来源、有效期和更新时间；
- 将检索结果写入统一证据上下文。

验收：知识类问题有可追踪来源，过期公告不会作为当前事实回答。

### P3：质量、观测与产品化

- SSE 流式输出；
- 工具耗时、模型耗时和 fallback 原因监控；
- 鉴权、频控和上下文大小限制；
- 固定评测集；
- 评估意图准确率、工具选择率、事实一致性和响应时延。

## 3. 下一步计划与停止边界

下一次进入代码实施时，建议先完成 **P0 后端部分**，暂不直接跨入 RAG，也不修改前端页面。

计划修改范围：

```text
backend/app/schemas/ai_travel.py
backend/app/services/ai_service/
backend/app/api/v1/ai/
backend/app/api/v1/dependencies.py（仅依赖注入需要时）
llm/prompts/travel_assistant.py
backend/tests/test_ai_api.py
backend/tests/test_ai_service.py
backend/tests/test_main_integration.py
```

计划做到以下状态后停止并交付前端联调：

1. 现有三种模式的契约明确且测试通过；
2. 缺少数据时返回可供前端展示的明确追问或提示；
3. `explain` 不再接受无法执行的空路线请求；
4. 偏好值与推荐服务保持一致；
5. Prompt 使用统一、受控的结构化证据；
6. DeepSeek 成功和 fallback 两条路径均有测试；
7. 输出一份最终接口示例供前端负责人接入。

当前已经达到上述 P0 后端停止边界。与前端负责人按《P0 后端接口联调说明》完成联调后，再决定是否继续 P1。不会在一次改动中同时加入会话存储、向量数据库和流式输出，以免扩大联调面。
