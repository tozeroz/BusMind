# P0 后端接口联调说明

## 1. 变更范围

接口地址保持不变：

```text
POST /api/v1/ai/travel
```

P0 没有修改任何前端文件。现有前端读取的 `answer`、`used_tools`、`related_routes`、`reminders` 和 `fallback` 均保留。

响应新增两个向后兼容字段：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `status` | string | `completed`、`needs_clarification` 或 `degraded` |
| `missing_fields` | string[] | 需要前端或用户补充的字段；正常完成时为空数组 |

## 2. 状态语义

| status | fallback | 场景 |
| --- | --- | --- |
| `completed` | `false` | 已获得可信路线证据且 DeepSeek 正常返回 |
| `needs_clarification` | `false` | 缺少起终点、路线上下文或路线 ID 不匹配；这是正常业务追问 |
| `degraded` | `true` | 路线证据存在，但 DeepSeek 未配置、超时或调用失败，已使用本地模板 |

前端不应把 `needs_clarification` 当作系统故障。

## 3. 普通问答

请求：

```json
{
  "mode": "qa",
  "question": "下一班车多久到？"
}
```

当前没有路线证据时返回 HTTP 200：

```json
{
  "data": {
    "answer": "当前没有可核验的路线、ETA 或客载数据。请先提供起点和终点并查询路线。",
    "mode": "qa",
    "status": "needs_clarification",
    "missing_fields": ["context.items"],
    "used_tools": [],
    "related_routes": [],
    "reminders": [],
    "fallback": false
  }
}
```

该分支不会调用 DeepSeek，避免空证据生成实时信息。

## 4. 路线推荐

完整请求：

```json
{
  "mode": "suggest",
  "question": "哪条路线比较舒适？",
  "start_station_id": 1,
  "end_station_id": 12,
  "preference": "comfort"
}
```

兼容旧值：

```json
{ "preference": "low_load" }
```

`low_load` 会被规范化为 `comfort`。建议新请求直接使用 `comfort`。

缺少起终点时仍返回 HTTP 200，方便界面展示追问：

```json
{
  "data": {
    "answer": "请提供完整的起点和终点，我才能查询候选路线、预计到站时间和客载情况。",
    "mode": "suggest",
    "status": "needs_clarification",
    "missing_fields": ["start_station_id", "end_station_id"],
    "used_tools": [],
    "related_routes": [],
    "reminders": [],
    "fallback": false
  }
}
```

只缺少一个站点时，`missing_fields` 只返回对应字段。

## 5. 路线解释

P0 阶段必须把 `/recommend-routes` 返回的完整路线对象放进 `context.items`：

```js
const recommendation = await recommendRoutes(options)
const selectedRoute = recommendation.data.items[0]

await askAiTravel({
  mode: 'explain',
  question: '为什么推荐这条路线？',
  route_id: selectedRoute.route_id,
  context: {
    items: recommendation.data.items
  }
})
```

P0 初始契约中，以下请求返回 HTTP 422：

- 只传 `route_id`，没有路线上下文；
- `context.items` 为空；
- `context.items` 中没有任何可解析的完整路线对象。

该校验已被 P1 会话快照契约替代：当前版本对这些请求返回 HTTP 200 和 `needs_clarification`，并允许通过有效 `conversation_id` 恢复路线。新联调以《P1 后端自动编排与会话说明》为准。

如果上下文有效但 `route_id` 不属于其中任何路线，返回 HTTP 200：

```json
{
  "data": {
    "status": "needs_clarification",
    "missing_fields": ["route_id"],
    "answer": "当前推荐结果中没有找到指定路线，请重新选择一条候选路线后再询问。"
  }
}
```

## 6. 模型证据契约

后端不再向模型只传裸路线数组，而是传递版本化证据：

```json
{
  "contract_version": "1.0",
  "mode": "suggest",
  "query": {
    "preference": "comfort",
    "start_station_id": 1,
    "end_station_id": 12,
    "route_id": null
  },
  "evidence": {
    "source": "recommendation_service",
    "routes": []
  },
  "missing_information": [],
  "constraints": {
    "route_facts_are_authoritative": true,
    "model_must_not_recalculate_metrics": true,
    "model_must_not_invent_realtime_data": true
  }
}
```

该结构只用于后端到模型的内部调用，前端无须构造。

## 7. 前端最小适配建议

现有页面不适配新增字段也能继续读取答案。前端负责人后续只需：

1. `status === "needs_clarification"` 时把 `answer` 作为正常追问展示；
2. `missing_fields` 可用于聚焦起点、终点或路线选择控件；
3. 仅在 `fallback === true` 时显示 AI 服务降级提示；
4. `explain` 必须回传推荐接口的完整原始路线，不能只传前端归一化后的卡片字段。
