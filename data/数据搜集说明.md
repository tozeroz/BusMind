> 本项目的数据采集不追求真实公交企业级数据，推荐采用“真实/半真实线路站点数据 + 规则模拟客流与车辆运行数据”的方案，优先保证系统能演示、字段能入库、模型能训练、图表能展示

# 一、文档目标

本文档面向“智行公交：基于拥挤预测与大模型助手的公交舒适出行系统”的数据处理工作，说明项目需要搜集哪些数据、每类数据服务于哪个功能模块、推荐如何获取，以及哪些数据可以直接模拟生成

数据处理工程师的核心职责是：

- 为数据库提供初始化数据
- 为后端接口提供稳定字段
- 为拥挤预测模型提供训练数据
- 为前端图表和管理端大屏提供展示数据
- 为 AI 出行助手和 AI 调度建议提供结构化上下文

# 二、总体数据策略

## (一) 推荐方案

本项目建议采用以下数据策略：

| **数据类型** | **推荐方式** | **原因** |
|---|---|---|
| 线路、站点、站点顺序 | 真实参考或手工设计 | 这些数据是系统基础，越像真实场景越容易答辩解释 |
| 站点经纬度 | 地图拾取或围绕校园中心点手工配置 | 用于地图展示、距离估算和路线可视化 |
| 站间距离、站间耗时 | 地图估算或人工设定 | 用于 ETA 预计到站时间 |
| 车辆数量、容量、所属线路 | 人工设定 | 真实数据难获取，模拟即可 |
| 实时车辆状态 | 程序模拟 | 用于车辆位置、拥挤状态和动态展示 |
| 历史客流 | 程序模拟 | 真实客流难获取，且涉及运营数据，模拟更可控 |
| 天气、节假日、特殊事件 | 简化模拟或手工配置 | 用于让拥挤预测更合理 |
| 拥挤预测结果 | 模型或规则生成 | 用于乘客端展示、推荐和管理端预警 |

## (二) 不建议投入大量时间寻找的数据

以下数据很难在短时间内可靠获取，不建议作为第一阶段重点：

- 真实公交 GPS 轨迹
- 真实车辆实时载客人数
- 真实刷卡记录、扫码记录、乘客个人出行记录
- 真实公交企业调度数据
- 需要登录、授权或爬虫抓取的大规模平台数据

这些数据可以在答辩中说明为“由于真实公交运营数据不公开，项目采用仿真数据进行建模和系统验证”

# 三、数据分层

项目数据建议分成 4 层：

| **层级** | **含义** | **主要文件** | **用途** |
|---|---|---|---|
| 基础静态数据 | 线路、站点、车辆等不频繁变化的数据 | `bus_line.csv`、`bus_station.csv`、`line_station.csv`、`bus_vehicle.csv` | 后端基础查询、前端线路展示 |
| 动态运行数据 | 车辆当前位置、速度、载客量、状态等 | `bus_realtime.csv` | 车辆状态展示、ETA 计算、拥挤展示 |
| 历史训练数据 | 按日期、小时、线路、站点统计的客流数据 | `crowd_history.csv` | 拥挤预测模型训练、客流趋势图 |
| 预测与分析数据 | 模型输出、预警、图表聚合结果 | `crowd_prediction.csv`、`station_heat.csv`、`dashboard_summary.csv` | 舒适推荐、管理端大屏、AI 调度建议 |

# 四、需要搜集或构建的数据明细

## (一) 公交线路数据 `bus_line`

### 1. 数据作用

公交线路数据是整个系统的主数据，用于：

- 乘客端线路查询
- 管理端线路管理
- 车辆绑定线路
- 站点关联
- 推荐和预测时标识线路

### 2. 推荐字段

| **字段名** | **含义** | **示例** | **采集方式** |
|---|---|---|---|
| `line_id` | 线路 ID | `L001` | 自定义编码 |
| `line_name` | 线路名称 | `校园 1 号线` | 手工设计或参考校园公交 |
| `start_station_id` | 起点站 ID | `S001` | 从站点表选择 |
| `end_station_id` | 终点站 ID | `S007` | 从站点表选择 |
| `first_time` | 首班时间 | `06:30` | 手工设定 |
| `last_time` | 末班时间 | `22:30` | 手工设定 |
| `interval_minutes` | 发车间隔 | `8` | 手工设定 |
| `status` | 运行状态 | `normal` | 手工设定 |

### 3. 推荐搜集方式

优先方案：

- 选择校园、园区或大学城公交场景
- 手工设计 3 条线路
- 每条线路 6 到 10 个站点
- 线路之间允许复用部分站点，便于展示换乘和热门站点

可参考来源：

- 学校官网或校园地图中的校车线路
- 高德地图、百度地图中的真实公交线路名称和站点结构
- 本校或附近园区的真实道路和建筑分布

第一版建议数量：

```text
3 条线路
每条线路 6-10 个站点
总站点数 20-30 个
```

## (二) 公交站点数据 `bus_station`

### 1. 数据作用

站点数据用于：

- 前端展示站点列表
- 地图定位
- 客流统计
- 站点热度计算
- 拥挤预测特征
- 管理端热门站点图表

### 2. 推荐字段

| **字段名** | **含义** | **示例** | **采集方式** |
|---|---|---|---|
| `station_id` | 站点 ID | `S001` | 自定义编码 |
| `station_name` | 站点名称 | `北门站` | 手工设计或参考地图 |
| `longitude` | 经度 | `113.923456` | 地图拾取或模拟 |
| `latitude` | 纬度 | `22.543210` | 地图拾取或模拟 |
| `station_type` | 站点类型 | `normal`、`hot` | 手工标注 |
| `area_type` | 所属区域类型 | `dormitory`、`teaching`、`canteen` | 手工标注 |
| `description` | 站点说明 | `靠近学生宿舍区` | 手工填写 |

### 3. 推荐搜集方式

推荐用地图拾取站点坐标：

```text
1. 先确定项目场景，例如校园或园区
2. 打开高德地图或百度地图
3. 搜索学校或园区名称
4. 选择北门、宿舍、食堂、教学楼、图书馆、体育馆、地铁站等关键位置
5. 使用地图坐标拾取工具记录经纬度
6. 将坐标整理到 `bus_station.csv`
```

如果时间紧，可以采用半模拟坐标：

```text
1. 选一个中心点经纬度
2. 围绕中心点上下左右微调 0.001-0.01
3. 保证线路上的站点坐标大致连续
4. 答辩时说明为校园仿真站点坐标
```

### 4. 推荐站点类型

建议站点覆盖以下场景，方便后续生成有规律的客流：

| **站点类型** | **示例站点** | **客流规律** |
|---|---|---|
| 宿舍区 | 学生公寓、宿舍区 | 早高峰上车多，晚高峰下车多 |
| 教学区 | 教学楼、实验楼 | 早高峰下车多，晚高峰上车多 |
| 餐饮区 | 食堂、商业街 | 午高峰客流明显 |
| 学习区 | 图书馆、创新楼 | 晚间客流较明显 |
| 出入口 | 北门、南门、地铁站 | 通勤客流集中 |
| 运动生活区 | 体育馆、医务室 | 平峰客流较分散 |

## (三) 线路站点关联数据 `line_station`

### 1. 数据作用

线路站点关联数据用于描述每条线路经过哪些站点，以及站点顺序，用于：

- 查询线路详情
- 计算车辆下一站
- 计算 ETA
- 模拟车辆在线路上移动
- 判断起点终点之间是否可达

### 2. 推荐字段

| **字段名** | **含义** | **示例** | **采集方式** |
|---|---|---|---|
| `id` | 关联记录 ID | `LS001` | 自定义编码 |
| `line_id` | 线路 ID | `L001` | 来自线路表 |
| `station_id` | 站点 ID | `S001` | 来自站点表 |
| `station_order` | 站点顺序 | `1` | 手工设定 |
| `distance_to_next` | 到下一站距离，单位米 | `650` | 地图估算或手工设定 |
| `avg_time_to_next` | 到下一站平均耗时，单位分钟 | `3` | 手工设定 |
| `direction` | 方向 | `up`、`down` | 第一版可只做 `up` |

### 3. 推荐搜集方式

推荐手工整理：

```text
1. 为每条线路列出站点顺序
2. 根据地图估算相邻站点距离
3. 按距离设置平均行驶时间
4. 将相邻站点耗时控制在 2-6 分钟
```

站间耗时可按这个规则估算：

| **距离范围** | **平均耗时建议** |
|---|---|
| 300-500 米 | 2 分钟 |
| 500-800 米 | 3 分钟 |
| 800-1200 米 | 4 分钟 |
| 1200-1800 米 | 5-6 分钟 |

第一阶段不需要追求真实路径距离，只要站点顺序、距离、时间逻辑一致即可

### 1. 数据作用

车辆数据用于：

- 查询某条线路有哪些车
- 绑定实时车辆状态

### 2. 推荐字段

| **字段名** | **含义** | **示例** | **采集方式** |

大型公交容量设为 80 人

建议第一版：

```text
L001：V001、V002、V003
L002：V004、V005
L003：V006、V007
```

## (五) 实时车辆状态数据 `bus_realtime`

### 1. 数据作用

实时车辆状态用于：

- 乘客端展示车辆当前站点、下一站、行驶进度
- 计算 ETA
- 展示当前拥挤等级
- 管理端车辆监控
- 生成推荐车次

### 2. 推荐字段

| **字段名** | **含义** | **示例** | **采集方式** |
|---|---|---|---|
| `id` | 记录 ID | `RT001` | 自定义或程序生成 |
| `vehicle_id` | 车辆 ID | `V001` | 来自车辆表 |
| `line_id` | 线路 ID | `L001` | 来自车辆表 |
| `current_station_id` | 当前站点 ID | `S003` | 程序模拟 |
| `next_station_id` | 下一站 ID | `S004` | 根据站点顺序计算 |
| `progress` | 两站之间进度 | `0.45` | 程序模拟 |
| `speed` | 当前速度，km/h | `24` | 程序模拟 |
| `onboard_count` | 当前车上人数 | `42` | 程序模拟 |
| `capacity` | 车辆容量 | `60` | 来自车辆表 |
| `crowd_rate` | 拥挤率 | `0.70` | 计算生成 |
| `crowd_level` | 拥挤等级 | `medium` | 规则或模型生成 |
| `status` | 运行状态 | `normal`、`delay`、`crowded` | 规则生成 |
| `update_time` | 更新时间 | `2026-07-06 08:10:00` | 程序生成 |

### 3. 推荐搜集方式

实时车辆状态不建议手工搜集，推荐程序模拟：

```text
1. 每辆车绑定一条线路
2. 从线路第一个站点开始
3. 按 `station_order` 依次移动
4. 每隔固定时间更新 `progress`
5. 到站后更新 `current_station_id` 和 `next_station_id`
6. 根据时段、站点类型和天气模拟 `onboard_count`
7. 根据 `onboard_count / capacity` 计算拥挤等级
```

第一版可以只生成静态快照，不必真的每 5 秒更新

## (六) 历史客流数据 `crowd_history`

### 1. 数据作用

历史客流是数据处理工程师最重要的数据，用于：

- 拥挤预测模型训练
- 生成 `crowd_level` 标签
- 管理端客流趋势图
- 站点热度统计
- 拥挤预警演示
- AI 调度建议的上下文输入

### 2. 推荐字段

| **字段名** | **含义** | **示例** | **采集方式** |
|---|---|---|---|
| `record_id` | 记录 ID | `CH000001` | 程序生成 |
| `date` | 日期 | `2026-06-01` | 程序生成 |
| `hour` | 小时 | `8` | 程序生成 |
| `weekday` | 星期 | `1` | 程序生成 |
| `is_weekend` | 是否周末 | `0` | 程序生成 |
| `is_peak` | 是否高峰 | `1` | 根据小时计算 |
| `line_id` | 线路 ID | `L001` | 来自线路表 |
| `station_id` | 站点 ID | `S003` | 来自站点表 |
| `area_type` | 站点区域类型 | `dormitory` | 来自站点表 |
| `weather` | 天气 | `rain` | 模拟或接口获取 |
| `temperature` | 温度 | `27` | 模拟 |
| `passenger_up` | 上车人数 | `35` | 程序模拟 |
| `passenger_down` | 下车人数 | `8` | 程序模拟 |
| `onboard_count` | 当前车上人数 | `48` | 累计或模拟 |
| `capacity` | 车辆容量 | `60` | 来自车辆表 |
| `crowd_rate` | 拥挤率 | `0.80` | 计算 |
| `crowd_level` | 拥挤等级 | `high` | 规则生成 |

### 3. 推荐搜集方式

真实客流很难拿到，推荐用 Python 规则生成

建议生成规模：

```text
30 天历史数据
每天 6:00-22:00
按小时统计
覆盖 3 条线路、20-30 个站点
```

粗略数据量：

```text
30 天 × 17 个小时 × 3 条线路 × 每条线路约 8 个站点 ≈ 12240 条记录
```

这个规模足够用于训练随机森林或 XGBoost，也足够做管理端图表

### 4. 推荐模拟规则

时段规则：

| **时段** | **规则** |
|---|---|
| 7:00-9:00 | 早高峰，宿舍区、地铁站上车人数增加，教学区下车人数增加 |
| 11:30-13:30 | 午高峰，教学区到食堂方向客流增加 |
| 17:00-19:00 | 晚高峰，教学区、图书馆上车人数增加，宿舍区下车人数增加 |
| 20:00-22:00 | 图书馆、实验楼到宿舍区客流增加 |
| 其他时段 | 客流保持较低水平 |

天气规则：

| **天气** | **影响** |
|---|---|
| `sunny` | 正常客流 |
| `cloudy` | 正常或略高 |
| `rain` | 公交需求增加 10%-25%，车辆速度下降 |
| `hot` | 中午短途客流略增加 |

站点类型规则：

| **区域类型** | **早高峰** | **午高峰** | **晚高峰** |
|---|---|---|---|
| `dormitory` | 上车多 | 中等 | 下车多 |
| `teaching` | 下车多 | 上下车都多 | 上车多 |
| `canteen` | 低 | 上下车都多 | 中等 |
| `library` | 中等 | 中等 | 上车多 |
| `subway` | 上车多 | 中等 | 下车多 |

拥挤等级规则：

```text
crowd_rate = onboard_count / capacity

0.00 <= crowd_rate < 0.40：low
0.40 <= crowd_rate < 0.75：medium
0.75 <= crowd_rate <= 1.00：high
```

## (七) 天气与日期数据 `weather_calendar`

### 1. 数据作用

天气和日期数据用于让预测模型更像真实业务，作为拥挤预测特征：

- 雨天公交需求通常增加
- 周末教学区客流下降
- 高峰时段拥挤概率上升
- 特殊活动会带来站点异常客流

### 2. 推荐字段

| **字段名** | **含义** | **示例** | **采集方式** |
|---|---|---|---|
| `date` | 日期 | `2026-06-01` | 程序生成 |
| `weekday` | 星期 | `1` | 程序生成 |
| `is_weekend` | 是否周末 | `0` | 程序生成 |
| `weather` | 天气 | `rain` | 模拟 |
| `temperature` | 温度 | `27` | 模拟 |
| `event_type` | 事件类型 | `exam`、`activity`、`none` | 手工配置 |
| `event_station_id` | 事件影响站点 | `S010` | 手工配置 |

### 3. 推荐搜集方式

第一版直接模拟，不需要调用真实天气 API：

```text
晴天 55%
阴天 25%
雨天 15%
高温 5%
```

特殊事件可以手工安排 2-3 次：

```text
考试周：教学楼、图书馆客流增加
运动会：体育馆站客流增加
大型讲座：报告厅或创新楼附近客流增加
```

## (八) 拥挤预测结果数据 `crowd_prediction`

### 1. 数据作用

拥挤预测结果用于：

- 乘客端显示未来车辆拥挤程度
- 舒适车次推荐
- 管理端预警列表
- AI 调度建议
- 算法效果展示

### 2. 推荐字段

| **字段名** | **含义** | **示例** | **采集方式** |
|---|---|---|---|
| `prediction_id` | 预测 ID | `CP000001` | 程序生成 |
| `line_id` | 线路 ID | `L001` | 输入特征 |
| `station_id` | 站点 ID | `S003` | 输入特征 |
| `vehicle_id` | 车辆 ID | `V001` | 输入特征 |
| `predict_time` | 预测时间 | `2026-07-06 08:30:00` | 程序生成 |
| `predicted_count` | 预测载客人数 | `51` | 模型或规则输出 |
| `predicted_rate` | 预测拥挤率 | `0.85` | 计算 |
| `predicted_level` | 预测拥挤等级 | `high` | 模型或规则输出 |
| `confidence` | 置信度 | `0.86` | 模型概率或模拟 |
| `model_version` | 模型版本 | `rule_v1`、`rf_v1` | 手工标识 |

### 3. 推荐搜集方式

预测结果不是外部搜集数据，而是由你生成：

```text
第一阶段：根据规则模型生成 `crowd_prediction.csv`
第二阶段：训练随机森林或 XGBoost，输出预测结果
第三阶段：将预测结果提供给后端接口或导入数据库
```

第一阶段优先用规则模型：

```text
输入：当前车上人数、车辆容量、小时、站点类型、天气
输出：low / medium / high
```

第二阶段推荐机器学习模型：

```text
RandomForestClassifier
```

原因：

- 对小规模表格数据友好
- 依赖比 XGBoost 少
- 容易解释特征重要性
- 训练速度快，适合实训项目

## (九) ETA 计算辅助数据

### 1. 数据作用

ETA 是 Estimated Time of Arrival，即预计到站时间，用于：

- 乘客端展示几分钟后到站
- 舒适推荐中比较“快”和“舒服”
- AI 出行助手生成推荐理由

### 2. 需要提供的数据

| **数据** | **来源** | **说明** |
|---|---|---|
| `avg_time_to_next` | `line_station` | 相邻站点平均耗时 |
| `distance_to_next` | `line_station` | 相邻站点距离 |
| `current_station_id` | `bus_realtime` | 当前站点 |
| `next_station_id` | `bus_realtime` | 下一站 |
| `progress` | `bus_realtime` | 两站之间进度 |
| `weather` | `crowd_history` 或实时模拟 | 雨天会增加耗时 |
| `is_peak` | 根据小时计算 | 高峰期会增加耗时 |
| `crowd_level` | 规则或模型生成 | 拥挤会增加上下客时间 |

### 3. 推荐搜集方式

ETA 不需要你单独找一份真实数据，主要由站间平均耗时和实时车辆状态计算得到

建议第一版 ETA 公式：

```text
ETA = 剩余站间平均耗时 × 时段系数 × 天气系数 + 停靠延迟 + 拥挤延迟
```

系数建议：

| **因素** | **系数或延迟** |
|---|---|
| 平峰 | `1.0` |
| 早晚高峰 | `1.3` |
| 雨天 | `1.2` |
| 中拥挤 | 每站额外 `0.5` 分钟 |
| 高拥挤 | 每站额外 `1` 分钟 |

# 八、数据质量要求

## (一) 基础数据要求

| **检查项** | **要求** |
|---|---|
| ID 唯一 | `line_id`、`station_id`、`vehicle_id` 不重复 |
| 关联有效 | `line_station.line_id` 必须存在于 `bus_line` |
| 站点有效 | `line_station.station_id` 必须存在于 `bus_station` |
| 顺序连续 | 同一线路的 `station_order` 从 1 开始连续递增 |
| 经纬度合理 | 同一线路站点坐标不能相距过大或完全相同 |
| 容量合理 | `capacity` 建议在 35-80 之间 |

## (二) 客流数据要求

| **检查项** | **要求** |
|---|---|
| 人数非负 | `passenger_up`、`passenger_down`、`onboard_count` 不能小于 0 |
| 不超过容量 | `onboard_count` 原则上不超过 `capacity` |
| 高峰明显 | 早晚高峰应比平峰有更高客流 |
| 天气有效 | `weather` 取值要统一 |
| 标签一致 | `crowd_level` 要和 `crowd_rate` 阈值一致 |

## (三) 字段命名要求

字段命名建议统一使用英文小写和下划线：

```text
line_id
station_id
vehicle_id
crowd_level
onboard_count
predict_time
```

不要混用：

```text
lineId
LineID
线路id
```

# 十、答辩表达建议

> 本项目采用“半真实线路站点数据 + 仿真车辆运行数据 + 模拟历史客流数据”的方式构建数据集。线路和站点参考校园公交与地图场景手工整理，站点经纬度通过地图拾取或仿真配置获得；车辆实时运行状态通过程序模拟生成；历史客流数据根据早晚高峰、天气、星期和站点类型规则生成，用于训练拥挤程度预测模型

> 数据处理部分负责设计公交基础数据结构，构建线路、站点、车辆、历史客流和预测结果数据集，并基于客流规律生成拥挤等级标签。后续使用规则模型和随机森林模型对拥挤程度进行预测，为乘客端舒适推荐、管理端预警和 AI 调度建议提供数据支撑
